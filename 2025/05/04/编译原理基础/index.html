概述

Compiler (编译器) 是一个程序, 将源代码翻译成目标代码. 如 GCC, Clang.

Interpreter (解释器) 是一个程序, 逐行或逐语句解释源代码, 不生成独立的目标代码. 如 Python, Perl 解释器.

编译器结构

编译器的结构非常模块化, 一般分为:

  • 前端, Front End, 含预处理, 词法分析, 语法分析, 语义分析, 中间代码生成, 如 Clang
  • 中端, Middle End, 含中间代码优化
  • 后端, Back End, 含目标代码生成, 如 llvm

这样就能为不同语言编写 Front End, 而共用 Back End, 以节约工作量.

也可以通过编写不同的 Back End 实现跨平台.

编译的主要步骤包括:

  • 预处理, Preprocessing, 处理源代码中的宏, 头文件包含, 条件编译等 之后:
  • 词法分析, Lexical Analysis, 将源代码拆分为词法单元 (Token), 如关键字, 标识符, 运算符等, 常见工具如 lex, flex 拆分为:
  • 语法分析, Syntax Analysis, 根据语法规则将 Token 组合成抽象语法树 (AST), 工具有 yacc 和 bison 等 生成:
  • 语义分析, Semantic Analysis, 检查类型, 作用域等语义规则 (如变量是否声明, 类型是否匹配, 查看语法树是否合规)
  • 中间代码生成, Intermediate Code Generation, 将 AST 转换为平台无关的中间表示 (如 LLVM IR, 三地址码)
  • 中间代码优化, Optimization, 对 IR 进行优化 (如常量折叠, 死代码消除)
  • 目标代码生成, Code Generation, 将 IR 转换为目标机器码 (如 x86, ARM) 等.

可以看作是 “流水线” 结构. 阶段的划分能让工程上的实现更容易.

上下文无关文法, CFG

上下文无关文法, Context-Free Grammer, 是一种形式文法 (某种格式), 用于描述编程语言, 自然语言的结构. 其核心特征是规则的左侧包含 “单个非终结符”, 且规则的生成不依赖上下文.

一个上下文无关文法 $G$ (Grammer) 是一个四元组 (由四个部分组成):
$$
\displaylines
{
\begin{aligned}
G = (T, N, P, S)
\end{aligned}
}
$$

  • T (Terminals), 终结符集合, 即不可再分的基本符号, 一般用小写符号表示, 如 a, b, 0, 1
  • N (Non-ternimals) 非终结符集合, 指可以被替换的符号, 一般用大写符号表示, 如 S, A, B
  • P (Productions) 一组产生式规则, 形式为 非终结符 -> 符号串, 如 $X \rightarrow \beta_1 \beta_2 … \beta_n$, 其中 $X \in N, \beta_i \in (T \cup N)$
  • S (Start Symbol) 初始非终结符

如, 生成一组平衡括号:

  • $S \rightarrow (S) | SS | \epsilon$, $\epsilon$ 指空串, 这个 production 可以生成: (), (()), ()()


(这里约定, 非终结符为大写, 终结符为小写. 也有的约定, 非终结符用尖括号包裹, 如 $<E>$, 终结符需要添加下划线, 如 $\underline{num}$)

自顶向下分析

自顶向下分析是一种语法分析 (Parsing) 方式, 从文法的开始符号出发, 逐步推导出与输入字符串匹配的语法树. 即分析方向是从根节点 (顶层) 向叶子节点 (底层) 展开, 尝试用产生式规则匹配输入符号.

如, 给定文法: $S\ \rightarrow\ aSb\ |\ \epsilon$.

  • 输入字符串 aabb, 分析过程为:
  1. 从 S 开始, 选择规则 $S\ \rightarrow\ aSb$
  2. 当前推导: $aSb\ \rightarrow\ aaSbb$
  3. 最后应用 $S\ \rightarrow\ \epsilon$ 得到 aabb, 匹配输入字符串

  • 在不匹配时进行回溯

Lookahead

Lookahead, 前看符号, 指解析器在决定使用哪条产生式规则时, 预先查看输入流中接下来的一个或多个符号, 以消除歧义或避免回溯. 帮助解析器在多个可能的产生式规则之间做出正确选择.

示例, 假如有文法规则:

当输入字符串 ab, 解析器在 S 处需要决定选择 aB 还是 aC:

  • 如果 lookahead=1, 即下一个符号是 b, 则选择 S -> aB

因为此时, 读入的第一个字符是 a, 为当前处理字符, 因此 lookahead=1 是下一个字符, 即 b.

避免回溯是指:

  • 选择了一个错误的规则, 但还有其他的规则可以尝试, 因此需要回溯, 而 lookahead 相当于提前做了比较, 选择了相对正确的规则, 此时如果错误, 则说明不匹配, 直接返回, 而非回溯

消除歧义是指:

  • 多个文法规则有相同的前缀, 如 if E then S else S 和 if E then S, 用 lookahead 可以确定使用哪一条

不动点算法

不动点算法的核心在于, 迭代更新集合, 直到集合的内容不发生变化为止.

FIRST 集

FIRST 集主要用于 LL(1) 分析, 即通过 lookahead_k=1, 从多个产生式中选择一条, 以避免回溯.

FIRST(a), 指从符号串 a 能推导出的首个终结符集合. 规则为:

  • 如果 a 以终结符开头, FIRST(a) = {a}
  • 如果 $a \rightarrow \epsilon$, 则 $\epsilon \in FIRST(a)$
  • 如果 $a \rightarrow Y1Y2…Yn$, 则 FIRST(a), 包含 FIRST(Y1), 若 Y1 可推导出 $\epsilon$, 则继续加入 FIRST(Y2)
    (也就是说, 对于每个非终结符, 都要利用其对应的文法规则来计算出 FIRST 集)
    示例, 如果文法为:
  • FIRST(S) = {a, b}
  • FIRST(B) = {c, epsilon}
  • FIRST(C) = {d}

又比如:

  • 则有: $FIRST(A) = FIRST(B) \cup FIRST(C)$ = {b, c, \epsilon}

FIRST 集避免回溯: 如果两条规则的 FIRST 集合不相交, 解析器可以无歧义地选择规则, 如:

  • 如果 lookahead='a', 则选择 A -> aB
  • 如果 lookahead='b', 则选择 A -> bC

Nullable 集

Nullable 集合也用于自顶向下语法分析, 其标识了哪些非终结符可以推导出空串 ($\epsilon$).

定义 Nullable(A): 如果非终结符 A 能够通过一系列产生式规则推导出空串 $\epsilon$, 则 A 是 Nullable 的.

Nullable 集合的作用:

  1. 如果 A 是 Nullable, 则计算 FIRST(a) 时需要检查 a 中 A 之后的符号

FOLLOW 集

FOLLOW(A) 用于自顶向下分析, 指可能出现在非终结符 A 之后的终结符集合. 如果 A 是规则的最右符号, 则 $ 也属于 FOLLOW(A).

主要解决以下两个问题：

  1. 何时使用 ε 产生式(如 A → ε). 决策依据: 当 lookahead 符号属于 FOLLOW(A) 时, 选择 A → ε, 如: • FOLLOW(A) = { b }（因为 A 后面必须是 b）。

• 当 lookahead = b 时，选择 A → ε（因为 b ∈ FOLLOW(A), 如果不选择空, 而是 A -> a, 则必然会回溯, 因为 a 与 b 不匹配）
2. 如何避免语法分析中的冲突（如 A → α 和 A → β 的 FIRST 集合有重叠时)

构架规则:

  1. 开始符号：
    • 如果 S 是开始符号，则 $ ∈ FOLLOW(S)。

  2. 产生式中的后继符号：
    • 如果存在产生式 B → α A β，则 FIRST(β) 的所有终结符属于 FOLLOW(A)。

    • 如果 β 是 Nullable 的（即 β →* ε），则 FOLLOW(B) 也属于 FOLLOW(A)。

  3. 递归传递：
    • FOLLOW 关系需要递归计算，直到不再变化。

** 计算 FOLLOW 的算法示例**
输入：文法规则、FIRST 集合、Nullable 信息。
输出：所有非终结符的 FOLLOW 集合。
步骤：

  1. 初始化所有 FOLLOW(A) = ∅，且 FOLLOW(S) = { $ }（S 是开始符号）。

  2. 对每条产生式 B → α A β：
    • 将 FIRST(β) 的非 ε 终结符加入 FOLLOW(A)。

    • 如果 β 是 Nullable 的（或 β 为空），则将 FOLLOW(B) 加入 FOLLOW(A)。

  3. 重复步骤2，直到所有 FOLLOW 集合不再变化。

• FIRST(B) = { b, ε }, Nullable(B) = true

• FIRST(C) = { c, ε }, Nullable(C) = true

• FIRST(S) = { a, b, c, ε }

步骤2：初始化 FOLLOW
• FOLLOW(S) = { $ }

• FOLLOW(A) = FOLLOW(B) = FOLLOW(C) = ∅

步骤3：逐条处理产生式
(1) 产生式 S → A B C
• A 的后继符号是 B C：

• FIRST(B C) = FIRST(B) = { b, ε }

• 加入 b 到 FOLLOW(A)。

• 因为 B 是 Nullable 的，继续检查 C： 

◦ `FIRST(C) = { c, ε }`  

◦ 加入 `c` 到 `FOLLOW(A)`。

• 因为 C 也是 Nullable 的，将 FOLLOW(S)（即 $）加入 FOLLOW(A)。

• 结果：FOLLOW(A) = { b, c, $ }

• B 的后继符号是 C：

• FIRST(C) = { c, ε }

• 加入 c 到 FOLLOW(B)。

• 因为 C 是 Nullable 的，将 FOLLOW(S)（即 $）加入 FOLLOW(B)。

• 结果：FOLLOW(B) = { c, $ }

• C 是最后一个符号：

• 直接继承 FOLLOW(S)（即 $）。

• 结果：FOLLOW(C) = { $ }

(2) 其他产生式
• A → a | ε、B → b | ε、C → c | ε 不改变 FOLLOW 集合。

最终 FOLLOW 集合

非终结符 FOLLOW
S { $ }
A { b, c, $ }
B { c, $ }
C { $ }

LL(k) 是一种自顶向下分析方法, 使用 k 个 lookahead 符号来预测产生式.

一般使用 LL(1), 即只前看一个符号.

  • (+ Term)* 表示 + Term 可以重复出现, 类似正则表达式中 * 的功能, () 表示 + 和 Term 需要一起出现
    (注意, 只有出现终结符时, 才读入一个字符并匹配和弹出规则)

若解析 1 + 2 * 3, 且 k=1 (lookahead 的长度):

  1. 初始化:
    • 输入流 ['1', '+', '2', '*', '3', '$'] ($ 是结束符)
    • 解析栈 [Expr, $] ($ 是栈底)
    • Lookahead(1), 当前符号为 1
  2. 栈顶为 Expr
    • 栈顶选择规则 Expr -> Term (+ Term)* (因为 lookahead=1 是数字, 属于 Term 的 FIRST 集)
    • 替换栈顶, 此时栈为 [Term, (+ Term)*, $] (Term (+ Term)* 是两部分)
  3. 栈顶为 Term
    • 选择规则 Term -> Factor (*Factor)* (因为 lookahead=1 是数字, 属于 Factor 的 FIRST 集)
    • 替换栈顶后, 栈为 [Factor, (*Factor)*, (+Term)*, $]
  4. 栈顶为 Factor
    • 选择规则 Factor -> number (因为 lookahead=1 是数字)
    • 匹配数字的流程为:
      • 弹出 Factor, 压入 number
      • 栈变为 [number, (*Factor)*, (+Term)*, $]
      • 输入 1 与栈顶 number 匹配, 消耗 1
      • 更新输入流: ['+', '2', '*', '3', '$']
      • 弹出 number, 栈变为 [(*Factor)*, (+Term)*, $]
  5. 栈顶 (*Factor)*
    • lookahead='+', 不属于 * 的 FIRST 集, 选择 $\epsilon$ (* 可以匹配 0 个重复项, 即空串)
    • 弹出 (*Factor)*, 栈: [(+Term)*, $]
  6. 栈顶 (+Term)*
    • lookahead='+', 属于 + 的 FIRST 集, 选择 + Term
    • 替换栈顶, 栈: ['+', 'Term', (+Term)*, $]
  7. 栈顶 +
    • 与输入 + 匹配, 消耗 +
    • 更新输入流 ['2', '*', '3', '$']
    • 弹出 +, 栈: [Term, (+Term)*, $]

    • 更新栈：[ Factor, Term', Expr', $ ]

  3. 处理 Factor：

    • lookahead：number

  4. 处理 Term'：

    • 匹配 +，消耗输入符号，弹出 Expr'。

    • 更新栈：[ Term, Expr', $ ]

  6. 继续处理（略）…
    • 最终栈和输入流均变为 $，解析成功。

自底向上分析

自底向上分析的核心, 思想是: 从输入符号串出发, 通过一系列 Reduce 到文法的开始符号, 最终构建出语法树. 如:

文法规则

输入串
1 + 2 * 3

分析步骤

栈 剩余输入 动作 说明
[] 1 + 2 * 3 $ 移进 1 压入数字
[1] + 2 * 3 $ 规约 Factor → number 1 是终结符，规约为 Factor
[Factor] + 2 * 3 $ 规约 Term → Factor 继续规约
[Term] + 2 * 3 $ 规约 Expr → Term 栈顶 Term 规约为 Expr
[Expr] + 2 * 3 $ 移进 + 压入运算符
[Expr, +] 2 * 3 $ 移进 2 压入数字
[Expr, +, 2] * 3 $ 规约 Factor → number 2 规约为 Factor
[Expr, +, Factor] * 3 $ 规约 Term → Factor 继续规约
[Expr, +, Term] * 3 $ 移进 * 压入运算符
[Expr, +, Term, *] 3 $ 移进 3 压入数字
[Expr, +, Term, *, 3] $ 规约 Factor → number 3 规约为 Factor
[Expr, +, Term, *, Factor] $ 规约 Term → Term * Factor 关键步骤：乘法优先规约
[Expr, +, Term] $ 规约 Expr → Expr + Term 加法规约
[Expr] $ 接受（Accept） 匹配完成

通过反复应用产生式规则的逆操作 (即规约), 将输入符号组合成非终结符, 最终规约为开始符号. 关键操作是:

  • Shift, 移进, 将输入符号压入栈中, 等待后续规约
  • Reduce, 规约, 当栈顶的符号序列匹配某条产生式的右侧 (如 A → α) 时, 将 α 替换为 A

本质上是逆向的最右推导 (Rightmost Derivation in Reverse):

  • 最右推导: 从开始符号出发, 每次替换最右侧的非终结符 (如 Expr → Expr + Term → Expr + Factor → Expr + 3)
  • 逆向操作: 自底向上分析是逆向执行这一过程 (如 3 → Factor → Term → Expr)

  • 如果能将一个串最终规约到开始符号, 那么就说明能推导出, 否则中间有错误语法.

LR 分析算法

LR 分析算法的设计初衷是为了解决自顶向下分析的局限性, 尤其是在处理复杂文法和左递归时的不足. (LR 是 Left-to-right input, Rightmost derivative in reverse 的缩写)

LL 分析, 要求文法不能左递归, 如 A -> Aa, 否则会导致无限循环. 如 Expr -> Expr + Term 这样的左递归规则在 LL 分析中无法直接使用.

LL 文法还需要保证 FIRST 和 FOLLOW 无冲突, 但许多实际语言的文法无法满足这一条件.

在不满足条件时, 需要修改文法, 将左递归改为有递归, 如 Expr -> Term Expr', 这样会降低代码的可读性和直观性.

而 LR 分析天然支持左递归文法, 无需改写即可直接处理, 且 LR 能解析几乎所有上下文无关文法 (CFG), 包括 LL 无法处理的复杂结构.

LR(0) 分析算法

LR(0) 是自底向上语法分析的基础算法, 其核心思想是通过有限状态自动机 (DFA) 和栈的组合, 实现对输入符号串的移进 (Shift) 和规约 (Reduce) 操作.

对于这里的 DFA, 有几个关键概念:

  • State, 状态, 表示当前分析的位置
  • State Transfer, 状态转移, 根据下一个输入符号或栈顶符号跳转到新状态
  • Item, 项目, 指一个添加了点记号的产生式, 如 A -> $a \cdot b$, 这里 a 表示已读部分, b 表示待读部分

Item 记录了一条产生式在分析过程中的状态, 如:

  • $A \rightarrow \cdot \alpha \beta$, 此时尚未解析任何部分
  • $A \rightarrow \alpha \cdot \beta$, 已解析 $\alpha$, 待解析 $\beta$
  • $A \rightarrow \alpha \beta \cdot$, 已解析整个产生式

Item set 通过闭包扩展操作来获得, 用来表示状态机中的某个状态, 得用一个例子来说明.

给定文法：

初始项目集 ( I_0 )
初始时，项目集仅包含增广文法的初始项目：
$$
\displaylines
{
\begin{aligned}
I_0 = { S \to \cdot AB }
\end{aligned}
}
$$

执行闭包操作

  1. 检查 $( S \to \cdot AB )$：

    • 点（$\cdot$）后面是 非终结符 $( A )$, 因此需要将 $( A )$ 的所有产生式的初始项目加入 $( I_0 )$.

    • $( A )$ 的产生式是 $( A \to a )$, 所以加入:

$$
\displaylines
{
\begin{aligned}
A \to \cdot a
\end{aligned}
}
$$

  • 现在项目集变为:

$$
\displaylines
{
\begin{aligned}
I_0 = { S \to \cdot AB, A \to \cdot a }
\end{aligned}
}
$$

  1. 检查 $( A \to \cdot a )$:

    • 点后面是 终结符 $( a )$, 没有非终结符需要扩展, 因此无需添加新项目.

  2. 检查 $( S \to \cdot AB )$ 的 $( B )$:

    • 虽然 $( B )$ 也是一个非终结符, 但点尚未移动到 $( B )$ 前面 (当前是 $( \cdot AB )$), 所以暂时不需要处理 $( B )$ 的产生式.

最终闭包 $( \text{Closure}(I_0) )$
$$
\displaylines
{
\begin{aligned}
\text{Closure}(I_0) = { S \to \cdot AB, A \to \cdot a }
\end{aligned}
}
$$

项目集的变化过程（读入字符时）
1. 从 $( I_0 )$ 读入 $( a )$

  • 当前项目集 $( I_0 )$:
    $$
    \displaylines
    {
    \begin{aligned}
    { S \to \cdot AB, A \to \cdot a }
    \end{aligned}
    }
    $$

  • 匹配的项目:

    • $( A \to \cdot a )$（可以移进 $( a )$）

  • 移进操作:

    • 读入 $( a )$，点向右移动:

$$
\displaylines
{
\begin{aligned}
A \to a \cdot
\end{aligned}
}
$$

  • 这是一个 规约项目（点在末尾），表示可以用 $( A \to a )$ 规约。

  • 规约时不读入新字符，因此项目集 不会立即改变，而是弹出栈顶状态并应用规约。

2. 规约后转移到新状态

  • 规约 $( A \to a )$ 后，相当于已经解析出 $( A )$，接下来需要处理 $( S \to A \cdot B )$。

  • 计算新状态 $( I_1 )$:

    • 从 $( I_0 )$ 的 $( S \to \cdot AB )$ 出发，读入 $( A )$ 后，点移动到 $( A )$ 后面:

$$
\displaylines
{
\begin{aligned}
S \to A \cdot B
\end{aligned}
}
$$

  • 检查点后面的 $( B )$（非终结符），加入 $( B )$ 的产生式:
    $$
    \displaylines
    {
    \begin{aligned}
    B \to \cdot b
    \end{aligned}
    }
    $$
  • 因此，新状态 $( I_1 )$ 为:

$$
\displaylines
{
\begin{aligned}
I_1 = { S \to A \cdot B, B \to \cdot b }
\end{aligned}
}
$$

3. 从 $( I_1 )$ 读入 $( b )$

  • 当前项目集 $( I_1 )$:

$$
\displaylines
{
\begin{aligned}
{ S \to A \cdot B, B \to \cdot b }
\end{aligned}
}
$$

  • 匹配的项目:

    • $( B \to \cdot b )$（可以移进 $( b )$）

  • 移进操作:

    • 读入 $( b )$，点向右移动:

$$
\displaylines
{
\begin{aligned}
B \to b \cdot
\end{aligned}
}
$$

  • 这是一个 规约项目，表示可以用 $( B \to b )$ 规约。

  • 规约后，弹出栈顶状态，并回到 $( I_1 )$。

4. 规约后转移到 $( I_2 )$

  • 规约 $( B \to b )$ 后，相当于已经解析出 $( B )$，接下来处理 $( S \to AB \cdot )$。

  • 新状态 $( I_2 )$:

    • 从 $( I_1 )$ 的 $( S \to A \cdot B )$ 出发，读入 $( B )$ 后:

$$
\displaylines
{
\begin{aligned}
S \to AB \cdot
\end{aligned}
}
$$

  • 这是一个 接受项目（如果是增广文法 $( S’ \to S \cdot )$），表示分析完成。

初始项目集 $I_0$ 为:
$$
\displaylines
{
\begin{aligned}
I_0 = {S \rightarrow \cdot AB }
\end{aligned}
}
$$

应用 LR(0) 算法的步骤:

1: 构造 LR(0) 项目集（状态）
对文法的每个产生式，生成带“点符号”的项目，例如：

生成的项目集：
• S → ·E

• E → ·E + T

• E → ·T

• T → ·id

2：构建 DFA 状态转移
通过闭包（Closure）和转移（Goto）生成所有可能的状态：

  1. 初始状态（State 0）：
    • 包含 S → ·E 及其闭包（所有 E 的展开项）。

  2. 状态转移：
    • 如果当前状态有 A → α·Xβ（X 是终结符或非终结符），则读入 X 后跳转到新状态 A → αX·β。

示例：
• 从 State 0 读 E 跳转到 State 1（S → E·）。

• 从 State 0 读 T 跳转到 State 2（E → T·）。

3：分析表（Action & Goto）
根据 DFA 状态生成分析表：
• Action 表：指导移进或规约。

• 如果状态中有 A → α·aβ（a 是终结符），则 Action[s, a] = Shift。

• 如果状态中有 A → α·，则 Action[s, *] = Reduce A → α。

• Goto 表：指导非终结符的跳转。

示例解析
文法规则

输入串
id + id

分析过程

栈 剩余输入 Action 说明
[State 0] id + id $ Shift (State 3) 移进 id
[0, id(3)] + id $ Reduce T → id 规约 id 为 T
[0, T(2)] + id $ Goto State 2 根据 E → T 跳转
[0, E(1)] + id $ Shift (State 4) 移进 +
[0, E(1), +(4)] id $ Shift (State 3) 移进 id
[0, E(1), +(4), id(3)] $ Reduce T → id 规约 id 为 T
[0, E(1), +(4), T(5)] $ Goto State 5 根据 E → E + T 跳转
[0, E(1)] $ Accept 完成规约（S → E）

LR(0) 的局限性

  1. 无前瞻（Lookahead）：
    • LR(0) 不预读输入符号，可能导致规约冲突（如 Reduce/Reduce 冲突）。

    • 示例：

    在状态 [A → id·, B → id·] 时，无法决定规约为 A 还是 B。

  2. 需改进为 SLR/LR(1)：
    • SLR(1) 通过 FOLLOW 集解决部分冲突。

    • LR(1) 通过动态前瞻符号彻底解决冲突。
