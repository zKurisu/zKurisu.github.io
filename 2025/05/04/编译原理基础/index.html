概述

Compiler (编译器) 是一个程序, 将源代码翻译成目标代码. 如 GCC, Clang.

Interpreter (解释器) 是一个程序, 逐行或逐语句解释源代码, 不生成独立的目标代码. 如 Python, Perl 解释器.

编译器结构

编译器的结构非常模块化, 一般分为:

  • 前端, Front End, 含预处理, 词法分析, 语法分析, 语义分析, 中间代码生成, 如 Clang
  • 中端, Middle End, 含中间代码优化
  • 后端, Back End, 含目标代码生成, 如 llvm

这样就能为不同语言编写 Front End, 而共用 Back End, 以节约工作量.

也可以通过编写不同的 Back End 实现跨平台.

编译的主要步骤包括:

  • 预处理, Preprocessing, 处理源代码中的宏, 头文件包含, 条件编译等 之后:
  • 词法分析, Lexical Analysis, 将源代码拆分为词法单元 (Token), 如关键字, 标识符, 运算符等, 常见工具如 lex, flex 拆分为:
  • 语法分析, Syntax Analysis, 根据语法规则将 Token 组合成抽象语法树 (AST), 工具有 yacc 和 bison 等 生成:
  • 语义分析, Semantic Analysis, 检查类型, 作用域等语义规则 (如变量是否声明, 类型是否匹配, 查看语法树是否合规)
  • 中间代码生成, Intermediate Code Generation, 将 AST 转换为平台无关的中间表示 (如 LLVM IR, 三地址码)
  • 中间代码优化, Optimization, 对 IR 进行优化 (如常量折叠, 死代码消除)
  • 目标代码生成, Code Generation, 将 IR 转换为目标机器码 (如 x86, ARM) 等.

可以看作是 “流水线” 结构. 阶段的划分能让工程上的实现更容易.

词法分析

词法分析器, lexical analyser, 作用是将字符流转换为记号流, 作为 parser 的输入.

一个 token 通常有两部分组成:

  • token class, 也可以叫 kind 或 type, 指记号的类别, 一般分 Identifier, Keywords, Number, Whitespace 等等
  • string 专业称呼为 lexeme, 实际的字符.

此时再描述 lexical analysis 的目标就为:

  • 将 input string 拆分为 lexemes
  • 并判断每个 lexeme 的类别

在一般的编程语言中:

  • Identifier 指一个字符开头的字符串
  • Number 指 digits 构成的字符串
  • Keyword 指 else, if, begin 等
  • Whitespace 指 blanks, newlines, tabs 的序列
  • LeftParen, ( 左圆括号
  • RightParen, ) 右圆括号
  • Semicolon, ; 分号

Regular languages

token class 需要根据 lexical structure 来判断, 其结构一般用 regular languages 来描述, 也就是 regular expressions.

每一个 regular expression 代表一个集合.

有两个最基本的 regular expression:

  • Single character, 集合包含一个单字符字符串

  • Epsilon, 集合包含一个空字符串, 需要注意, 不是空集

有三个复合 regular expression:

  • Union, 两个集合的并集
    $$
    \displaylines
    {
    \begin{aligned}
    A + B = { a | a \in A } \cup { b | b \in B }
    \end{aligned}
    }
    $$

  • Concatenation, 两个集合的笛卡尔积 (所有组合方式)
    $$
    \displaylines
    {
    \begin{aligned}
    AB = { ab | a \in A \land b \in B }
    \end{aligned}
    }
    $$

  • Iteration, 一种循环结构, 读作 A star 或 Kleene iteration, Kleene closure, 意思是重复 0 次到 i 次
    $$
    \displaylines
    {
    \begin{aligned}
    A^* &= \bigcup_{i \ge 0} A ^i \newline~ \newline
    A^i &= A … A (重复i次连接) \newline~ \newline
    A^0 &= \epsilon
    \end{aligned}
    }
    $$

  • 左边正则表达式是到右边的集合, 需有某种映射关系, 这就是 L (meaning function) 完成的事

注意 expressions 和 meanings 之间不是 1 对 1 的关系, 多个 expressions (也可以说是 notation) 可以映射到同一个 meanings 上. 比如:

  • 两种不同的记号 (阿拉伯记号和罗马记号) 都能表示相同的数字含义

把 syntax 和 semantic 分离的意义也就在此:

  • 罗马记号不便用于计算, 而阿拉伯记号很容易让人学会基础的计算
  • 用多种 syntax 表达同一 semantic, 能让我们从多种方法中, 找到更好的一个

正则表达式的 Lexical specification

Lexical specification 分几个步骤:

  1. 为每个 token class 编写 regx
  1. 构建 $R$, 匹配所有 lexemes

  1. Let input be $x_1, x_2, …, x_n$
    $$
    \displaylines
    {
    \begin{aligned}
    For\ 1 \le i \le n\ check \newline~ \newline
    x_1 … x_i \in L(R)
    \end{aligned}
    }
    $$
    (一个一个增加前缀, 看是否匹配 $L(R)$)

  2. If success, then we know that
    $$
    \displaylines
    {
    \begin{aligned}
    x_1 … x_i \in L(R_j)\ for\ some\ j
    \end{aligned}
    }
    $$
    (也就是说存在 $R = R_1 + R_2 + R_3 …$, 其中 $x_1 = L(R_1), x_2 = L(R_2)…$)

  3. Remove $x_1, x_2, …, x_n$ from input and go to (3), 相当于一直从输入中截取, 直到输入为空

Maximal Match, 处理 ambiguities

  • 当有两段前缀都匹配 $L(R)$, 此时选择最长匹配的那个, 这比较符合人类的阅读习惯, 比如看到 How 这个单词, 我们不会把它当作三个独立的字母

当一个前缀与多个 $L(Ri)$ 匹配

  • 此时应有优先级排列, 这里是 keyword 优先级高于 identifiers

No rule matches, handle errors

  • 此时编写一个 regex 匹配所有不符合 lexical specification 的字符串, 且其 priority 最低

上下文无关文法, Context-Free Grammer, 是一种形式文法 (某种格式), 用于描述编程语言, 自然语言的结构. 可以判断输入的内容里, 哪些是合法的, 哪些是不合法的.

CFG 的核心特征是规则的左侧包含 “单个非终结符” (毕竟左边的就是可以被替换的), 且规则的生成不依赖上下文.

CFG 中的规则不断替换, 直到全为 Terminals 为止. 每一次替换都可以视为进入一个新状态.

一个上下文无关文法 $G$ (Grammer) 是一个四元组 (由四个部分组成):
$$
\displaylines
{
\begin{aligned}
G = (T, N, P, S)
\end{aligned}
}
$$

  • T (Terminals), 终结符集合, 即不可再分的基本符号, 一般用小写符号表示, 如 a, b, 0, 1
  • N (Non-ternimals) 非终结符集合, 指可以被替换的符号, 一般用大写符号表示, 如 S, A, B
  • P (Productions) 一组产生式规则, 形式为 非终结符 -> 符号串, 如 $X \rightarrow \beta_1 \beta_2 … \beta_n$, 其中 $X \in N, \beta_i \in (T \cup N)$
  • S (Start Symbol) 初始非终结符

如, 生成一组平衡括号:

  • $S \rightarrow (S) | SS | \epsilon$, $\epsilon$ 指空串, 这个 production 可以生成: (), (()), ()()

CFG 的右侧可以有 T, 也可以有 N, 也可以是 $\epsilon$.

Terminals 通常就是 lexer 输出的 tokens.

Error handling

Error handling 是 Parsing 部分的重要功能, 有两种常见的处理策略:

  • Panic mode
  • Error productions

这些机制都尽可能在一次编译过程中发现所有错误.

Panic mode

Panic mode 的核心思想是, 当 parser 遇到错误时, 跳过当前错误的符号或语句, 直到找到一个 synchronization point, 然后恢复分析.

适用于错误较多或难以预测的情况.

工作步骤:

  1. parser 发现当前输入符号不符合任何语法规则, 如 if 后缺少 ()
  2. 丢弃当前符号 (或语句), 直到遇到 synchronization token, 比如 ;, }, 或关键字等
  3. 恢复分析, 从 synchronization point 继续分析

示例:

  1. 检测到 { 之前缺少 (, 报告错误
  2. 跳过 { 和 printf("x is positive");, 直到遇到 }
  3. 从 } 后恢复分析

Lookahead

Lookahead, 前看符号, 指解析器在决定使用哪条产生式规则时, 预先查看输入流中接下来的一个或多个符号, 以消除歧义或避免回溯. 帮助解析器在多个可能的产生式规则之间做出正确选择.

示例, 假如有文法规则:

当输入字符串 ab, 解析器在 S 处需要决定选择 aB 还是 aC:

  • 如果 lookahead=1, 即下一个符号是 b, 则选择 S -> aB

因为此时, 读入的第一个字符是 a, 为当前处理字符, 因此 lookahead=1 是下一个字符, 即 b.

避免回溯是指:

  • 选择了一个错误的规则, 但还有其他的规则可以尝试, 因此需要回溯, 而 lookahead 相当于提前做了比较, 选择了相对正确的规则, 此时如果错误, 则说明不匹配, 直接返回, 而非回溯

消除歧义是指:

  • 多个文法规则有相同的前缀, 如 if E then S else S 和 if E then S, 用 lookahead 可以确定使用哪一条

不动点算法

不动点算法的核心在于, 迭代更新集合, 直到集合的内容不发生变化为止.

FIRST 集

考虑 non-terminal $A$, production $A \rightarrow \alpha$, 以及 token $t$, 如果有 $T[A,t] = \alpha$, 则需要满足下面条件之一:

  • $\alpha \rightarrow^* t \beta$, 即某个 $\alpha$ 的 production 能够在推出 $t$ 在 first position (即开头, 这也是名字的由来), 此时我们可以说 $t \in First(\alpha)$.

换个思路讲, $First(\alpha)$ 指所有 $\alpha$ 产生式中 first position 为 terminal 的 terminal 的集合. 如果 $\alpha$ 能推导出 $\epsilon$, 也要包含在其中.

$$
\displaylines
{
\begin{aligned}
First(X) = {t | X \rightarrow^* t \alpha } \cup {\epsilon | X \rightarrow^* \epsilon }
\end{aligned}
}
$$

当 non-terminal $X$ 能直接推导出 $\epsilon$, 或能够间接推导出 $\epsilon$, 都有 $\epsilon \in First(X)$.

对于 terminal 而言, 其 First 集是它本身:

  • $First(t) = { t }$

FIRST 集主要用于 LL(1) 分析, 即通过 lookahead_k=1, 从多个产生式中选择一条, 以避免回溯.

FIRST(a), 指从符号串 a 能推导出的首个终结符集合. 规则为:

  • 如果 a 以终结符开头, FIRST(a) = {a}
  • 如果 $a \rightarrow \epsilon$, 则 $\epsilon \in FIRST(a)$
  • 如果 $a \rightarrow Y1Y2…Yn$, 则 FIRST(a), 包含 FIRST(Y1), 若 Y1 可推导出 $\epsilon$, 则继续加入 FIRST(Y2)
    (也就是说, 对于每个非终结符, 都要利用其对应的文法规则来计算出 FIRST 集)
    示例, 如果文法为:
  • FIRST(S) = {a, b}
  • FIRST(B) = {c, epsilon}
  • FIRST(C) = {d}

又比如:

  • 则有: $FIRST(A) = FIRST(B) \cup FIRST(C)$ = {b, c, \epsilon}

FIRST 集避免回溯: 如果两条规则的 FIRST 集合不相交, 解析器可以无歧义地选择规则, 如:

  • 如果 lookahead='a', 则选择 A -> aB

FOLLOW(A) 用于自顶向下分析, 指可能出现在非终结符 A 之后的终结符集合. 如果 A 是规则的最右符号, 则 $ 也属于 FOLLOW(A).

主要解决以下两个问题：

  1. 何时使用 ε 产生式(如 A → ε). 决策依据: 当 lookahead 符号属于 FOLLOW(A) 时, 选择 A → ε, 如: • FOLLOW(A) = { b }（因为 A 后面必须是 b）。

• 当 lookahead = b 时，选择 A → ε（因为 b ∈ FOLLOW(A), 如果不选择空, 而是 A -> a, 则必然会回溯, 因为 a 与 b 不匹配）
2. 如何避免语法分析中的冲突（如 A → α 和 A → β 的 FIRST 集合有重叠时)

构架规则:

  1. 开始符号：
    • 如果 S 是开始符号，则 $ ∈ FOLLOW(S)。

  2. 产生式中的后继符号：
    • 如果存在产生式 B → α A β，则 FIRST(β) 的所有终结符属于 FOLLOW(A)。

    • 如果 β 是 Nullable 的（即 β →* ε），则 FOLLOW(B) 也属于 FOLLOW(A)。

  3. 递归传递：
    • FOLLOW 关系需要递归计算，直到不再变化。

** 计算 FOLLOW 的算法示例**
输入：文法规则、FIRST 集合、Nullable 信息。
输出：所有非终结符的 FOLLOW 集合。
步骤：

  1. 初始化所有 FOLLOW(A) = ∅，且 FOLLOW(S) = { $ }（S 是开始符号）。

  2. 对每条产生式 B → α A β：
    • 将 FIRST(β) 的非 ε 终结符加入 FOLLOW(A)。

    • 如果 β 是 Nullable 的（或 β 为空），则将 FOLLOW(B) 加入 FOLLOW(A)。

  3. 重复步骤2，直到所有 FOLLOW 集合不再变化。

  • Shift, 移进, 将输入符号压入栈中, 等待后续规约
  • Reduce, 规约, 当栈顶的符号序列匹配某条产生式的右侧 (如 A → α) 时, 将 α 替换为 A

  • 如果能将一个串最终规约到开始符号, 那么就说明能推导出, 否则中间有错误语法.

LR 分析算法

LR 分析算法的设计初衷是为了解决自顶向下分析的局限性, 尤其是在处理复杂文法和左递归时的不足. (LR 是 Left-to-right input, Rightmost derivative in reverse 的缩写)

LL 分析, 要求文法不能左递归, 如 A -> Aa, 否则会导致无限循环. 如 Expr -> Expr + Term 这样的左递归规则在 LL 分析中无法直接使用.

LL 文法还需要保证 FIRST 和 FOLLOW 无冲突, 但许多实际语言的文法无法满足这一条件.

在不满足条件时, 需要修改文法, 将左递归改为有递归, 如 Expr -> Term Expr', 这样会降低代码的可读性和直观性.

而 LR 分析天然支持左递归文法, 无需改写即可直接处理, 且 LR 能解析几乎所有上下文无关文法 (CFG), 包括 LL 无法处理的复杂结构.

LR(0) 分析算法

LR(0) 是自底向上语法分析的基础算法, 其核心思想是通过有限状态自动机 (DFA) 和栈的组合, 实现对输入符号串的移进 (Shift) 和规约 (Reduce) 操作. (主要是通过项目集来构建 ACTION 和 GOTO 表)

对于这里的 DFA, 有几个关键概念:

  • State, 状态, 表示当前分析的位置
  • State Transfer, 状态转移, 根据下一个输入符号或栈顶符号跳转到新状态
  • Item, 项目, 指一个添加了点记号的产生式, 如 A -> $a \cdot b$, 这里 a 表示已读部分, b 表示待读部分

Item 记录了一条产生式在分析过程中的状态, 如:

  • $A \rightarrow \cdot \alpha \beta$, 此时尚未解析任何部分
  • $A \rightarrow \alpha \cdot \beta$, 已解析 $\alpha$, 待解析 $\beta$
  • $A \rightarrow \alpha \beta \cdot$, 已解析整个产生式

Item set 通过闭包扩展操作来获得, 用来表示状态机中的某个状态, 得用一个例子来说明.

SLR 分析算法

SLR 分析算法是对 LR 的改进.

LR(0) 容易实现, 但能分析的文法有限.

SLR 分析算法与 LR(0) 基本步骤相同, 仅区别于对归约的处理:

  • 对于状态 $i$ 上的项目 $X \rightarrow \alpha \cdot$
  • 仅对 $y \in FOLLOW(X)$ 添加 $ACTION[i,y]$

LR(1) 分析算法

SLR 分析算法有可能减少需要规约的情况, 以及有可能去除移进-规约冲突, 但仍有冲突出现的可能.

在 LR(1) 分析算法中, 一个项目是一个二元组, 如: $[X \rightarrow \alpha \cdot \beta, a]$, 含义是:

  • $\alpha$ 在栈顶上
  • 剩余的输入能够匹配 $\beta a$
  • $a$ 是某一个或某几个终结符

当归约 $X \rightarrow \alpha \beta$ 时, $a$ 是前看符号, 只把 $reduce\ by\ X\ \rightarrow \alpha \beta$ 填入 ACTION[s,a]

LR(1) 与 LR(0) 的基本步骤也相同, 仅闭包的计算不同.
