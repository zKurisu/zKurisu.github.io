RISC-V ISA

RISC 是 “Reduced Instruction Set Computer” (精简指令集计算机) 的缩写. 是加州大学伯克利分校研究团队开源的一个指令集.

V 是罗马数字的 5, 表示这是第五代 RISC 架构.

ISA 是 “Instruction Set Architecture” (指令集架构) 的缩写. 是底层硬件向上层软件提供的一层接口规范:

其定义了, 如 CPU 支持哪些指令, 指令的编码格式, 宽度, 寄存器数量与用途, 内存访问方式, 异常/中断处理等.

指令宽度通常等价于通用寄存器的宽度, 其决定了寻址范围的大小以及数据运算的能力, 比如:

  • 8 位, 寻址范围 $2^8 = 256$
  • 16 位, 寻址范围 $2^{16} = 65536$
  • 32 位, 寻址范围 $2^{32} = 4294967296$
  • 64 位, 寻址范围 $2^{64} = 18446744073709551616$

x86 和 arm 都需要付费授权, 而 RISC-V 是开源免费的, 可以自由使用和修改, 常用于教学和科研. 现在生态也逐渐成熟, 各公司也在推出 RISC-V 芯片.

常见的 ISA 有:

  • x86
  • Power
  • ARM
  • MIPS
  • RISC-V

RISCV ISA 命名格式

格式为:

  • RV 是 RISC-V 的缩写
  • [###] 标识处理器的字宽
  • [abc...xyz] 标识指令集模块集合

比如: RV32IMA:

  • 32 位芯片
  • 支持 I, M, A 三个模块

模块化 ISA

RISC-V ISA 由 1 个基本整数指令集和多个可选的扩展指令集组成, 基础指令集是固定的, 不会改变.

对于 CPU 的实现而言, 基本整数指令集是需要强制实现的.

基本指令集其分为 4 种:

  • RV32I (Integer), 32 位整数指令集
  • RV32E (Embedding), RV32I 的子集, 用于小型的嵌入式场景
  • RV64I, 64 位整数指令集, 兼容 RV32I
  • RV128I, 128 位整数指令集, 兼容 RV64I 和 RV32I

扩展模块指令集有标准化的和非标准化的指令集扩展, 如:

  • M, 整数乘法 (Multiplication) 与除法指令集
  • A, 存储器原子 (Automic) 指令集
  • F, 单精度 (32bit) 浮点 (Float) 指令集
  • D, 双精度 (64bit) 浮点 (Double) 指令集, 兼容 F
  • C, 压缩 (Compressed) 指令集
  • 等等

IMAFD 被称为 General 组合, 用 G 标识.

General Purpose Registers

RISC-V 的 Unprivileged Specification 定义了 32 个通用寄存器以及一个 PC 寄存器, 这些定义对 RV32I, RV64I, RV128I 都一样. 如果实现支持 F/D 扩展则需要额外支持 32 个浮点寄存器.

对于 RV32E, 其将 32 个通用寄存器缩减为 16 个.

每个寄存器在编程时的特定用途和别名定义在 RISC-V Application Binary Interface (ABI) 中.

Privileged level

RISC-V 的 Privileged specification 定义了三个特权级别 (privilege level).

  • User mode
  • Supervisor mode
  • Machine mode

不同 privilege level 下对应各自的 register 用于控制, 称 controal and status registers (CSR).

比如:

  • st 是 Supervisor Trap 的缩写, 表示在 Supervisor 模式下发生的 Exception 或 Interrupt
  • vec 是 Vector 的缩写, 表示 trap 的入口地址或者 trap handler 程序的向量表基址
  • stval 是 Supervisor trap value 的缩写, 表示与此次 trap 相关联的附加信息, 出错地址等

高级别的特权级别可以访问低级别的 CSR, 但反过来不行.

不同特权级别之间可以切换, 比如用 ecall 和 ebreak 指令.

内存管理与保护

分两种:

  • Physical Memory Protection
  • Virtual memory, 需要支持 Supervisor mode, 用于实现高级的操作系统特性

RISC-V 汇编

RISC-V 汇编由多条语句 (statement) 构成, 一条语句有 3 部分组成 (以 GNU 汇编为例):

  • label, 标号, 在 GNU 汇编中, 任何以 : 结尾的标识符都被认为是一个标号, 在用作跳转时汇编器会自动计算与 PC 的 Offset
  • operation 分几类:
    • instruction, 对应机器指令的字符串
    • pseudo-instruction (伪指令), 为了提高编写代码的效率, 可以用一条伪指令指示汇编器产生多条实际的 instructions
    • directive (指示/伪操作), 通过类似指令的形式 (但以 . 开头) 通知汇编其如何控制代码的产生, 不对应具体的指令
    • macro, 用 .macro/.endm 自定义的宏
  • comment: 常用 # 开始到当前行结束 (也有以其他开头的)

操作对象

RISC-V 汇编指令的操作对象包括:

  • 寄存器: 比如 32 个通用寄存器 (限于 RV32I 来讲) x0 ~ x31. 在 RISC-V 中, Hart 在执行算术逻辑运算时所操作的数据必须直接来自寄存器


x0 比较特殊, 只能读, 且值为 0.

  • 内存: Hart 可以执行在寄存器和内存之间的数据读写操作, 对于 RV32, 最多可以访问 $2^{32}$ 个字节的内存空间. 注意寻址的基本单位是 1 byte, 即一个地址对应的空间大小是 1 byte

编码格式

RISC-V 汇编 (RV32I) 有 6 种编码格式:

  • funct3/funct7 (“function”, funct3 表示占了 3 个 bits, funct7 表示占了 7 个 bits) 和 opcode (“operate code”) 决定 instruction 类型
  • rs 是 “register source”, 提供数据; rd 是 “register destination” 存储计算结果
  • R-type, “Register”, 每条指令中有三个 fields, 用于指定 3 个寄存器参数
  • I-type, “Immediate”, 每条指令除了带有两个寄存器参数外, 还带有一个立即数参数
  • S-type, “Store” (与内存相关), 每条指令除了带有两个寄存器参数外, 还带有一个立即数参数 (宽度为 12 bits, 但 fields 的组织方式不同于 I-type)
  • B-type, “Branch” (与分支相关), 每条指令除了带有两个寄存器参数外, 还带有一个立即数参数 (宽度为 12 bits, 但取值为 2 的倍数)
  • U-type, “Upper”, 每条指令含有一个寄存器参数再加上一个立即数参数 (宽度为 20 bits, 用于表示一个立即数的高 20 位)
  • J-type, “Jump”, 每条指令含有一个寄存器参数再加上一个立即数参数 (宽度为 20 bits)

opcode 值解析:

  • inst[1:0] 表示 opcode 的前两位 (从右往左看), 都是 11
  • inst[4:2] 表示 opcode 的第 3, 4, 5 位, 横轴
  • inst[6:5] 表示 opcode 的第 6 和 7 位, 纵轴

比如 ADD 指令的就是 0110011. 在上面 table 里属于 OP 这个类型 (“OPeration”).

注意, RISC-V 指令在内存中按照 “小端序” 排列, 即低位放在低地址.

操作 CSR 寄存器

CSR (Control and Status Register, 控制与状态寄存器), 是 RISC-V 架构中用于控制 CPU 行为, 管理中断/异常, 配置系统模式, 设置内存状态, 读取 CPU 状态等的寄存器. 比如:

  • sstatus, Supervisor Status, 控制 S-Mode 的各种状态和权限
  • sepc, Supervisor Exception Program Counter, 保存发生异常时的 PC
  • sscratch, Supervisor Scratch, 用于 trap 处理中切换栈
  • stvec, Supervisor Trap Vector, 设置 trap handler 的入口地址
  • scause, Supervisor Cause, 记录发生 trap 的原因 (比如中断类型, 异常代码)
  • stval, Supervisor Trap Value, 记录与 trap 相关的附加信息
  • satp, Supervisor Address Translation Protection, 控制页表 (内存虚拟化)

OS 可以通过设置 CSR 来实现隔离:

  • mstatus/sstatus 等地址空间配置寄存器, 能防止应用访问系统管控相关寄存器
  • sstatus/stvec 等中断配置寄存器, 可以防止应用长期使用 100% 的 CPU
  • sstatus/satp/stvec 等地址空间相关寄存器能防止应用破坏窃取数据

RISC-V 提供了专门的指令来访问 CSR, 主要有:

指令 全称 作用 示例
csrrw CSR Read and Write 读取 CSR 到 rd, 同时将 rs1 的值写入 CSR csrrw rd, csr, rs1
csrrs CSR Read and Set bits 读取 CSR 到 rd, 同时将 rs1 中为 1 的位设置到 CSR 中 csrrs rd, csr, rs1
csrrc CSR Read and Clear bits 读取 CSR 到 rd, 同时将 rs1 中为 0 的位清除 CSR 中的对应位 csrrc rd, csr, rs1
csrr CSR Read 仅将 CSR 的值读到 rd, 不修改 CSR csrr rd, csr
csrw CSR Write 将 rs1 的值写入 CSR, 不读取旧值 csrw csr, rs1

还有两条伪指令可用:

  • csrs csr, imm, CSR Set, 对 CSR 的某些位进行置位 (写 1), imm 是立即数
  • csrc csr, imm, CSR Clear, 对 CSR 的某些位进行清零 (写 0), imm 是立即数

Trap 处理机制

在 RISC-V 的 $用户态 \rightarrow 内核态$ trap 流程中, sscratch 寄存器的使用为:

  1. 初始化阶段, 内核启动时, 需要将内核栈的地址写入 sscratch, 即 csrw sscratch, kernel_sp
  2. 用户态运行时, 此时不需要修改该寄存器内容, sp 保存的是 U-Mode 的栈
  3. Trap 发生 (比如系统调用等), CPU 会从 U-Mode trap 进 S-Mode, 此时交换 sscratch 和 sp 的值, 以使用内核栈, csrrw sp, sscratch, sp

在上电时, sscratch 默认为 0 或未初始化的随机值, 不会自动保存任何有意义的地址.

跨特权级的指令: ecall

ecall (Environment Call, 环境调用) 是 RISC-V 中一条特权指令, 用于从低特权级 (如 User Mode) 向高特权级 (如 Supervisor Mode 或 Machine Mode) 发起一个请求, 通常是请求操作系统内核提供服务 (执行系统调用).

当 CPU 执行 ecall 时, 会触发一个 Environment Call Exception, 并跳转到对应的 trap 处理程序, 由操作系统处理该请求.

ecall 执行不需要任何参数:

当 CPU 执行到 ecall 时, 会进行:

  1. 立即停止当前程序的执行流程
  2. 产生一个 Environment Call Exception 的特殊异常
  3. 根据当前 Mode 和 mtvec/stvec 的设置, 跳转到对应的 Trap 处理程序 (通常是内核的 trap_handler)
  4. 操作系统内核在 trap 处理程序中, 通过检查 scause/mcause 得知其是一个 ecall, 然后根据 a7 (系统调用号) 和其他寄存器决定提供什么服务

不同 Mode 下 ecall 触发的异常类型不同:

特权模式 异常类型 (cause 值) 描述
User Mode (U-Mode) 8 (Exception Code: Environment Call from U-Mode) 用户态程序执行 ecall
Supervisor Mode (S-Mode) 9 (Exception Code: Environment Call from S-Mode) 操作系统执行 ecall
Machine Mode (M-Mode) 11 (Exception Code: Environment Call from M-Mode) 一般用不着, 保留给底层调用

操作系统的 trap handler 逻辑一般是:

  • 检查 scause 得知是 ecall
  • 通过 ctx.xregs[17] (a7 寄存器) 得知系统调用号
  • 通过 ctx.xregs[10] (a0 寄存器) 获取参数
  • 处理完成后通过 a0 返回
  • 通过 sret 指令返回到用户程序

从高特权级返回: sret

sret (Supervisor Return) 也是一条特权指令, 用于从 Supervisor Mode (S-Mode, 通常是操作系统内核) 返回到更低特权级模式 (User Mode), 会恢复执行用户程序的上下文, 包括程序计数器, 特权级别等. 通常用于从 trap 中返回.

CPU 执行 sret 指令时, 通常会完成下面操作:

  1. 从 sepc 寄存器中读取地址, 设置为新的 PC, 即返回到发生 trap 前用户程序执行的下一条指令
  2. 将当前 privilege 从 S-Mode 返回到之前的 privilege, 该信息一般保存在 sstatus 中的 SPP (Supervisor Previous Privilege) 位
  3. 恢复 interrupt enable 状态, 控制返回后是否允许中断
  4. 继续执行用户程序

另一类似指令是 mret (Machine Return), 从 M-Mode 返回到 S-Mode 或 U-Mode.

特权操作

特权操作包含特权指令以及 CSR 读写操作, 如:

  • mret
  • sret
  • wfi (Wait For Interrupt), 等待中断, 进入省电状态, 中断到来才继续干活
  • sfense.vma, 虚拟地址屏障指令
  • fense.i, i-cache 屏障指令

其他很多特权指令是通过读写 CSR 寄存器来实现的.

M-Mode

M-Mode (Machine Mode) 是 RISC-V 中 hart (hardware thread) 的最高权限模式, 有对计算机系统的底层功能的完全使用权.

M-Mode 最重要的特性是拦截和处理异常与中断:

  • 异常是同步的: 在执行期间产生, 比如访问无效寄存器地址或者执行了无效操作码的指令
  • 异步的中断: 指令流是异步的外部事件, 比如时钟中断
