UNIX-环境高级编程-Notes

线程和线程 ID

一个进程内的所有线程共享同一地址空间, 文件描述符, 栈以及与进程相关的属性. 因为它们能访问同一存储区, 所以各线程在访问共享数据时需要采取同步措施以避免不一致性.

与进程相同, 线程也用 ID 标识. 但其只在所属的进程中起作用.

线程是在进程模型建立很久之后引入的, 使用另一套控制函数.

用户标识

每个用户都有一个唯一的数值 ID 标识 (uid), 内核通过该 ID 来检验用户是否有执行某些操作的权限.

用户 ID 为 0 的为 root 用户或 superuser, 如果一个进程有 root 权限, 则大多数文件权限检查都不再进行.

每个用户也会包括 组 ID (group id, gid), 允许各个成员之间共享资源 (如文件).

信号

信号, Signal, 用于通知进程发生了某种情况. 当向一个进程发送信号时, 我们必须是那个进程的所有者或者是超级用户.

终端上产生信号的方法一般有:

  • Ctrl+C, Ctrl+\, delete 用于中断当前运行的进程
  • kill 发送指定信号

常见的信号有:

  • SIGINT, 值为 2, 终止进程
  • SIGQUIT, 值为 3, 终止进程 + core dump
  • SIGKILL, 值为 9, 立即终止
  • SIGTERM, 值为 15, 请求终止

可以用 signal 函数捕获指定信号, 并调用一个函数.

  • signum 是要捕获的信号
  • handler 是信号处理函数, 可以是:
    • 自定义函数, 信号发射时调用, 接收信号编号作为参数
    • SIG_IGN, 忽略这个信号
    • SIG_DFL, 使用系统默认行为

返回值:

  • 成功时, 返回信号的处理方式, 可能是一个函数指针, 或 SIG_IGN, SIG_DFL
  • 失败时返回 SIG_ERR, 并设置 errno

时间值

UNIX 使用过两种不同的时间值:

  • 日历时间, 从 1970 年 1 月 1 日 00:00:00 到当前积累的秒数, 一般用于记录文件最近一次修改时间, 通常用 time_t 类型保存
  • 进程时间, 也称 CPU 时间, 用于度量进程使用的 CPU 资源, 以时钟滴答计算, 通常用 clock_t 来保存

UNIX 会为一个进程维护 3 个进程时间值:

  • 时钟时间, 指进程运行的时间总量
  • 用户 CPU 时间, 指执行用户指令所用的时间量
  • 系统 CPU 时间, 指为了执行该进程内核程序所经历的时间

用户 CPU 时间和系统 CPU 时间之和称 “CPU 时间”.

限制

限制分两种:

  • 编译时限制 (比如, short 的最大值是什么)
  • 运行时限制 (比如, 文件名最多有多少字符)

编译时限制可以在头文件中定义, 程序在编译时包含这些头文件.

运行时限制需要进程调用一个函数来获得限制值.

为了解决这些问题, 提供了 3 种限制查询方法:

  1. 编译时限制 (头文件)
  2. 与文件或目录无关的运行时限制 (sysconf 函数)
  3. 与文件或目录有关的运行时限制 (pathconf 和 fpathconf 函数)

文件 I/O

文件 I/O 相关接口定义在 <fcntl.h> (“File Control”) 头文件中.

open 和 openat 函数返回的文件描述符一定是最小的未用描述符数值.

几个基础接口:

close 函数用于关闭一个打开文件, 但有时不会显式调用, 因为当一个进程终止时, 内核会自动关闭它所有的打开文件.

lseek (l 是 “long” 长整型的意思, 是过去接口的返回值) 函数用于显式为一个打开文件设置偏移量:

  • 通常, 文件的读, 写操作都从当前文件偏移量处开始
  • 默认情况下, 当打开一个文件时, 除非指定 O_APPEND 选项, 否则偏移量都会设置为 0
  • whence 的设置决定如何解析 offset 值:
    • SEEK_SET, 将文件的偏移量设置为距文件开始处 offset 个字节
    • SEEK_CUR, 将该文件的偏移量设置为其当前值加 offset, offset 可为正或负
    • SEEK_END, 将该文件的偏移量设置为文件长度加 offset, offset可正可负

返回值:

  • 成功时, 返回新的文件偏移量
  • 失败时, 返回 -1

文件偏移量可以大于文件长度, 这种情况下, 下次 “写” 会加长该文件, 并在文件中形成空洞, 中间没有写过的字节都读为 0. 通常这些空洞不会占据磁盘块.

read 函数从打开的文件中读数据:

  • 返回读到的字节数, 如果到文件尾则返回 1, 出错返回 -1

write 函数向打开文件写数据:

  • 返回已写的字节数, 若出错则返回 -1

由于磁盘块一般分为 4kb, 因此用于读写的 buf_size 设为 4096 比较好.

文件共享

UNIX 系统支持在不同进程见共享打开文件. 这里需要先了解内核表示打开文件的 3 种方式:

  1. 每个进程在进程表中都有一个记录项, 记录项中包含一张打开文件描述符表, 每个描述符占用一项. 与每个文件描述符相关联的有:
    a. 文件描述符标志 (如 close_on_exec)
    b. 指向一个文件表项的指针
  2. 内核为所有打开文件维持一张文件表. 每个文件表项包含:
    a. 文件状态标志 (读, 写, 追加, 同步, 非阻塞等)
    b. 当前文件偏移量
    c. 指向该文件 v 节点表项的指针
  3. 每个打开文件都有一个 v-node 结构, 包含文件类型和对此文件进行各种操作函数的指针. 对于大多数文件, v-node 还包含该文件的 i-node (index node, 索引节点). 这些信息会在打开文件时从磁盘读入内存

Linux 中没有使用 v 节点, 而是使用了通用 i-node 结构, 虽然实现不同, 但在概念上是一样的, 都指向文件系统特有的 i-node 结构.

原子操作

任何要求多于一个函数调用的操作都不是原子操作, 因为在两个函数调用之间, 内核有可能会临时挂起进程.

比如:

  • 假设两个进程同时修改一个文件
  • 两个进程都用 lseek 定位到文件末尾然后 write 100 个字节
  • 假设进程 A lseek 到文件末尾后被内核挂起, 进程 B 也 lseek 到末尾
  • 此时恢复进程 A, 其写入 100 字节后, 文件长度发生了变化, 但 B 并不会定位到新的末尾
  • 此时进程 B 再写入则会覆盖 A 写入的内容

这里一个进程包含两个系统调用, 但在执行完成一个系统调用后可能就会被挂起, 因此不是原子的.

pread 相当于把 lseek 和 read 合并为一个系统调用:

  • 若成功, 返回值是读到的字节数, 若到文件末尾则返回 0, 若出错则返回 -1

pwrite 相当于把 lseek 和 write 合并为一个系统调用:

  • 若成功, 返回写入的字节数, 若出错则返回 -1

sync 和 fsync 和 fdatasync

传统 UNIX 系统实现在内核中设有缓冲区高速缓存或页高速缓存, 大多数的磁盘 I/O 都会通过缓冲区进行.

当我们向文件写入数据时, 内核通常先将数据复制到缓冲区, 然后排入队列, 晚些时候再写入磁盘. 这种策略称 delayed write (延迟写).

当内核需要再次复制数据到缓冲区时, 它会先将延迟写数据块写入磁盘.

UNIX 系统提供了 sync, fsync, fdatasync 来保证磁盘上实际文件系统与缓冲区中内容的一致:

  • 对于有返回值的: 若成功, 返回 0; 若出错, 返回 -1
  • sync 只是将修改过的块缓冲区排入写队列, 然后返回, 而不会等待磁盘操作结束
  • fsync 对指定文件描述符 fd 对应的文件起作用, 且会等待写磁盘操作结束才返回
  • fdatasync 与 fsync 类似, 但只影响文件的数据部分, fsync 还会更新文件的属性

fcntl

fcntl (“file control”) 可以改变已经打开文件的属性:

  • 若成功, 返回依赖于 cmd; 若失败, 返回 -1

fcntl 的功能有:

  1. 复制一个已有的描述符 (cmd=F_DUPFD 或 F_DUPFD_CLOESEC)
  2. 获取/设置文件描述符标志 (cmd=F_GETFD 或 F_SETFL)
  3. 获取/设置文件状态标志 (cmd=F_GETFL 或 F_SETFL)
  4. 获取/设置异步 I/O 所有权 (cmd=F_GETOWN 或 F_SETOWN)
  5. 获取/设置记录锁 (cmd=F_GETLK, F_SETLK 或 F_SETLKW)

在修改文件描述符或文件状态标志时需要谨慎, 先获得现在的标志值, 再修改为新的标志值.

只执行 F_SETFD 或 F_SETFL 命令会关闭以前设置的标志位.

文件类型

UNIX 系统中文件类型包括:

  1. Regular file (普通文件), 包含某种形式的数据 (不管是二进制还是文本).
  2. Directory file (目录文件), 包含其他文件名字以及指向与这些文件有关信息的指针. 只有内核可以直接写目录文件.
  3. Block special file (块特殊文件), 提供对设备 (如磁盘) 带缓冲的访问, 每次访问以固定长度为单位进行
  4. Character special file (字符特殊文件), 提供对设备不带缓冲的访问, 每次访问长度可变. 系统中的设备要么是字符特殊文件, 要么是块特殊文件
  5. FIFO, 也称 named pipe, 用于进程间通信
  6. Socket (套接字), 用于进程间的网络通信
  7. Symbolic link (符号链接), 指向另一个文件

文件类型信息包含在 stat 结构的 st_mode 成员中, 并作为一些宏的参数来判断文件类型:

  • S_ISREG(), 是否为普通文件
  • S_ISDIR(), 目录文件
  • S_ISCHR(), 字符特殊文件
  • S_ISBLK(), 块特殊文件
  • S_ISFIFO(), 管道或 FIFO
  • S_ISLNK(), 符号链接
  • S_ISSOCK(), 套接字

Set-user-id 和 set-group-id

与一个进程相关联的 ID 有 6 个或更多, 如:

  • Real User ID (RUID, 实际用户 ID), 启动该进程的用户 ID
  • Real Group ID (RGID, 实际组 ID), 启动该进程的用户所属的 primary group 的 ID
    (表示是谁启动了职责个进程)
  • Effective User ID (EUID, 有效用户 ID), 权限判断是看到的是 EUID 而不是 RUID, 一个 Set-UID 程序会临时将 EUID 设置为文件拥有者的 UID 以获取更高权限
  • Effective Group ID (EGID, 有效组 ID)
  • Supplementary Group IDs (SGIDs, 附属组 ID)
    (用于文件访问权限检查)
  • Saved Set-User-ID (SSUID, 保存的设置用户 ID), 在程序设置了 Set-UID 运行时, 保存原来的 EUID
  • Saved Set-Group-ID (SSGID 保存的设置组 ID)
    (由 exec 函数保存)

stat 结构中的 st_mode 字段中, 可以设置一个特殊标志位 (Set-User-ID 位), 实现:

  • 当执行此文件时, 将进程的 Effective user id 设置为文件所有者的 user id (st_uid)

另一个特殊标志位 (Set-Group-ID 位), 实现:

  • 当执行此文件时, 将进程的 Effective group id 设置为文件所有者的 group id (st_gid)

文件访问权限

st_mode 值也包含了对文件的访问权限位, 每个文件有 9 个访问权限位 (分为 3 组):

  • S_IRUSR, 用户读 (文件所有者)
  • S_IWUSR, 用户写
  • S_IXUSR, 用户执行
  • S_IRGRP, 组读 (文件所有者所在 primary group)
  • S_IRGPR, 组写
  • S_IXGRP, 组执行
  • S_IROTH, 其他读
  • S_IROTH, 其他写
  • S_IXOTH, 其他执行

对于普通文件, 执行权限 (x) 表示该文件可以被执行 (作为程序或脚本).

对于目录文件而言, 执行权限 (x) 与普通文件的含义不同, 被称为搜索位. 决定: 是否允许 “进入” (cd) 该目录, 访问其内部的文件元数据 (比如文件内容), 或者搜索其下的内容. 此时读权限 (r) 决定是否能列出目录下的文件名列表 (ls). 写权限 (w) 决定能否在目录中添加, 删除, 重命名文件.

标准 I/O 库

流和 FILE 对象

之前的 I/O 操作都是借助文件描述符, 而标准 I/O 围绕流 (stream) 进行.

用标准 I/O 库打开或创建一个文件时, 就让一个流与一个文件相关联了.

流的定向 (stream’s orientation) 指读, 写的字符是单字节还是多字节. 当一个流最初被创建时, 其并没有定向:

  • 若在未定向的流上使用一个多字节 I/O 函数, 则会将该流的定向设置为宽定向的.
  • 若在未定向的流上使用一个单字节 I/O 函数, 则会将该流的定向设置为单字节定向的.

只有两个函数可以改变流的定向:

  • freopen 清除一个流的定向
  • fwide 设置一个流的定向, 但注意, 其不会改变已定向流的定向
  • 若 mode 值为负, fwide 将试图使指定流为字节定向的
  • 若 mode 值为正, fwide 会试图使指定流为宽定向的
  • 若 mode 值为 0, fwide 将不试图设置流的定向, 但返回标识该流定向的值
  • 返回值: 若流是宽定向, 返回正值; 若流是字节定向的, 返回负值; 若流是未定向的, 返回 0

用 fopen 打开一个流时, 其返回一个指向 FILE 对象的指针, 其包含标准 I/O 库为管理该流需要的所有信息, 如用于实际 I/O 的文件描述符, 指向用于该流缓冲区的指针, 缓冲区的长度, 当前在缓冲区中的字符数以及出错指标等.

引用一个流就是指, 将 FILE 指针 (通常也叫文件指针) 传递给标准 I/O 函数.

每个进程都预定义了 3 个流, 其对应的文件指针定义在 <stdio.h> 中, 即 stdin (标准输入), stdout (标准输出), stderr (标准错误).

缓冲

标准 I/O 提供缓冲的目的是尽可能减少使用 read 和 write 调用的次数.

标准 I/O 提供了 3 种类型的缓冲:

  1. 全缓冲. 在填满标准 I/O 缓冲区后才进行实际 I/O 操作. 也可以手动 flash, 将缓冲区的内容写到磁盘上
  2. 行缓冲. 当在输入和输出中遇到换行符时, 标准 I/O 库执行实际 I/O 操作, 但有两个限制:
    a. 标准 I/O 库用来收集每一行的缓冲区的长度是固定的, 只要填满了缓冲区, 即使没有写一个换行符, 也会进行 I/O 操作
    b. 标准 I/O 库对一些流的读取会 flash 所有行缓冲输出流. 比如读取不带缓冲的流 (此时要从内核获取数据) 或者读取一个带行缓冲的流 (需要从内核获取数据时)
  3. 不带缓冲. 标准 I/O 库不对字符进行缓冲存储. 此时写入到流中的字符会立刻输出.

进程标识

每个进程都有一个非负整型表示的唯一进程 ID. 该 ID 是唯一的, 也是可复用的. 当一个进程终止后, 该 ID 就成为复用的候选者.

大多数 UNIX 系统实现延迟复用算法, 使得赋予新建进程的 ID 不同于最近终止进程所使用的 ID, 这防止新进程被误认为是使用同一 ID 的某个已终止的先前进程.

ID 为 0 的进程通常是调度进程, 也被称为交换进程, 该进程是内核的一部分, 并不执行任何磁盘上的程序, 也称为系统进程.

  • /etc/inittab 文件
  • /etc/init.d 中的文件

函数 fork

一个进程可以调用 fork 函数来创建一个新进程:

  • 在子进程中返回 0, 父进程返回子进程 ID; 出错时返回 -1

由 fork 创建的新进程被称为 child process (子进程), 通常 fork 被调用一次会返回两次, 两次返回的区别是:

  • 子进程的返回值是 0, 因为一个进程只会有一个父进程, 可以用 getppid() 来获取其父进程 ID
  • 父进程的返回值是子进程的 PID, 因为一个进程可以创建多个子进程 (需要区分), 且没有函数可以直接获取所有子进程的进程 ID
  • 这里相当于父子进程都会从 fork() 返回后开始运行, 都会经过这个判断语句

子进程是父进程的副本, 也就是说, 会复制父进程的数据空间, 堆和栈. 但要注意, 会共享 text 段 (也就是代码部分).

由于在 fork 之后通常会执行 exec (用新程序替代当前程序, 包括数据空间, 堆, 栈等), 因此很多实现不会执行复制父进程的数据段等的副本 (复制了就被替换了, 相当于浪费了). 而是采用写时复制的模式 (Copy-on-Write, COW)

数据空间, 堆, 栈这些部分先共享, 内核将其访问权限改为只读, 当父进程和子进程中的任一个试图修改这个区域, 此时内核才会为这个区域制作一个副本.

一般来说, fork 之后父子进程的执行顺序是不确定的, 取决于内核使用的调度算法. 若想实现同步, 则需要实现父子进程间的通信.

fork 失败的原因通常有两个:

  1. 系统已经有太多的进程了
  2. 实际用户 ID 的进程总数超过了系统限制

操作系统通常会将 fork 之后执行 exec 组合成一个操作, 称为 spawn.

进程终止

当子进程终止时, 父进程可以通过 wait 或 waitpid 函数来取得其终止状态.

内核会为每个终止子进程保存一定量的信息, 所有当终止进程的父进程调用 wait 或 waitpid 时, 可以得到这些信息. 这信息至少包括: 进程 ID, 该进程的终止状态, 以及该进程使用的 CPU 时间总量. 之后内核会释放终止进程所使用的所有存储区, 关闭其所有打开的文件.

这里的重点是,, 一个子进程终止时:

  1. 进程的 PCB 和大部分资源会被回收
  2. 但子进程的内存映像 (堆栈, 数据段等) 和退出状态信息, 不会马上释放, 需要等待父进程拉取后再释放

如果在子进程终止前, 父进程已经终止了, 那么这些子进程的父进程会改变为 init 进程. 称这些进程由 init 进程收养.

在 UNIX 术语中, 一个已经终止, 但是其父进程没有对其进行善后处理 (获取终止子进程的有关信息, 释放其占用的资源) 的进程, 被称为僵尸进程 (zombie).

信号是一种软件中断. 信号往往是异步事件 (一个会产生信号的事件对进程而言是随机的), 可以通过捕获信号来处理.

每个信号都有一个名字, 以 SIG 这 3 个字符开头. 比如 SIGABRT, SIGALRM.

在头文件 <signal.h> 中, 信号名都被定义为正整数常量.

注意不存在编号为 0 的信号, 这种信号称空信号, 有特殊的作用.

产生信号的方式有:

  • 用户的按键触发
  • 硬件产生
  • kill 函数
  • kill 命令
  • 等

可靠信号术语

当造成信号的事件发生时, 称 “为进程产生了一个信号”. 比如发生了硬件异常 (如除零), 软件条件 (alarm 超时), 终端调用 kill 等. 当一个信号产生时, 内核会在进程表中以某种形式设置一个标志.

当进程对信号采取措施时, 称 “向进程传递了一个信号”.

在 “产生 (generation)” 和 “传递 (delivery)” 之间的时间间隔, 称信号是 “未决的 (pending)”.

进程可以选择 “阻塞信号的传递”, 也就是在信号 “产生” 后, 如果对该信号的动作是系统默认动作或捕获该信号, 则维持 “pending” 的状态. 直到进程对此信号解除了阻塞或者设置为忽略.

在一个信号阻塞时, 如果产生了多次, 大多数 UNIX 都认为信号只传递了一次而非排队.

总结来看, 就是信号有 3 个状态:

  • 产生
  • pending
  • 传递 (执行处理函数)

在 pending 时可以改变对该信号的处理方式.

每个进程都有一个信号屏蔽字 (signal mask), 它规定了当前要阻塞递送到该进程的信号集.

函数 kill 和 raise

kill 函数将信号发送给进程或进程组.

raise 函数允许进程向自身发送信号.

  • 成功时返回 0; 出错时返回 -1

调用 raise(signo) 等价于 kill(getpid(), signo).

kill 的 pid 参数有以下 4 种情况:

  • pid > 0, 发送给进程 ID 为 pid 的进程
  • pid == 0, 发送给, 与发送进程属于同一进程组的所有进程. 需要保证发送进程具有权限向这些进程发送信号
  • pid < 0, 发送给, 进程组 ID 等于 pid 绝对值的进程组. 发送进程需要有权限向其发送信号
  • pid == -1, 将信号发送给, 发送进程有权限向其发送信号的所有进程

对于这里的权限问题:

  • 超级用户可将信号发送给任一进程
  • 对于非超级用户, 发送者的实际用户 ID 或有效用户 ID 必须等于接收者的实际用户 ID 或有效用户 ID

权限测试有一个特例, 如果发送的信号是 SIGCONT, 则进程可将它发送给属于同一会话的任一其他进程.

发送空信号, 一般用来进行权限检查:

  • 如果目标进程不存在, 或没有权限, kill 会返回 -1 并设置 errno

函数 alarm 和 pause

使用 alarm 函数可以设置一个定时器, 在超时时产生 SIGALRM 信号. 如果忽略或不捕捉该信号, 默认的行为是终止调用该 alarm 函数的进程.

  • 返回 0 或以前设置的闹钟时间的余留秒数
  • 设置的时间到达时, 由内核产生信号, 但由于进程调度延迟, 当进程得到控制从而能够处理该信号还需要一个时间间隔
  • 每个进程同一时刻只能有一个闹钟时间, 如果在一个 alarm 没有结束时再次调用, 则之前闹钟的余留值作为本次 alarm 函数的返回值

pause 函数使进程挂起, 直到捕捉到一个信号:

  • 返回 -1, errno 设置为 EINTR
  • 只有在执行了一个信号处理程序并返回后, pause 才会返回

函数 sigsetjmp 和 siglongjmp

当捕捉到一个信号, 进入信号处理函数时, 该信号会被自动加入到进程的信号屏蔽字中, 以阻止后来又产生这种信号来中断当前信号处理程序.

如果在信号处理函数中, 使用 longjmp 跳出, 则不会自动恢复该信号的屏蔽.

函数 sigsuspend

sigsuspend 能在一个原子操作中, 先设置信号屏蔽字, 然后使程序休眠 (pause, 等待一个信号):

  • 在捕捉到一个信号或发生了一个会终止该进程的信号之前, 该进程被挂起

函数 abort

  • 将 SIGABRT 信号发送给调用进程, 表示程序异常终止
  • 通常需要在 abort 的信号处理函数中调用 exit 等来退出

线程

一个进程中的所有线程都可以访问该进程的所有组成部件, 比如文件描述符和内存.

线程概念

典型的 UNIX 进程可以看作是只有一个控制线程: 一个进程在某一时间只能做一件事.

有多个控制线程, 表示进程在某一时刻能做多件事, 每个线程处理各自独立的任务.

每个线程都包含有表示执行环境所必须的信息, 包括:

  • 标识线程的线程 ID
  • 一组寄存器值
  • 栈
  • 调度优先级和策略
  • 信号屏蔽字
  • errno 变量
  • 线程私有数据等

线程标识

每个线程都有一个线程 ID, 但其只在其所属的进程上下文中有意义.

线程 ID 用 pthread_t 数据类型来表示, 通常是一个 struct 而不是整数, 因此需要用函数来比较:

  • 相等时返回非零数; 否则返回 0

线程可以通过调用 pthread_self 函数来获得自身的线程 ID:

  • 返回线程的线程 ID

线程创建

程序开始运行时, 是以单进程中的单个控制线程启动的, 此时与传统的进程并无区别.

可以用 pthread_create 函数来创建线程:

  • 成功时返回 0; 否则返回错误码
  • 新线程的 ID 保存在 tidp 指向的内存空间
  • attr 用于设置线程属性
  • 新创建的线程从 start_rtn 函数的地址开始运行
  • 如果 start_rtn 需要超过 1 个参数, 则放在 arg 结构体中

线程创建并不能保证是哪个线程线运行, 是新创建的线程还是调用线程.

新创建的线程可以访问进程的地址空间, 并且继承调用线程的浮点环境和信号屏蔽字, 但是挂起信号集会被清除.

线程终止

如果进程中的任意线程调用了 exit, _Exit 或者 _exit, 那么整个进程都会被终止.

线程可以有 3 中方式退出 (不终止整个进程的情况下停止该控制流):

  1. 在 start_rtn 中用 return 返回
  2. 被同一进程中的其他线程取消
  3. 调用 pthread_exit

进程中的其他线程可以通过 pthread_join 函数来访问这个 rval_ptr 指针:

  • 成功时返回 0; 否则返回错误编号
  • 调用线程会一直阻塞, 知道指定的线程调用 pthread_exit 或者返回或取消
  • 如果线程被取消, 那么 rval_ptr 指向的内存单元被设置为 PTHREAD_CANCELED

需要注意, pthread_exit 中的 rval_ptr 可以包含复杂的信息, 但要保证其他线程获取到这个值时其仍然有效, 比如在栈上分配可能会导致该栈已经弹出, 内存用作其他. 可以考虑使用全局变量或者在堆上分配.

线程可以调用 pthread_cancel 来请求取消同一进程中的其他线程:

  • 成功时返回 0; 否则返回错误编号
  • 默认情况下, tid 对应的线程会表现得像调用了参数为 PTHREAD_CANCELD 的 pthread_exit 函数
  • 线程可以选择忽略取消或控制如何取消
  • 该函数也不会等待线程终止, 只是发出一个请求

线程也可以设置退出时自动调用的函数, 类似 atexit 函数. 这些函数称线程清理处理程序, 同样可以建立多个, 处理函数会记录在栈中, 因此调用顺序与注册时相反:

下面几种情况会触发 cleanup 函数的调用:

  • 调用 pthread_exit 时
  • 响应取消时
  • 用非零的 execute 参数调用 pthread_cleanup_pop 时

pthread_cleanup_pop 会删除上次 pthread_cleanup_push 添加的一个清理处理程序, 当 execute 为 0 时, 删除时不会调用该清理处理程序.

默认情况下, 线程的终止状态会保存直到对该线程调用 pthread_join. 如果对线程的终止状态不感兴趣, 可以让其处于分离状态. 如果线程已经被分离, 线程的底层存储资源可以在终止时立即被回收, 此时也不能用 pthread_join 函数来等待其终止状态:

  • 成功时返回 0; 出错时返回错误编号

线程同步

当多个控制线程共享相同的内存时, 需要确保每个线程看到的数据视图是一致的.

假设线程 A 要写一个变量, 但是写操作被分为多个存储周期:

  1. 先读取 x 的当前值
  2. 计算新值
  3. 把新值写回 x 的内存, 但是 “写回操作” 并不是一下子完成的, 分为了多个存储周期, 比如先写低字节, 再写高字节等

如果线程 B 在线程 A 写操作的中间时刻读取了这个变量 (比如只写了一半), 此时:

  • B 读取到的值, 既不是线程 A 修改前的旧值, 也不是修改后的新值, 而是一个 “中间状态”, 这种值通常是无意义的, 不符合预期, 会导致程序逻辑错误

为了解决这个问题, 可以使用锁, 同一时间只允许一个线程访问该变量.

两个线程在同一时间修改一个变量也需要考虑这个问题. 但如果修改操作是原子操作, 则不存在竞争.

互斥量

互斥量 (mutex), 本质上是一把锁, 在访问共享资源前对互斥量进行设置 (加锁), 在访问完成后进行释放 (解锁) .

对互斥量进行加锁后, 任何其他试图再次对互斥量加锁的线程都会被阻塞, 直到当前线程释放该互斥锁.

互斥变量用 pthread_mutex_t 数据类型表示. 使用互斥变量以前, 必须先对其进行初始化:

  • 设置为 PTHREAD_MUTEX_INITIALIZER 常量
  • 或通过调用 pthread_mutex_init 函数进行初始化

如果互斥变量是动态分配的, 则需要在释放内存前先调用 pthread_mutex_destroy.

  • 成功时返回 0; 出错时返回错误编号

对互斥量进行加锁和解锁:

  • 成功时返回 0; 出错时返回错误编号
  • 如果线程不希望被阻塞, 则可以用 pthread_mutex_trylock, 如果此时互斥量没有锁住, 则能够成功加锁; 否则失败, 返回 EBUSY

避免死锁

死锁指多个执行单元在互相等待对方释放资源, 彼此都无法继续进行, 导致相关执行单元都卡死, 系统无法向前推进的一种状态.

理论上, 死锁的产生有四个必要条件:

  1. 互斥条件 (Mutual Exclusion), 资源一次只能被一个执行单元占用, 比如一个互斥锁
  2. 占有并等待 (Hold and Wait), 某个执行单元已经持有了至少一个资源, 但同时又请求其他资源, 而该资源被其他执行单元占用
  3. 非抢占条件 (No Preemption), 已经分配的某个执行单元的资源, 不能被其他执行单元抢走, 只能由该执行单元主动释放
  4. 循环等待 (Circular Wait), 存在一个执行单元的循环等待链, 比如 A 等 B 的资源, B 等 C 的资源, C 等 A 的资源, 形成一个闭环

这样看, 发生死锁至少要有多份资源 (多个互斥量), 比如:

  • 线程 A 占有一个互斥量, 在试图锁住第二个互斥量时处于阻塞状态
  • 拥有第二个互斥量的线程也正在试图锁住第一个互斥量

由于两个线程都在请求另一个线程占有的资源, 因此都无法继续运行, 出现了死锁.

可以通过控制多个互斥量加锁的顺序来避免死锁, 比如有 A 和 B 两个互斥量, 如果所有线程都按照:

  • 先锁住 A 之后再锁住 B, 或
  • 先锁住 B 在锁住 A

此时就不会发生死锁.

函数 pthread_mutex_timedlock

pthread_mutex_timedlock 允许线程等待一个未加锁的互斥量时, 设置一个阻塞的时间, 如果到了超时时间还没加锁, 就会返回 ETIMEDOUT 错误码 (避免永久阻塞):

  • 成功时返回 0; 否则返回错误编号

读写锁

读写锁 (reader-writer lock), 与互斥量类似, 不过读写锁允许更高的并行性.

互斥量要么是锁住状态, 要么就是不加锁状态, 而且一次只有一个线程可以对其加锁.

读写锁可以有 3 种状态:

  • 读模式下加锁状态
  • 写模式下加锁状态
  • 不加锁状态

一次只有一个线程可以占有写模式的读写锁, 但是多个线程可以同时占有读模式的读写锁.

  • 当读写锁是写加锁状态时, 所有试图对这个锁加锁的线程都会被阻塞
  • 当读写锁是读加锁状态时, 所有试图以读模式对这个锁加锁的线程都可以得到访问权, 但是任何希望以写模式对此锁进行加锁的线程都会阻塞, 直到所有的线程释放它们的读锁为止

当读写锁处于读模式锁住的状态, 如果此时有一个线程试图以写模式获取锁时, 读写锁通常会阻塞随后的读模式锁. 这样可以避免读模式锁长期占用, 而等待的写模式锁请求一直得不到满足.

读写锁也可以叫做共享互斥锁 (shared-exclusive lock), 因为在读模式锁住, 可以说是以共享模式锁住. 而在写模式锁住, 则称是互斥模式锁住.

读写锁在使用之前必须先初始化, 在释放它们底层的内存之前必须销毁:

  • 成功时返回 0; 否则返回错误编号

在读模式下锁定读写锁, 用 pthread_rwlock_rdlock; 在写模式下锁定读写锁, 用 pthread_rwlock_wrlock; 都用 pthread_rwlock_unlock 进行解锁:

  • 成功时返回 0; 否则返回错误编号

不阻塞加锁的版本:

  • 成功时返回 0; 否则返回错误编号

带有超时的读写锁

  • 成功时返回 0; 否则返回错误编号

条件变量

条件变量是另一个多线程下的同步机制. 条件变量与互斥量一起使用时, 允许线程以无竞争的方式等待特定的条件发生.

条件变量本身由互斥量保护. 线程在改变条件状态之前必须先锁住互斥量.

同样, 使用条件变量之前, 需要先初始化. 条件变量用 pthread_cond_t 数据类型表示, 有两种初始化方法:

  • 设置为 PTHREAD_COND_INITIALIZER 常量
  • 用 pthread_cond_init 初始化

在释放条件变量底层的内存之前, 也需要先用 pthread_cond_destroy 函数来销毁:

  • 成功时返回 0; 否则返回错误编号

用 pthread_cond_wait 阻塞等待条件变量变为真, pthread_cond_timedwait 提供了超时时间, 如果在给定的时间内条件不能满足, 那么会生成一个返回错误码的变量:

  • 成功时返回 0; 否则返回错误编号
  • 这里传入的 mutex 会用来保护传入的 cond. 当调用者将加锁的互斥量传给函数后. 函数会先将调用线程加入到等待条件的线程列表中, 然后对互斥量进行解锁. 在 pthread_cond_wait 返回时, 互斥量会再次锁住

有两个函数用于通知线程条件已经满足:

  • pthread_cond_signal 函数至少能唤醒一个等待该条件的线程
  • pthread_cond_broadcast 函数能唤醒等待该条件的所有线程
  • 成功时返回 0; 否则返回错误编号

自旋锁

自旋锁与互斥量类似, 但它不是通过休眠使进程阻塞, 二十载获取锁之前一直处于忙等 (自旋) 阻塞状态. 通常用于锁被持有的时间短, 并且线程不希望在重新调度上花费太多成本的情况.

当线程自旋等待锁变为可用时, CPU 不能做其他事情 (这也是为什么用于锁被持有一小段时间的原因).

  • 成功时返回 0; 否则返回错误编号

可以用 pthread_spin_lock 或 pthread_spin_trylock 对自旋锁进行加锁, 前者在获取锁之前一直自旋, 后者如果不能获取锁, 就立即返回 EBUSY 错误. 两者都能用 pthread_spin_unlock 解锁:

  • 成功时返回 0; 否则返回错误编号

屏障

屏障 (barrier) 是另一种多线程的同步机制.

屏障允许每个线程等待, 直到所有合作线程都到达某个点, 然后从该点继续执行. 比如 pthread_join 函数就是一种屏障, 允许一个线程等待, 直到另一个线程退出.

屏障允许任意数量的线程等待, 直到所有线程完成处理工作, 而线程不需要退出. 在所有线程达到屏障后, 可以接着工作.

用 pthread_barrier_init 函数对屏障进行初始化, 用 thread_barrier_destroy 函数反初始化:

  • 成功时返回 0; 否则返回错误编号
  • count 参数制定, 在允许所有线程继续运行之前, 必须到达屏障的线程数目

可以用 pthread_barrier_wait 函数来表明, 线程已完成工作, 准备等所有其他线程赶上来:

  • 成功时返回 0 或 PTHREAD_BARRIER_SERIAL_THREAD; 否则返回错误编号

线程在调用 pthread_barrier_wait 时, 如果 count 没有达到, 则会进入休眠. 如果该线程是最后一个调用 pthread_barrier_wait 的线程, 满足 count 的同时会唤醒所有线程.

对于一个任意线程, 如果 pthread_barrier_wait 函数返回了 PTHREAD_BARRIER_SERIAL_THREAD. 那么剩下的线程看到的返回值都是 0. 这使得一个线程能够在其他线程完成的工作结果之上进行工作.

线程控制

线程属性

在使用 pthread_create 创建线程时, 如果传入的属性是 NULL 空指针, 表示使用默认的线程属性. 也可以用指向 pthread_attr_t 结构的指针, 来更改线程的默认属性.

可以用 pthread_attr_init 来初始化 pthread_attr_t 结构, 此时该结构包含所有线程属性的默认值:

  • 成功时返回 0; 出错时返回错误编号
  • 如果 pthread_attr_t 对象是动态分配的, 那么 pthread_attr_destroy 就会释放其空间, 除此之外, 其还会用无效的值初始化属性对让, 让 pthread_create 使用时返回错误码

如果对线程的终止状态不感兴趣, 可以设置其 detachstate 属性, 让线程从一开始就处于 detach 状态: PTHREAD_CREATE_DETACHED (以分离状态启动线程), PTHREAD_CREATE_JOINABLE (正常启动线程):

  • 成功时返回 0; 出错时返回错误编号

可以用 pthread_attr_getstack 和 pthread_attr_setstack 对线程栈属性进行管理:

  • 成功时返回 0; 出错时返回错误编号
  • 这用来处理线程栈空间不够, 通过 malloc 或 mmap 重新分配栈空间的情况, stackaddr 是新的栈尾

也可以读取和设置栈的大小:

  • 成功时返回 0; 出错时返回错误编号
  • 这个适用于希望改变默认的栈大小, 但不想自己分配的情况

进程间通信

管道

管道只能在具有公共祖先的两个进程之间使用. 通常, 一个管道由一个进程创建, 在进程调用 fork 之后, 这个管道就能在父进程和子进程之间使用了.

最常见的就是 shell 中, shell 会为每一条命令单独创建一个进程, 然后用管道将前一条命令的标准输出与后一条命令的标准输入相连接.

管道用 pipe 函数创建:

  • 成功时返回 0; 出错时返回 -1
  • 参数 fd 中的两个文件描述符, fd[0] 为读而打开, fd[1] 为写而打开
  • fd[1] 的输出是 fd[0] 的输入

如果想要子进程到父进程的管道, 则父进程关闭写端 (fd[1]), 子进程关闭读端 (fd[0]).

当管道的一端被关闭后, 有两条规则会起作用:

  1. 当 “读 (read)” 一个写端已经被关闭的管道时, 在所有数据都被读取后, read 返回 0, 表示文件结束
  2. 如果 “写 (write)” 一个读端已被关闭的管道, 则产生信号 SIGPIPE. 如果忽略该信号或者捕捉该信号并从其处理程序返回, 则 write 返回 -1, errno 设置为 EPIPE

在写管道时, 常量 PIPE_BUF 规定了内核的管道缓冲区大小. 如果对管道调用 write, 并且要求写的字节数小于等于 PIPE_BUF, 则此操作不会与其他进程对同一管道的 write 操作交叉进行.

如果多个进程同时写一个管道, 而且要求写的字节数超过 PIPE_BUF, 那么所写的数据可能会与其他进程所写的数据相互交叉.

FIFO

FIFO 称命名管道. 未命名的管道只能在两个相关的进程之间使用, 而且这两个相关的进程还要有一个共同的创建了它们的祖先进程. 而使用 FIFO, 可以让不相关的进程交换数据.

FIFO 本质上是一种文件类型, 可以通过 stat 结构的 st_mode 成员来判断其类型.

创建 FIFO, 其路径名会存在于文件系统中:

  • 成功时返回 0; 出错时返回 -1
  • mode 参数与 open 相同

用 mkfifo 和 mkfifoat 创建 FIFO 之后, 需要用 open 来打开使用.

当 open 一个 FIFO 时, 如果没有指定 O_NONBLOCK flag, 则:

  • 一个 Read Only 的 open 会阻塞, 直到某个进程为了 write 打开了这个 FIFO
  • 一个 Write Only 的 open 会阻塞, 直到某个进程为了 read 打开了这个 FIFO

如果指定了 O_NONBLOCK, 则:

  • Read Only 的 open 会立即返回, 如果没有进程为了 write 打开一个 FIFO, 那么 open 将返回 -1, 且将 errno 设为 ENXIO

FIFO 有以下两种用途:

  1. shell 命令使用 FIFO 将数据从一条管道传送到另一条时, 无需创建中间临时文件
  2. 客户进程-服务进程应用程序中, FIFO 用作汇聚点, 在客户进程和服务进程两者之间传递数据

UNIX 域套接字

UNIX 域套接字可以让统一计算机系统上运行的两个进程之间传送打开的文件描述符. 其仅仅复制数据, 不执行协议处理, 不添加或删除网络报头等, 因此效率较高.

UNIX 域套接字提供流和数据报两种接口. 其类似套接字和管道的混合:

  • SOCK_STREAM, 类似 TCP, 提供面向连接的, 可靠的, 双向的字节流通信
  • SOCK_DGRAM, 类似于 UDP, 提供无连接的, 不可靠的, 但保留消息边界的数据传输

可以用 socketpair 函数来创建一对无命名的, 相互连接的 UNIX 域套接字:

  • 成功时返回 0; 出错时返回 -1
  • sockfd[2] 是一对相互连接的套接字, 从 sockfd[0] 写入的数据可以从 sockfd[1] 中读取
