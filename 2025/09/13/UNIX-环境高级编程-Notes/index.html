UNIX-环境高级编程-Notes

线程和线程 ID

一个进程内的所有线程共享同一地址空间, 文件描述符, 栈以及与进程相关的属性. 因为它们能访问同一存储区, 所以各线程在访问共享数据时需要采取同步措施以避免不一致性.

与进程相同, 线程也用 ID 标识. 但其只在所属的进程中起作用.

线程是在进程模型建立很久之后引入的, 使用另一套控制函数.

errno 的相关常量定义在 <errno.h> 头文件中. 对于 errno 需要注意两个规则:

  • 如果没有出错, 则 errno 的值不会被例程清除, 因此需要在函数返回后就检验是否出错, 不然 errno 中可能存储的是以前的错误信息

用户标识

每个用户都有一个唯一的数值 ID 标识 (uid), 内核通过该 ID 来检验用户是否有执行某些操作的权限.

用户 ID 为 0 的为 root 用户或 superuser, 如果一个进程有 root 权限, 则大多数文件权限检查都不再进行.

每个用户也会包括 组 ID (group id, gid), 允许各个成员之间共享资源 (如文件).

信号

信号, Signal, 用于通知进程发生了某种情况. 当向一个进程发送信号时, 我们必须是那个进程的所有者或者是超级用户.

终端上产生信号的方法一般有:

  • Ctrl+C, Ctrl+\, delete 用于中断当前运行的进程
  • kill 发送指定信号

常见的信号有:

  • SIGINT, 值为 2, 终止进程
  • SIGQUIT, 值为 3, 终止进程 + core dump
  • SIGKILL, 值为 9, 立即终止
  • SIGTERM, 值为 15, 请求终止

可以用 signal 函数捕获指定信号, 并调用一个函数.

  • signum 是要捕获的信号
  • handler 是信号处理函数, 可以是:
    • 自定义函数, 信号发射时调用, 接收信号编号作为参数
    • SIG_IGN, 忽略这个信号
    • SIG_DFL, 使用系统默认行为

返回值:

  • 成功时, 返回信号的处理方式, 可能是一个函数指针, 或 SIG_IGN, SIG_DFL
  • 失败时返回 SIG_ERR, 并设置 errno

时间值

UNIX 使用过两种不同的时间值:

  • 日历时间, 从 1970 年 1 月 1 日 00:00:00 到当前积累的秒数, 一般用于记录文件最近一次修改时间, 通常用 time_t 类型保存
  • 进程时间, 也称 CPU 时间, 用于度量进程使用的 CPU 资源, 以时钟滴答计算, 通常用 clock_t 来保存

UNIX 会为一个进程维护 3 个进程时间值:

  • 时钟时间, 指进程运行的时间总量
  • 用户 CPU 时间, 指执行用户指令所用的时间量
  • 系统 CPU 时间, 指为了执行该进程内核程序所经历的时间

用户 CPU 时间和系统 CPU 时间之和称 “CPU 时间”.

限制

限制分两种:

  • 编译时限制 (比如, short 的最大值是什么)
  • 运行时限制 (比如, 文件名最多有多少字符)

编译时限制可以在头文件中定义, 程序在编译时包含这些头文件.

运行时限制需要进程调用一个函数来获得限制值.

为了解决这些问题, 提供了 3 种限制查询方法:

  1. 编译时限制 (头文件)
  2. 与文件或目录无关的运行时限制 (sysconf 函数)
  3. 与文件或目录有关的运行时限制 (pathconf 和 fpathconf 函数)

运行时查看环境限制

可以用 sysconf, pathconf, fpathconf 三个函数在运行时查询系统运行环境中的各种配置限制或参数, 比如:

  • 文件系统支持的最大路径长度
  • 一个进程最多可以打开多少个文件
  • 当前系统允许的线程数, 进程数, 文件大小限制
  • 等等

这些限制都不是编译时常量, 而是运行时由操作系统决定的配置值.

sysconf

sysconf 用于查询系统级别的配置限制 (与整个操作系统相关), 比如一个进程最多能打开多少文件, 其函数原型为:

  • name 是一个预定义的宏, 即要查询的配置参数
  • 成功时返回对应整数, 出错时返回 -1

常见的 name 参数定义在 <unistd.h> 中, 比如:

  • _SC_ARG_MAX, 命令行参数的最大总长度 (包括环境变量)
  • _SC_CHILD_MAX, 每个用户 ID 最多可拥有的子进程数
  • _SC_OPEN_MAX, 一个进程最多可打开的文件描述符数目
  • _SC_PAGESIZE, 系统内存页大小
  • 等等

可以在 man 3 sysconf 中查看.

Single UNIX Specification

Single UNIX Specification 是 POSIX.1 标准的一个超集, 定义了一些附加接口.

Open Group 拥有 UNIX 商标, 他们使用 Single UNIX Specification 定义了一系列接口, 一个系统想要称 UNIX 系统, 其实现必须支持这些接口并通过验证测试.

文件 I/O

文件 I/O 相关接口定义在 <fcntl.h> (“File Control”) 头文件中.

open 和 openat 函数返回的文件描述符一定是最小的未用描述符数值.

几个基础接口:

close 函数用于关闭一个打开文件, 但有时不会显式调用, 因为当一个进程终止时, 内核会自动关闭它所有的打开文件.

lseek (l 是 “long” 长整型的意思, 是过去接口的返回值) 函数用于显式为一个打开文件设置偏移量:

  • 通常, 文件的读, 写操作都从当前文件偏移量处开始
  • 默认情况下, 当打开一个文件时, 除非指定 O_APPEND 选项, 否则偏移量都会设置为 0
  • whence 的设置决定如何解析 offset 值:
    • SEEK_SET, 将文件的偏移量设置为距文件开始处 offset 个字节
    • SEEK_CUR, 将该文件的偏移量设置为其当前值加 offset, offset 可为正或负
    • SEEK_END, 将该文件的偏移量设置为文件长度加 offset, offset可正可负

返回值:

  • 成功时, 返回新的文件偏移量
  • 失败时, 返回 -1

文件偏移量可以大于文件长度, 这种情况下, 下次 “写” 会加长该文件, 并在文件中形成空洞, 中间没有写过的字节都读为 0. 通常这些空洞不会占据磁盘块.

read 函数从打开的文件中读数据:

  • 返回读到的字节数, 如果到文件尾则返回 1, 出错返回 -1

write 函数向打开文件写数据:

  • 返回已写的字节数, 若出错则返回 -1

由于磁盘块一般分为 4kb, 因此用于读写的 buf_size 设为 4096 比较好.

文件共享

UNIX 系统支持在不同进程见共享打开文件. 这里需要先了解内核表示打开文件的 3 种方式:

  1. 每个进程在进程表中都有一个记录项, 记录项中包含一张打开文件描述符表, 每个描述符占用一项. 与每个文件描述符相关联的有:
    a. 文件描述符标志 (如 close_on_exec)
    b. 指向一个文件表项的指针
  2. 内核为所有打开文件维持一张文件表. 每个文件表项包含:
    a. 文件状态标志 (读, 写, 追加, 同步, 非阻塞等)
    b. 当前文件偏移量
    c. 指向该文件 v 节点表项的指针
  3. 每个打开文件都有一个 v-node 结构, 包含文件类型和对此文件进行各种操作函数的指针. 对于大多数文件, v-node 还包含该文件的 i-node (index node, 索引节点). 这些信息会在打开文件时从磁盘读入内存

Linux 中没有使用 v 节点, 而是使用了通用 i-node 结构, 虽然实现不同, 但在概念上是一样的, 都指向文件系统特有的 i-node 结构.

原子操作

任何要求多于一个函数调用的操作都不是原子操作, 因为在两个函数调用之间, 内核有可能会临时挂起进程.

比如:

  • 假设两个进程同时修改一个文件
  • 两个进程都用 lseek 定位到文件末尾然后 write 100 个字节
  • 假设进程 A lseek 到文件末尾后被内核挂起, 进程 B 也 lseek 到末尾
  • 此时恢复进程 A, 其写入 100 字节后, 文件长度发生了变化, 但 B 并不会定位到新的末尾
  • 此时进程 B 再写入则会覆盖 A 写入的内容

这里一个进程包含两个系统调用, 但在执行完成一个系统调用后可能就会被挂起, 因此不是原子的.

pread 相当于把 lseek 和 read 合并为一个系统调用:

  • 若成功, 返回值是读到的字节数, 若到文件末尾则返回 0, 若出错则返回 -1

pwrite 相当于把 lseek 和 write 合并为一个系统调用:

  • 若成功, 返回写入的字节数, 若出错则返回 -1

dup2 可以用 fd2 指定一个新描述符的值.

sync 和 fsync 和 fdatasync

传统 UNIX 系统实现在内核中设有缓冲区高速缓存或页高速缓存, 大多数的磁盘 I/O 都会通过缓冲区进行.

当我们向文件写入数据时, 内核通常先将数据复制到缓冲区, 然后排入队列, 晚些时候再写入磁盘. 这种策略称 delayed write (延迟写).

当内核需要再次复制数据到缓冲区时, 它会先将延迟写数据块写入磁盘.

UNIX 系统提供了 sync, fsync, fdatasync 来保证磁盘上实际文件系统与缓冲区中内容的一致:

  • 对于有返回值的: 若成功, 返回 0; 若出错, 返回 -1
  • sync 只是将修改过的块缓冲区排入写队列, 然后返回, 而不会等待磁盘操作结束
  • fsync 对指定文件描述符 fd 对应的文件起作用, 且会等待写磁盘操作结束才返回
  • fdatasync 与 fsync 类似, 但只影响文件的数据部分, fsync 还会更新文件的属性

fcntl

fcntl (“file control”) 可以改变已经打开文件的属性:

  • 若成功, 返回依赖于 cmd; 若失败, 返回 -1

fcntl 的功能有:

  1. 复制一个已有的描述符 (cmd=F_DUPFD 或 F_DUPFD_CLOESEC)
  2. 获取/设置文件描述符标志 (cmd=F_GETFD 或 F_SETFL)
  3. 获取/设置文件状态标志 (cmd=F_GETFL 或 F_SETFL)
  4. 获取/设置异步 I/O 所有权 (cmd=F_GETOWN 或 F_SETOWN)
  5. 获取/设置记录锁 (cmd=F_GETLK, F_SETLK 或 F_SETLKW)

在修改文件描述符或文件状态标志时需要谨慎, 先获得现在的标志值, 再修改为新的标志值.

只执行 F_SETFD 或 F_SETFL 命令会关闭以前设置的标志位.
