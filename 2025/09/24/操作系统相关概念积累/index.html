虚拟地址

虚拟地址 (Virtual Address), 指程序运行视角看到的地址, 并不是物理内存中的真实地址.

比如一个 C 程序声明了一个变量或者调用 malloc() 分配了一块内存, 其得到的地址就是 virtual address, 并不是直接对应计算机物理内存条上的某个位置.

  • 56474469c000-56474469d000 是虚拟地址范围 (start-end)
  • r--p 是权限标志, 其中:

  • heap

这一段输出可以分为多个部分:

  1. 程序代码段和只读数据:

  • 权限为 r-xp 的行, 可读可执行, 是 .text 代码指令段
  • 权限为 rw-p, 可读可写, 存放已初始化的全局/静态变量, 如 .data 段, 程序输出的 global val 的地址是 0x5647446a0038 对应这部分
  1. Heap
  • 是程序通过 malloc() 函数使用的堆内存区域, 程序输出 heap var 的地址是 0x5647573f02a0, 对应这部分
  • 权限可读可写不共享
  1. 动态库映射
  • 这里把 C 标准库 libc.so.6 映射为多个部分
  • r--p 是其中的只读数据, 如字符串常量, 常量表
  • r-xp 代码段, 包含可执行指令
  • rw-p 可读写数据, 如全局变量, malloc 的堆管理等
  • 动态链接库会在程序启动时由动态链接器加载进内存并映射到进程的虚拟地址空间
  1. 动态链接器 ld-linux
  • ld-linux-x86-64.so.2 是 Linux 的动态链接器, 负责程序启动时加载需要的动态库, 并解析符号
  • 它也被映射到进程的虚拟空间, 包括代码 (r-xp 部分) 和数据 (rw-p 部分)
  1. 内核提供的特殊内存区域
  • [vvar], [vdso], [vsyscall] 是内核与用户空间共享的虚拟内存区域, 用于加速某些系统调用 (比如获取时间, CPU 信息等), 无需陷入内核态
  1. Stack
  • 程序运行时栈, 用于存放:
    • 函数调用帧, 如局部变量, 返回地址
    • 参数传递
  • 权限是 rw-p, 可读可写, 私有
  • 地址位于高地址区域
  • 程序输出的 local var 的地址是 0x7ffc9e6a4f5c 对应这部分
  1. 其他内核区域
  • [vsyscall] 用于加速一些固定的系统调用

虚拟地址的好处

  • 隔离性, 每个进行都有自己独立的虚拟地址空间, 互不干扰, 提高安全性与稳定性, 毕竟不能让一个程序去修改和查看另一个程序的内存空间
  • 内存管理简化, 程序员或编译器不需要关心物理内存的实际布局, 由操作系统和硬件负责映射
  • 支持更大的地址空间, 即使物理内存较小, 也可以通过虚拟地址使用更大的逻辑地址空间

内存分页

Geeksforgeeks blog

内存分页 (Paging) 是一种内存管理技术, 其将虚拟地址空间和物理地址空间分成固定大小的块, 分别称为:

  • Page (页), 虚拟地址空间中的块
  • Page Frame (物理页), 物理内存中的块

从而方便虚拟地址到物理地址的映射.

通常 Page 和 Page Frame 的大小一致, 如 4KB, 8KB, 16KB.

通过把程序占用拆为多个 pages, 能够实现:

  • 整个程序不需要一大段连续的空间
  • 不需要所有 pages 同时存在于 memory 中, 需要时再加载
  • 简化内存分配, 每次只需要分配固定大小的 page

每个 page 可以离散地映射到任意一个 page frame 中, 不必连续.

操作系统需要用一个 page table (保存在内存中) 来跟踪 pages 在内存中的位置. 也就是将 logical page number 映射到 physical page frames 中.

Page table 中的每项称 page table entries, 记录一条映射关系.

为什么需要 paging

当程序需要一个很大的连续 block 时, memory 可能无法提供, 而 paging 将内存分为 smaller, fixed-size pieces, 能更容易分配分散的内存空间.

将程序内存 paging 也能让 memory 将暂时不需要的程序占用交换回磁盘, 空出空间给其他程序使用.

Paging 的工作过程

当 process 请求 memory 时, 操作系统会分配一个或多个 page frame 给该进程, 然后将进程的 logical pages (虚拟内存分块) 映射到这些 physical page frame 上.

每一个 process 都会有独立的 page table 用来跟踪映射关系.

具体的映射工作由 Memory Management Unit 这个物理设备完成.

具体的例子, 假设:

  • Page 大小为 4KB (4096 bytes = 2^12 bytes)
  • Virtual address 为 32 位, 比如 0x12345678

Virtual address 分为两部分:

  • 高 20 位 表示 page number, 用来在 page table 中找到对应 page frame
  • 低 12 位 表示 offset, 表示地址在页内的具体位置 (毕竟分配一个 page 但不一定能占满)

比如对于 virtual address 0x12345678, 有:

  • 高 20 位: 0x12345
  • 低 12 位: 0x678

MMU 通过 page number, 在 page table 中找到对应 page frame number 作为物理地址的高部分, 将 virtual address 中的偏移量作为低部分, 然后合并成最终的物理地址.

比如:

  • page table 中有 page number: 0x12345 -> page frame bumber 0x89ABC
  • 偏移量是 0x678
  • 物理地址为: 0x89ABC000 + 0x678 = 0x89ABC678

Monolithic Kernel

Monolithic 意为 “庞大的, 整体的”, Monolithic kernel 称 “单体内核/宏内核”, 是一种操作系统内核架构, 指操作系统的主要功能模块 (比如进程管理, 内存管理, 文件系统, 设备驱动, 网络协议栈等) 全部运行在 kernel mode, 作为一个整体程序运行在内核空间中.

此时系统核心功能都会编译进同一个内核镜像 (比如 vmlinuz), 应用程序通过内核系统调用访问硬件功能.

Linux 是典型的 monolithic kernel, 但其也支持模块化设计, 可以动态加载内核模块.

由于核心功能都在内核态, 相互之间可以直接调用, 通信效率高, 没有跨进程/空间的开销. 带来的问题是, 如果其中一个模块崩溃, 可能导致整个系统崩溃 (因为所有模块共享同一个地址空间).

Micro kernel

Micro 意为 “微小的”, Micro Kernel (微内核) 是一种操作系统内核架构, 它将操作系统的核心功能 (如进程管理, 基本内存管理, 进程间通信 IPC) 保留在 kernel mode 中运行, 而将其他传统上属于内核的功能 (如文件系统, 设备驱动, 网络协议栈等) 移到 user space 运行, 通过严格定义的通信机制来交互 (通常是 IPC).

最小化 kernel mode 的代码, 能提高操作系统的稳定性 (user mode 崩溃不会让系统崩溃), 安全性 (不能通过 user mode 的核心功能干扰 kernel mode 程序, 且代码量少攻击面小), 以及模块化程度, 但会牺牲一些性能.

Micro kernel 的问题在于 IPC 通信开销大, 以及模块间通信的逻辑复杂, 大多数软件为 Monolithic 内核设计, 因此适配 micro kernel 的成本较高.

Micro kernel 操作系统一般用于嵌入式, 实时, 安全型系统.

Unikernel

Unikernel (单体内核), 是一种为单一应用定制的, 极简的操作系统镜像, 其将应用程序与精简的操作系统库 (比如网络, 文件系统, 驱动等) 一起编译成一个单独的, 无传统内核结构的可运行镜像, 一般用于虚拟化或云环境.

传统的操作系统如 Linux, Windows, 可以跑浏览器, 数据库, 游戏等各种服务, 其提供通用的抽象并通过系统调用暴露给所有应用.

但如果你只想运行一个应用 (比如一个 Web 服务器), 不需要这些通用功能, 也不需要多进程, 多用户等复杂管理, 此时出现了 unikernel, 其思想就是: 为某个特定应用, 打包它需要的操作系统功能, 并把应用和这些必要的库编译成一个单独的镜像, 没有传统内核中的多余服务, 进程, shell 等等组件.

Unikernel 是单地址空间, 即应用和操作系统运行在同一个内存地址空间中, 没有内核-用户态分离, 能避免上下文切换和系统调用开销.

Unikernel 启动速度极快, 且更加安全 (没有多余服务, 内核漏洞利用面).

批处理 OS

批处理 OS (Batch Processing Operating System), 是早期一种操作系统类型: 用户不直接与计算机交互, 而是准备一组作业提交给操作系统, 其按照一定顺序先后运行, 整个过程无需人工干扰.

虽然称 batch processing, 但同一时刻只有一个任务运行, 这里的 batch 更多是指提交的时候包含多个任务:

批处理 OS 虽然受限时代, 基本无交互, 但是其自动批量运行还是能最大化硬件利用率并进行任务调度.

LibOS

Medium blog about libOS

LibOS (Library Operating System, 库操作系统), 是一种将传统操作系统内核的大部分功能 (如文件系统, 网络, 进程管理, 线程等) 以库的形式链接到用户程序中的操作系统架构.

应用程序不再先通过调用 “系统调用”, 经内核控制硬件再返回, 而是直接与硬件交互.

这种方法减少了传统系统调用的上下文切换开销, 且针对应用的定制化强, 可以舍弃冗余的组件.

前面说的 Unikernel 就是一种 LibOS.

Sandboxed environment

Sandbox 是一个儿童玩具, 是一个装有沙子的小箱子或围栏区域, 小孩可以在里面玩沙子, 而且不会弄脏院子或房间其他地方.

在计算机领域 Sandbox 的概念和上面类似, 指一种隔离的, 受控的运行环境, 能够在里面安全执行代码, 程序, 进程, 限制其对系统资源的访问, 防止它对主机系统造成破坏. 这里 “程序” 就像 “小孩”, sandbox 就是限制它活动, 保护周围环境的 “围栏+沙地”.

Sandbox 并不是某种单一技术, 而是一类通过不同机制实现 “隔离与限制” 的通用思想, 比如:

  • Linux 容器, 操作系统级隔离, 利用 namespace, cgroups, chroot 等
  • Virtual machine, 在虚拟硬件上运行完整操作系统与主机隔离
  • Browser sandbox, 浏览器会让插件运行在受限环境中

Virtual file system

Virtual file system (VFS, 虚拟文件系统) 是操作系统提供的一种抽象层, 其定义了一组通用的文件系统操作接口, 让用户程序和内核能够以统一的方式访问不同类别的文件系统, 而不需要关心底层文件系统的实现细节:

一个新的文件系统只需按照 VFS 的接口规范编写驱动, 就能融入系统.

原子操作

原子操作 (Atomic Operation) 是指一个操作在执行过程中, 要么全部完成, 要么完全不执行, 不会被其他线程或进程中断, 也不会出现 “执行到一半” 的状态.

在 Linux 系统中, 一次系统调用是 atomic operation, 但是两次系统调用则不是, 比如:

  • 假设两个进程同时修改一个文件
  • 两个进程都用 lseek 定位到文件末尾然后 write 100 个字节
  • 假设进程 A lseek 到文件末尾后被内核挂起, 进程 B 也 lseek 到末尾
  • 此时恢复进程 A, 其写入 100 字节后, 文件长度发生了变化, 但 B 并不会定位到新的末尾
  • 此时进程 B 再写入则会覆盖 A 写入的内容

这里一个进程包含两个系统调用, 但在执行完成一个系统调用后可能就会被挂起, 因此不是原子的.

Trap

Trap 是一种程序指令主动触发的控制流转移, 一般用于:

  • 执行系统调用
  • 触发异常
  • 调试断点
  • 执行特权操作

此时会将当前上下文保存, 以便处理完 trap 后, 能恢复现场, 让程序从 trap 点继续执行下去.

  • 发生 trap 时会陷入 kernel mode, 处理结束后恢复到 user mode, 上图就像凹陷了一样, 也因此得名 trap

Trap handler 是操作系统内核提供的一段代码, 用于处理某种 trap 事件. 常见的 trap 类型有:

  • syscall, 处理用户发起的 read/write 等请求
  • page fault, 程序访问未加载的内存页时触发
  • divide by zero, 处理非法算术操作
  • breakpoint, 利用 trap 来暂停程序执行, 检查变量等

Interrupt 和 Timer

Interrupt 是处理器在执行程序时, 由于内部或外部某个事件的发生, 暂停当前程序的执行, 转而去处理该事件的一种机制. 在处理完毕后, 再返回到原来暂停的程序处继续执行.

Interrupt 常分为:

  • Hardware interrupt, 来自外部硬件 (比如鼠标, 键盘, 磁盘, 网卡, 时钟等), 异步的, 随时可能发生
  • Sofrware interrupt, 由程序执行特定指令触发 (比如 x86 汇编中的 int 指令)
  • Exception, 由 CPU 执行指令时发生错误或特殊情况, 同步的, 比如执行 x/0 就会触发异常

Timer 是一种硬件设备或软件机制, 能够在指定时间间隔之后产生一个事件 (通常是中断), 从而通知 CPU 执行特定的操作.

脏数据

CISC 和 RISC

CISC (Complex Instruction Set Computing, 复杂指令集), 会针对特定功能来实现指令, 导致指令数目比较多, 但生成的程序长度相对较短.

RISC (Reduced Instruction Set Computing, 精简指令集), 只定义常用指令, 对复杂的功能常采用常用指令的组合来实现. 指令数目比较精简, 但生成的程序长度相对较长.

Host Byte Order

Host Byte Order, 主机字节序, 指一个 “多字节整数” (一个数由多个字节表示) 在计算机内存中存储的字节顺序.

不同类型的 CPU HBO 会不同, 分:

  • Big-Endian, 大端序, 高位存放在内存低地址
  • Little-Endian, 小端序, 低位存放在内存低地址

(名字的由来是, 吃鸡蛋先敲大端还是小端, 假设桌子是低地址, 大端序就是用大端敲桌子)

函数调用栈

在操作系统中, Stack 是一个动态的数据结构, 用于控制函数间的调用.

当调用一个函数时, 计算机需要记住:

  1. 当前执行的位置, 即返回地址, 函数调用结束后, 要回到这里继续执行
  2. 函数参数
  3. 函数的局部变量, 只会在函数运行期间存在
  4. 函数调用上下文, 保存当前调用者的堆栈信息, 保存的寄存器等

每一次函数调用, 都会在 stack 上创建一个新的 stack frame, 用于保存该函数的执行上下文.

Stack frame 是 stack 上的一个连续的区域, 用于存储某一次函数调用的相关信息, 包括:

  • 函数参数 (Arguments)
  • 返回地址 (Return address)
  • 局部变量 (Local variables)
  • 保存的寄存器 (Saved registers)
  • 调用者的栈帧指针 (Frame Pointer, FP)

ABI 和 SBI

ABI

ABI, Application Binary Interface (应用二进制接口), 是一个接口规范, 其规定了代码编译出的二进制如何与其他二进制交互:

  • 函数调用时参数如何传递 (通过寄存器还是栈)
  • 返回值如何返回
  • 调用约定
  • 寄存器的使用约定
  • 栈的布局
  • 二进制的文件格式 (如 ELF, PE)
  • 数据类型在内存中的布局

SBI

SBI, Supervisor Binary Interface (监管者二进制接口), 是 RISC-V 框架中, Supervisor Mode 与 Machine Mode 之间交互的一套标准接口.

用 Rust 写的操作系统内核一般运行在 Supervisor Mode, 其不能直接操作某些底层硬件功能, 而是调用 SBI 接口, 由 Machine Mode 来完成.

SBI 提供了一系列的标准化函数来完成底层操作, 比如:

  • sbi_console_putchar, 向控制台输出一个字符
  • sbi_console_getchar, 从控制台读取一个字符
  • sbi_set_timer, 设置定时器中断
  • sbi_shutdown, 关机或重启

Prologue 和 epilogue

Prelogue (序言) 和 epilogue (结尾) 是函数调用中的概念, 分别在函数开始执行和结束执行时, 负责保护上下文, 设置栈帧, 恢复现场等.

ELF 文件格式

ELF, Executable and Linkable Format (可执行与链接格式), 定义了操作系统如何加载和运行文件.

其结构如下:

  • ELF Header 包含 elf 文件的元信息, 比如 type, architecture, entry point
  • Program Header Table 描述在 runtime 使用的 segments 的信息
  • Sections and Segments 包含实际的数据, 如 code, data, symbols, debug information
  • Section Header Table 描述在 linking 和 debuging 时使用的 sections 信息

ELF Header

ELF Header 中重要的 fields 有:

  • e_ident, “ELF Identification”, 一个标识此文件为 ELF file 的 magic number
  • e_type, 标记 ELF 类型 (有 3 种, executable, shared object, relocatable)
  • e_machine, 指明当前 ELF 是运行在哪个 architecture 上的 (x86_64, arm, riscv)
  • e_version, ELF 文件格式的版本
  • e_entry, entry point 的入口地址
  • e_phoff, program header table 的偏置
  • e_shoff, section header table 的偏置

Program header table

Program header table 包含程序运行需要的信息, 其告诉 loader 如何将文件中的各部分装载进内存.

Key fields 有:

  • p_type, segment 的类型 (如 LOAD, DYNAMIC)
  • p_off, “offset”, segment 在文件里的偏移量
  • p_vaddr, “virtual address”, segment 该被加载到哪个虚拟地址
  • p_paddr, “physical address”, 物理地址
  • p_filesz, “file size”, segment 的大小
  • p_memsz, “memory size”, segment 在内存中占用的大小

segment types 有:

  • LOAD, 包含需要被加载进内存的 code or data
  • DYNAMIC, 包含 dynamic linking 信息
  • INTERP, 包含 dynamic linker 的名称

Section Header table

Section header table 包含 linking 和 debugging 阶段需要用到的信息, 描述各个 section, 如 .text (包含代码), .data (包含数据).

Key fields 有:

  • sh_name, section 的名称
  • sh_type, section 的类型 (SHT_PROGBITS 表示 code/data)
  • sh_flagss, 表明 section attibutes
  • sh_addr, 该 section 要被加载到哪个 virtual address
  • sh_offset, section 在文件中的偏移量
  • sh_size, section 的大小

文件通常会包含下面 sections:

  • .text, 包含 executable code
  • .data, 包含 initialized data
  • .bss, 包含 uninitialized data
  • .symtab, linker 要用的 symbol table
  • strtab, symbol table 要用的 string table
  • .rel.text, .text section 的重定向信息

E_TYPE

有 3 种类型, 在 ELF Header 中的 e_type 字段来标识:

ELF Type value 宏定义 含义 常见扩展名
ET_REL 1 可重定位文件 (relocatable file) .o
ET_EXEC 2 可执行文件 (Executable file) a.out
ET_DYN 3 位置无关可执行文件 (position independent executable file, PIE) 或共享文件 (shared object file) .so, PIE 可执行程序
ET_CORE 4 Core Dump 文件 core

Relocatable file

Relocatable file 是汇编器生成的, 尚未链接的中间文件, 其包含:

  • 机器码
  • 符号表 (函数名, 全局变量名等)
  • section 信息 (如 .text, .data, .bss 等)

由于该文件中地址是相对的, 因此不能直接运行. 需要 linker 将地址修改为运行时的正确地址.

Position independent executable file

Position independent executable file (PIE) 能够在运行时由动态链接器加载到随机地址, 可以直接运行.

Shared object file

Shared object file 设计为在运行时被动态加载和共享, 多个进程可以共用同一份物理内存中的代码段, 节省内存.
