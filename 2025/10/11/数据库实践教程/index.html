记录数据库开发一些概念和技巧. 教材推荐: “数据库系统实现”, “数据库系统概念”.

概述

数据库的接口是:

  • 数据定义语言 DDL
  • 数据查询语言 SQL
  • 数据操作语言 DML

数据库具体需要实现:

  • DDL, DML, SQL
  • 持久化能力
  • 故障恢复能力
  • 并发控制与事物管理

早期的数据库管理系统通常是层次模型或网状模型, 缺点是没有高级查询语言且操作复杂.

当前主流的是关系数据库系统, 并提供 SQL.

常见的非关系数据库是 NoSQL 数据库:

  • 通过 API 访问
  • 不需要预定义模式
  • 分布式支持
  • 对象存储

数据库简史:

NewSQL

SQL 引擎, 事务引擎, 存储引擎.

数据库架构:

关系数据库 = 存储 + 事务 + SQL:

  • 存储: 关系数据库的数据结构
  • 事务: 关系数据库的算法
  • SQL : 关系数据库操作的描述语言 (一层接口)

工业界实现存储通常有两种数据结构:

  • 哈希表, $O(1)$, 支持 put/get, 不支持范围 scan
  • B+ 树, $O(logn)$, 支持 put/get, 支持范围 scan

事务中最重要的是 ACID:

  • 原子性 (Atomic), 事务操作要么全部成功, 要么全部失败
  • 一致性 (Consistent), 一个事务只能使数据库从一个一致的状态跳转到另一个一致的状态
  • 隔离性 (Isolation), 多个并发事务互相不影响, 就像多个事务串行执行一样
  • 持久性 (Durability), 一旦事务成功提交, 其对数据库的影响是永久的

Miniob 中的重要类

Page

Page 就是一个 8kb 的 struct, 其中 data 是一个字节数组.

Frame

除了包裹一个 page 外, 还包含一些其他元信息.

DiskBufferPool

一个 DiskBufferPool 与一个 table 文件相关联 (描述符和文件名). 管理该 table 内 page 从磁盘到内存的交互.

BPFrameManager

整个数据库只有一个全局的 BPFrameManager, 管理整个物理内存.

BufferPoolManager

管理所有 DiskBufferPool.

RecordFileHandler

管理整个 table 文件内记录的增删改查.

RecordPageHandler

管理一个 page 内记录的增删改查.

RowRecordPageHandler

管理以行存存储的一个 page 内记录的增删改查.

HeapTableEngine

为上层提供操作 Record 的接口, 底层利用 RecordFileHandler 和 RecordPageHandler 来进行实际的读写.

SessionEvent

SessionEvent 代表了从接收到客户端请求到返回结果的整个生命周期, 是一个贯穿始终的 “上下文” 或 “事件” 对象. 包含原始的查询语句 query_, 包含 SqlResult 对象, 用于存放查询的最终结果.

SqlResult

SqlResult 提供一个访问结果集的接口, 包含:

  • 返回码 (return_code_), 表示查询是否成功执行
  • TupleSchema, 描述结果集的元数据, 即列信息 (列名, 类型等)
  • Operator*, 查询结果就是通过迭代 Operator 树来动态生成的

TupleSchema

TupleCellSpec

TupleCellSpec 包含单个 field 的信息.

Tuple

表示一行数据, 当返回客户端时使用.

索引

数据库中的索引 (Index), 是一种特殊的数据结构, 目的是加速数据的查找. 索引可以快速定位到表中的某条记录, 而不必全表扫描.

比如:

  • 如果没有索引, 数据库引擎需要从第一行开始一行一行扫描, 直到找到 department = 'IT' 的那些行
  • 如果在 department 字段上建立了索引, 数据库可以通过索引结构快速定位到 department = 'IT' 对应的记录位置

在 department 字段上建立索引的意思, 就是利用 department 字段上的所有可能值来建立一个 table (实际上不是 table, 是一个特殊的数据结构), 每一个 entry 是 “值-行指针” 这样的结构.

常见的索引实现有 B+ 树, Hash, 位图 等.

索引键

索引键 (index key), 指用来构建 index 的列或列的组合 (构建索引需要输入一些列). key 通常是这些列内的值.

索引条目

索引条目 (index entry), 指包含索引键值和指向实际数据行的指针.

Projection

Projection 是 SQL 查询的一个阶段. 从查询结果中选取指定的列 (或表达式), 进行操作后 (计算, 重命名等) 决定最终返回给用户的内容.

SQL 中的 SELECT 子句就是 projection 操作, 从输入行中提取/计算出指定的列, 形成新的输出行:

  • id, name 是原表中的列
  • age + 1 AS next_age 是一个表达式, 最列进行计算并赋予新名字

Projection 操作本身不改变行数, 这个由 filter 负责.

Double Write

Double Write (双写), 是数据库存储引擎中的一种机制, 是为了保证数据库崩溃恢复能力和页完整写入而设计的.

Double Write 用于在将 Page (页) 写入磁盘时, 先将其安全写入磁盘中一个专门的连续区域 (Double Write Buffer, 通常是系统表空间的一块区域), 然后再将其写入真正的位置, 以避免部分页写入 (Partial Page Write) 导致数据损坏. 这个过程 Write 了两次, 因此也得名 Double Write.

Partial Page Write 问题主要源自磁盘写入不保证 “原子性”, 在数据库中, 数据通常以 Page (页) 为单位进行管理, 比如 InnoDB 默认的页大小是 16 KB.

当更新一条数据时, 最终会修改某一个 16KB 大小的数据页, 然后需要将整个页写回磁盘. 当系统在写入一个页的过程中, 如果发生了崩溃 (如断电, 宕机), 可能出现:

  • 该页只写了一般 (比如只写了前 8 KB, 后 8 KB 没写进去)
  • 写入的页内容不完整/损坏

这会导致数据库重新运行时, 会发现某个页的数据是不完整的, 且没有备份, 可能会影响事务的一致性, 索引正确性.

如果在写入 Double Write Buffer 过程中出现崩溃, 也只是丢失了脏页的数据, 对于数据库中已经持久化的数据而言, 不会破坏一致性.

如果在从 Double Write Buffer 写入真正区域的过程中出错, 此时不会有影响. 在数据库恢复后, 会扫描 Double Write Buffer 区域, 找到成功写入 Double Write Buffer 但没有成功写入最终目标的页, 然后将这些页从 Double Write Buffer 中拷贝/写入到它们应该在的数据文件位置.
