相关概念

Stub

Stub, 存根, 在中文语境下, 有含义为:

  • 票据, 记录交易信息
  • 凭证, 验证身份

在计算机科学中指一个简化的占位代码或对象.

Client Stub

Client Stub, 客户端存根, 就是指远程服务的一个代理对象.

hello_pb2.py

该文件主要

  • 将 .proto 文件中的消息和服务定义转换为 Python 类
  • 提供消息的序列化和反序列化功能

分段解释文件内容.

  1. 文件头的注释信息如下 其主要指定
  • “编码信息”, 这里是 utf-8
  • “source” 即原 .proto 文件名
  • Protobuf Python 的版本
  1. 依赖信息
  • 这些不用管, 导入的 Protobuf 库的核心模块
  1. 验证版本
  • 其验证当前运行的 Protobuf 版本是否与生成代码时的 Protobuf 版本一致
  1. 初始化符号库
  • _symbol_database 是 Protobuf 的符号数据库, 用于管理消息类型和枚举类型. Default() 方法返回默认的符号数据库实例
  1. 定义消息描述符
  • DESCRIPTOR 是 Protobuf 消息描述符的核心数据结构, 包含了 .proto 文件中定义的所有消息和服务的信息
  • AddSerializedFile 方法将 .proto 文件的内容序列化为二进制格式, 并添加到描述符池中
  • 二进制数据 b'\n\x0bhello.proto...' 是 hello.proto 文件的序列化表示, 包含了消息和服务的定义
  1. 构建消息和枚举描述符
  • BuildMessageAndEnumDescriptors: 根据 DESCRIPTOR 构建消息和枚举的描述符, 并将其注册到全局命名空间 (_globals)
  • BuildTopDescriptorsAndMessages: 构建顶级消息和服务描述符, 并将其注册到 hello_pb2 模块中
  1. 设置加载选项
  • 这段代码检查是否使用 C++ 实现的描述符. 如果不是, 则设置描述符的加载选项为 None
  1. 定义消息和服务的全局变量
  • 这些变量记录了每个消息和服务在 DESCRIPTOR 中的起始和结束位置. 例如:
    • _HELLOREQUEST 消息的序列化数据在 DESCRIPTOR 中的位置是 15 到 43
    • _GREETER 服务的序列化数据在 DESCRIPTOR 中的位置是 72 到 123
  1. 模块作用域标记
  • 这是一个特殊的注释标记, 用于指示 Protobuf 编译器在生成代码时可以插入额外代码的位置

基本使用

hello_pb2_grpc.py

该文件主要负责:

  • 实现客户端和服务端的通信逻辑
  • 其基于 .proto 文件中定义的 service

分段解释下面代码:

  1. 文件头信息
  • 声明别手动改这文件, 以及其内容是包括 client 和 server 相关类的
  1. 导入依赖
  • grpc 提供客户端和服务端的通信功能
  • hello_pb2 提供 .proto 中定义的消息类型以及序列化功能
  1. 检查 gRPC 版本兼容性
  • GRPC_GENERATED_VERSION: 生成代码时使用的 gRPC 版本
  • GRPC_VERSION: 当前安装的 gRPC 版本
  • first_version_is_lower: 检查当前 gRPC 版本是否低于生成代码所需的版本
  • 如果版本不兼容, 抛出 RuntimeError, 提示用户升级或降级 gRPC 模块
  1. 客户端存根 (Stub)
  • GreeterStub: 客户端存根类, 用于调用服务端的方法
  • channel: gRPC 通信的通道, 用于发送和接收消息
  • SayHello: 定义了一个 RPC 方法, 对应 .proto 文件中的 SayHello 方法
    S*S unary_unary: 表示这是一个一元 RPC 调用 (客户端发送一个请求, 服务端返回一个响应)
    • request_serializer: 将请求消息序列化为字节流
    • response_deserializer: 将响应字节流反序列化为消息对象
    • _registered_method=True: 表示这是一个已注册的 RPC 方法
  1. 服务端实现
  • GreeterServicer: 服务端实现类, 用于处理客户端请求
  • SayHello: 实现 .proto 文件中定义的 SayHello 方法
    • request: 客户端发送的请求消息
    • context: gRPC 的上下文对象, 用于设置状态码和详细信息
    • 默认实现抛出了 NotImplementedError, 提示方法未实现. 实际开发中需要重写此方法
  1. 将服务端实现注册到服务器
  • add_GreeterServicer_to_server: 将 GreeterServicer 实现注册到 gRPC 服务器
  • rpc_method_handlers: 定义了 RPC 方法的处理逻辑
  • unary_unary_rpc_method_handler: 处理一元 RPC 调用
  • request_deserializer: 将请求字节流反序列化为消息对象
  • response_serializer: 将响应消息序列化为字节流
  • generic_handler: 将 RPC 方法处理逻辑与服务名称绑定
  • add_generic_rpc_handlers: 将处理逻辑添加到服务器
  1. 实验性 API
  • Greeter: 实验性 API 类, 提供了一种更简洁的方式调用 RPC 方法
  • SayHello: 静态方法, 用于调用服务端的 SayHello 方法
    • grpc.experimental.unary_unary: 实验性的一元 RPC 调用方法
    • target: 服务端地址
    • options: gRPC 调用选项
    • channel_credentials: 通道凭证 (如 TLS 证书)
    • call_credentials: 调用凭证 (如 OAuth2 令牌)
    • metadata: 元数据, 用于传递额外的信息 (如身份验证令牌)

基本使用

基本流程是:

  • 客户端: 通过 GreeterStub 调用服务端的方法
  • 服务端: 通过 GreeterServicer 实现服务端逻辑, 并通过 add_GreeterServicer_to_server 注册到 gRPC 服务器
  • 实验性 API: 提供了一种更简洁的方式调用 RPC 方法

hello_pb2.pyi

.pyi 是 Python 的类型存根文件 (Stub Files), 它们用于为 Python 模块提供类型提示信息, 但不包含实际的实现代码. 这类文件通常用于静态类型检查工具 (如 mypy) 或 IDE (如 PyCharm, VSCode) 中. (并不是用来调用的模块, 仅方便编辑器自动提示)

grpc 库

由于我是本地安装, 因此可以查看 $HOME/.local/lib/python3.13/site-packages/grpc 目录下定义的文件.

grpc.Channel 类

Channel 类定义在 grpc/__init__.py. 用于管理客户端和服务端之间的通信连接. 负责:

  • 建立连接, 与服务端建立网络连接

  • 发送请求, 将客户端的 RPC 请求发送到服务端

  • 接收响应, 接收服务端返回的 RPC 响应

  • 管理连接状态, 处理连接的生命周期

创建 ClientStub 类实例时, 都需要提供 channel 成员值 (grpc.Chennel 类实例).

grpc.Server 类

Server 类也定义在 grpc/__init__.py 文件中, 负责:

  • 管理 grpc 服务端的生命周期 (start, stop)
  • 管理服务实现类与 RPC 方法的注册, 要把 .proto 中定义的 service 绑定到一个 server

创建

其通过 grpc.server() 函数创建:

  • futures.ThreadPoolExecutor(max_workers=10): 指定服务器的线程池, 用于处理客户端的并发请求
    (需要注意这里用的是 ThreadPoolExecutor 线程池, 若用成了 ProcessPoolExecutor 进程池之类的, 则会阻塞)

注册服务

绑定端口

  • add_insecure_port, 指添加一个不加密的端口
  • add_secure_port, 指添加一个加密的端口
  • [::] 表明所有地址
  • :port 指定端口

启动

等待终止

  • wait_for_termination 方法阻塞主线程, 直到服务器被终止

停止

  • grace: 指定一个超时时间, 等待未完成的请求处理完毕后再停止服务器

代码编写逻辑

  • 利用 .proto 文件编译生成的 *_pb2.py 和 *_grpc_pb2.py 文件定义核心类以及方法
  • 利用 grpc.server() 创建 Server, 监听端口, 添加服务实例, 并由此编写逻辑
  • 利用 grpc.Channel() 创建 chennel, 并用 ClientStub 创建代理类实例, 由此编写具体客户端逻辑
