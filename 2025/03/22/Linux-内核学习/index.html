Chapter 1 Linux kernel development process

Linux kernel 的发布是 time-based 而不是 feature-based, 基本上 10-11 周发布一次.

当发布一个稳定版本后, 就开始为下一个版本合并代码, 主要分两阶段:

  • 2 周的合并窗口期 (Merge window), Subsystem Maintainers 通过签名的 Git pull request 向 Linus 提交代码, 所有主要新功能都会在这个窗口期内合并, 之后会关闭
  • 6-8 周的稳定期 (Bug Fixes-Only Mode), 仅允许修复严重错误, 禁止新功能, 稳定后即发布新版本

在新版本发布前 1 周, 加上 2 周合并窗口期, 称静默期, 这段时间维护者和核心开发者会专注于准备代码, 对社区的响应会比较慢.

Active Kernel Releases 分三类:

  • Release Candidate (RC) releases, 由 Linus 发布的用于测试新功能的版本 (意思是可以添加 features), 必须从源代码编译安装, 通常不提供预编译的二进制包
  • Stable releases, 正式发布的稳定版本, 当 RC 版本通过测试后发布, 此时进入维护模式, 禁止添加新功能, 发布补丁修复 bug
  • Long-term releases, 指长期支持的 Stable 版本, 在 2~6 年内都会修复 bug

Kernel tree

Kernel tree 指 Linux kernel 源码的组织方式, 其分布在多个 Git 仓库中, 包括 main 和 subsystem git repositories.

  • Mainline kernel tree 由 Linus 发布
  • Stable tree 由 Greg Kroah-Hartman 维护, 包含 stable release branches
  • Linux-next tree 由 Stephen Rothwell 维护, 其集成了大量 subsystem trees

Maintainer 首先要将自己的 tree 提交到 linux-next tree, 在 3-7 天的等待周期中让其他开发者测试代码发现问题. 当你的代码添加到 linux-next tree 之后, 再提交的补丁会在 merge window 期间被考虑是否合并到主线内核中.

Subsystem Maintainers

每一个重要的 subsystem 都有自己的 tree 以及 designated maintainers, 可以在内核源代码的 MAINTAINERS 文件中查看.

每个 subsystem 的 mailing list 可以在 这里查看.

更细致的开发流程可以在 A guide to the Kernel Development Process 处查看.

Chapter 2 Patches

可以用:

  • 将指定 commit 转换为 .patch 文件

一个 Patch 包含的内容有:

  • Commit ID, 一个基于 commit 中的 metadata 计算出的 哈希值
  • Commit header, 指标题, 格式为 subsystem: submodule: brief description
  • Commit log, 需包含该变更的目的和描述, 考虑了哪些其他可选方案, 以及测试的结果
  • Author, 包含 author’s name and email information. 可以在 .gitconfig 或者 git commit 中提供
  • AuthorData, 自动生成的提交日期
  • Commit, committer 的 name 和 email information. 注意其与 author 的区别:
    • Author, 指实际修改代码的人
    • Committer, 指将补丁合并到仓库的维护者
  • CommitDate, 维护者合并补丁时的系统时间
  • Signed-off-by, 签名之后, 开发者才能证明该补丁是自己的工作, 且将其视为 open source patch

Chapter 6 Exploring Linux Kernel sources

Cregit 工具

Cregit 工具用于 Code Review (代码审查) 和 Code Provenance (代码溯源), 专门针对 C 代码库 (尤其是 Linux 内核) 进行代码作者归属分需和变更历史追踪. 其结合了 git 版本控制信息和静态代码分析, 以便于理解代码的演变.

其构建较复杂, 建议直接在 https://cregit.linuxsources.org/code/5.1 查看.
