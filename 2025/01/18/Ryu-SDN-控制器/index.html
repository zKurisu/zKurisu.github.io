Architecture

Ryu Controller 的编写逻辑为:

  • 定义继承自 app_manager.RyuApp 的类, 这是入口, 整个文件就是在完善这个类
  • 利用 set_ev_cls 修饰器添加事件处理函数

OFPFlowRemoved

当 switch 中某个流表项被删除 (或超时) 了, 向 controller 发送的消息会转为这个类.

其成员有:

  • cookie, 流表项的标识符 (如果 controller 端也记录了 cookie, 就知道是哪个流表项被删除了)
  • priority, 流表项的优先级
  • reason, 流表项被移除的原因, 其值可能是:
    • OFPRR_IDLE_TIMEOUT, 一段时间没有数据包匹配
    • OFPRR_HARD_TIMEOUT, 存在时间超时
    • OFPRR_DELETE, 被直接删除
    • OFPRR_GROUP_DELETE, 删除 group 时一起被删除的
  • table_id
  • duration_sec, 该流表项已存在的时间, 以 seconds 为单位
  • duration_nsec
  • idle_timeout
  • hard_timeout
  • packet_count, 该流表项已处理的数据包数量
  • match, 一个 OFPMatch 实例

OFPPort

OFPPort 用来描述一个 port, 其不与某个事件和某种类型的 msg 绑定.

其成员有:

  • port_no, 端口号
  • hw_addr, MAC 地址
  • name
  • config, bitmap of port configuration flags, 值可能是:
    • OFPPC_PORT_DOWN, 端口被管理员关闭
    • OFPPC_NO_RECV, 端口禁止接收数据包
    • OFPPC_NO_FWD, 端口禁止转发数据包
    • OFPPC_NO_PACKET_IN, 静默丢弃未匹配的流量, 不通知控制器
  • state, bitmap of port state flags, 值可能是:
    • OFPPS_LINK_DOWN, 端口物理链路断开
    • OFPPS_BLOCKED, 端口被阻塞
    • OFPPS_LIVE, 端口可用于快速故障切换
  • curr, 当前使用的 feature 信息
  • advertised, 端口向对端设备通告的 feature 信息
  • supported, 端口所支持的 feature 信息
  • peer 对端设备通告的 feature 信息
  • curr_speed
  • max_speed

OFPPortStatus

当 Switch 上 port 的状态变化时, 向 controller 发送的数据包转为这个类.

其成员有:

  • reason, 即 port 状态变化的原因, 值可能为:
    • OFPPR_ADD, switch 新添加了一个 port
    • OFPPR_DELETE, switch 删除了一个 port
    • OFPPR_MODIFY, switch 上一个 port 的状态修改了
  • desc, 一个 OFPPort 实例, 用来描述出问题的那个 port

OFPPacketOut

OFPPacketOut 是 controller 向 switch 发送的一种数据包, 可以让其执行 action, 用这个类.

其成员有:

  • buffer_id, 要让 switch 处理的数据包在缓冲区内的 id, 其与 data 成员互斥, 设置了这个, data 字段可省略或为空
  • in_port, 设置要处理的数据包的 in_port 值, 如果要伪装成是 controller 自己生成的包, 值可以设为 OFPP_CONTROLLER
  • actions 要让 switch 执行的 action 列表
  • data, 包的 binary type value 或者 packet.Packet 实例

Chapter 2 Switching Hub

Switching Hub 所具有的功能有:

  • 学习连接到 port 的 host 的 MAC 地址, 并写入 MAC address table
  • 当接收到的数据包指向一个已知地址的 host 时, 从对应端口发出
  • 当接收到的数据包指向一个未知地址的 host 时, 执行 flooding

OpenFlow 交换机从 Ryu Controller 接收指示后需要完成:

  • 对接收的 packet 进行修改或者将 packet 从指定 port 发出
  • 将 packet 发送到 controller (Packet-In)
  • 根据从 controller 接收到的指示将数据包从指定 port 发出 (Packet-Out)

注意, Packet-In 和 Packet-Out 是 OpenFlow 交换机与控制器之间的两种主要通信类型:

  • Packet-In 消息是由交换机发送给控制器的, 当交换机无法匹配流表规则时, 会将数据包发送给控制器处理
  • Packet-Out 消息是从控制器向 OpenFlow 交换机发送的消息, 是包含数据包发送命令的消息. 控制器可以使用 Packet-Out 消息, 告诉交换机某一个数据包如何处理

每当 Switch 产生 Packet-In packet 时也会学习 MAC 地址与 port 的对应.

当 switch 转发包时:

  • 目的 Host 已知, 则转发到对应 port
  • 目的 Host 未知, 则 flooding

Class Definition and Initialization

构建 Ryu application 的第一步, 就是创建一个继承自 ryu.bash.app_manager.RyuApp 类的类. 且需要用 OFP_VERSIONS = [ xxx ] 来指定 OpenFlow 协议版本.

Event Handler

对于 Ryu 而言, 每接收到一个 OpenFlow message, 就会产生一个对应的事件. 其针对每一个 message 都有对应的 event handler.

这些 event handler 都以 event object 作为参数, 且用 ryu.controller.handler.set_ev_cls 修饰器来修饰.

  • set_ev_cls 指定对哪个 event class message 做处理, 以及此时 switch 需要满足的状态

Event class 的命名规则为: ryu.controller.ofp_event.EventOFP + <OpenFlow message name>, 比如 EventOFPPacketIn.

Switch state 有:

  • ryu.controller.handler.HANDSHAKE_DISPATCHER, 交换 HELLO message 时 (Controller 的 HELLO 对应 HANDSHAKE)
  • ryu.controller.handler.CONFIG_DISPATCHER, 等待接收 SwitchFeatures message 时 (Controller 的 SET_CONFIG 对应这里的 CONFIG)
  • ryu.controller.handler.MAIN_DISPATCHER, normal status
  • ryu.controller.handler.DEAD_DISPATCHER, 与控制器断开连接时 (断开对应 DEAD)
    (都有 _DISPATCHER, 分发, 可以理解为分发流表项)

Table-miss event

Table-miss flow entry 是 OpenFlow 交换机中的一种特殊流表项, 用于处理无法匹配任何现有流表项的数据包. 当交换机收到一个数据包时, 它会尝试在流表中查找匹配的流表项. 如果没有任何流表项与数据包匹配, 就会发生 Table-miss 事件.

Table-miss flow entry 也可以视为对数据包的默认处理.

Table-miss flow entry 用来初始化流表, 在 OpenFlow 交换机与控制器完成握手后, 控制器通常会向交换机下发一个 Table-miss flow entry, 以确保交换机能够正确处理未知流量的数据包.

Feature Request

当控制器与交换机完成 HandShake 后, 控制器会向交换机发送一个 Features Request 消息, 请求交换机的特性信息. 交换机收到该请求后, 会回复一个 Switch Features 消息 (即 Features Reply 消息):

从 msg 中获取重要对象:

  • ev.msg 就是从 switch 传来的 packet, 表示当前 event class 的 instance, 其类型取决于接收到的 OpenFlow 消息类型, 比如
    • 收到 Switch Features 消息, 则 ev.msg 是 ryu.ofproto.ofproto_v1_3_parser.EventOFPSwitchFeatures 类的实例
  • msg.datapath, 是 ryu.controller.controller.Datapath 类的实例, 表示与当前消息相关的交换机交互的接口 (Datapath), 能够:
    • 获取交换机的 Datapath ID
    • 获取交换机 OpenFlow 协议版本
    • 获取 OpenFlow 消息解析器, 这个解析器能用于 “解析” 和 “构造” OpenFlow 消息
    • 向交换机发送消息

构建 Match-Action 流表项并下发到交换机:

  • match = parser.OFPMatch() 创建一个匹配所有数据包的 rule (即 table-miss)
  • parser.OFPActionOutput(port, len) 指 Action 是 Output, 其两个参数:
    • port 指发送到哪个端口 ofproto.OFPP_CONTROLLER 指发送给控制器
    • max_len 指如果数据包需要被截断并发送的最大长度, ofproto.OFPCML_NO_BUFFER 表示不会被阶段
      (可以看出 parser 用于构建 match 和 action 项)
  • self.add_flow(datapath, priority, match, actions), 该方法是自己定义的, 将流表项下发给交换机, priority=0 表示最低优先级

add_flow 方法定义

add_flow 本质上也是向 switch 发送一个指示如何修改流表的数据包:

  • parser.OFPInstructionActions(action_type, actions) 指令类, 用于指定流表项的动作

  • parser.OFPFlowMod(datapath, priority, match, instructions) 是一个流表修改消息类, 用于修改交换机的流表. 其还可指定其他成员:

Chapter 5 Link Aggregation

Link Aggregation, 链路聚合, 是一种将多个物理链路捆绑为一个逻辑链路的技术. 通过链路聚合, 可以增加链路带宽, 提高网络的可靠性和冗余性, 同时在某个链路出现故障时, 其他链路仍能继续工作, 保证网络的稳定运行.

Instruction

OpenFlow 中 Instruction, 用于指定交换机在匹配到数据包后应该执行的操作. (table-miss 也算一种匹配成功)

有以下 Instruction 类型:

  • Goto Table, 将数据包的处理转移给指定 flow table. OFPInstructionGotoTable 类
  • Write Metadata, 设置能被后续流表引用的 metadata (一个单独的数据结构, 而不是修改数据包内容). OFPInstructionWriteMetadata 类
  • Write Actions, 将指定 action 添加到当前 actions set 中 (不会立刻执行), 如果以设置相同类型的 action, 则用新的替换旧的. OFPInstructionActions 类
  • Apply Actions, 立即执行的 action, 不改变 action set. OFPInstructionActions 类
  • Clear Actions, 删除当前 action set 中的所有 actions. OFPInstructionActions 类
  • Meter, 将数据包给指定 Meter. OFPInstructionMeter 类

Instruction 和 Action 的区别

  • Instruction 是流表项的一部分, 用于定义数据包的处理流程, 例如如何修改 Action Set, 如何跳转到其他流表等, 不直接作用于数据包.
  • Action 是对数据包执行的具体操作, 例如转发, 修改, 丢弃等, 直接作用于数据包, 修改数据包的内容或决定数据包的转发行为

Ryu 控制器获取 switch 信息的频率

Ryu 控制器会在下面三种情况下获取 switch 信息:

  • 事件发生 (被动), 交换机和控制器之间发生事件, 如
    • 交换机连接控制器
    • 交换机发送 Packet_in
  • 交换机状态变化 (被动), 如端口 up/down, 流表项添加/删除, 会主动向控制器发送 Port Status 或 Flow Removed 消息
  • 主动查询 (主动), 控制器发送 Multipart Request 等消息, 主动查询交换机状态

解释下 OFPFlowMod 类型

OFPFlowMod 是一种 controller 向 switch 发送的消息类, 主要工作是指示流表的修改.

比如这里的 mod:

  • 其属于 OFPFlowMod 类, 源码位于 ofproto/ofproto_v1_3_parser.py
  • 这里并没有判断要修改的流表项是否存在, 也就是说, 其默认行为就是添加流表项 OFPFC_ADD

根据源码对 OFPFlowMod 的定义, 其各成员解释如下, 其主要指示流表项的信息:

  • cookie, controller 端设置的对流表项的标识符
  • cookie_mask, 仅对 OFPFC_MODIFY* 和 OFPFC_DELETE* 有效, 用来限制对 cookie 的匹配范围, 比如 cookie=0x1234, cookie_mask=0xFF00 匹配 cookie 的高 8 位
  • table_id, 指明该流表项所属的 table ID, 默认是 0
  • command, 指定对流表的操作类型, 默认是 OFPFC_ADD, 添加流表项
  • idle_timeout, 流表项空闲时间, 即该流表项在指定时间内没有匹配的数据包, 则会自动删除
  • hard_timeout, 无论是否有匹配流量, 流表项在指定时间后会被删除
  • priority, 流表项的优先级
  • buffer_id, 指定交换机缓冲区内对应 id 的数据包, 其会在流表项安装后立刻被处理
  • out_port, 仅对 OFPFC_DELETE* 有效, 指定匹配流表项的输出端口 (将其删除)
  • out_group, 仅对 OFPFC_DELETE* 有效, 指定匹配流表项的输出组, 如 out_group=1 表示仅删除输出组为 1 的流表项
  • flags, 流表项的附加标志位, 用于控制特殊行为
  • match, 定义流表项的匹配条件
  • instructions, 定义流表项匹配后的动作指令
