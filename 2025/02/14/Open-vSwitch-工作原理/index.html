Open-vSwitch-工作原理

Open vSwitch 官方文档
Open vSwitch Tutorial

简介

Open vSwitch 是一个 multilayer (暂时不清楚啥意思) 软交换机. 其开放转发功能以供编程扩展和控制.

OVS 适合在虚拟机环境中用作虚拟交换机, 支持多种 Linux 虚拟化技术, 如 KVM, VirtualBox:

运行模式

OVS 支持两种主要的运行模式:

  • High-performance forwarding using a Linux kernel module, 需要访问 kernel space, 性能高
  • operate entirely in userspace, 完全在 user space 运行, 会损失部分性能
    (内核空间可以操控底层硬件, 而用户空间不能)

Open vSwitch 工作在 OSI 模型中的 Data Link Layer (Layer 2), 可以用于虚拟环境下的 Ethernet frames 管理.

相关概念

Bridge

Bridge 工作在 OSI 模型中的 Data Link Layer (Layer 2), 用于连接多个网络段, 使其看起来像一个单一的网络. 其主要功能是根据 MAC 地址在不同网络段之间转发帧.

一个 Bridge 就代表一个 Ethernet switch.

Bridge domains, 指 bridge 的工作范围, 即被整合的整个网络段. 这些网络段被视为一个单一的广播域.

简单来说, bridge 把多个网络整合成一个.

Port

Port 是 Bridge 上的逻辑接口, 用于连接外部网络设备 (如物理网卡, 虚拟网卡, 其他交换机等).

而 Interface 是 Port 的具体实现, 表示实际的网络接口 (如物理网卡, 虚拟网卡等).

一个 Port 可以对应一个或多个 Interface.

在 OVS 中, Port 有以下几种类型:

  • Normal, 用于连接外部网络设备
    • 会将接收到的数据包转发到 OVS 的 datapath 中, 由 OVS 进行处理
  • Internal, 用于连接 OVS 本身或其他 OVS 实例
    • 同样将接收到的数据包直接转发到 OVS 的 datapath 中
  • Patch, 用于连接两个 OVS 实例, 通常用于实现 OVS 之间的通信
    • 将接收到的数据包直接转发到另一个 OVS 实例的对应 Port 上
  • Tunnel, 用于创建虚拟网络隧道, 通常用于实现虚拟网络之间的通信
    • 将接收到的数据包封装到隧道协议中 (如 VXLAN, GRE 等), 然后转发到另一个 OVS 实例的对应 Port 上

Flow Table ID

在 OpenFlow 中, table id 用于区分不同的流表.

数据包在交换机中处理时, 会按照流表的顺序进行匹配.

通常, table id 越小, 流表的优先级越高, 数据包会先匹配低 table id 的流表.

一般 table 0 表示默认流表.

相关工具

  • ovs-ofctl, Open vSwitch OpenFlow Control, 用于与 Open vSwitch 的流表进行交互, 可以添加, 修改, 删除流表项, 以及查询流表信息
  • ovsdb-server, Open vSwitch Database Server, OVS 的数据库服务器, 它存储了 OVS 的配置信息, 包括交换机端口, 流表等数据
  • ovs-dpctl, Open vSwitch Datapath Control, 用于管理 OVS 的 datapath, 可以查看和管理内核 datapath 的接口和流表
  • ovs-vswitchd, Open vSwitch VSwitch Daemon, OVS 的守护进程, 负责管理 OVS 交换机的行为, 包括处理数据包的转发, 维护流表等
  • ovs-vsctl, Open vSwitch VSwitch Control, 用于管理 OVS 交换机的配置, 可以创建, 删除, 修改交换机, 端口, 接口等
  • ovs-appctl, Open vSwitch Application Control, 通常用于调试和监控, 与 OVS 守护进程交互

流表匹配规则

分四部分区分这些条件:

  • OVS 基础匹配条件
  • OSI Layer 2 (Data Link Layer)
  • OSI Layer 3 (Network Layer)
  • OSI Layer 4 (Transmission Layer)

基础匹配条件

in_port=port

流量进入的端口, 如 in_port=1

table=number

Flow table id, 范围是 0-254, 默认值是 0

OSI Layer 2

这一部分的匹配条件以 dl (data link) 开头.

dl_type=ethertype

这里的 ethertype 用十进制或十六进制表示, 如:

  • dl_type=0x0800, 匹配 IPv4 数据包
  • dl_type=0x086dd, 匹配 IPv6 数据包
  • dl_type=0x0806, 匹配 ARP 数据包
  • dl_type=0x8035, 匹配 RARP 数据包

dl_vlan=vlan

指定数据包的 VLAN tag 值, 范围是 0-4095, 0xffff 表示不包含 VLAN Tag 的数据包.

dl_vlan_pcp=priority

设置 VLAN 的优先级, 范围是 0-7, 数字越大优先级越高. 和某个 VLAN Tag 无关.

dl_src=xx:xx:xx:xx:xx:xx

源 MAC 地址.

dl_dst=xx:xx:xx:xx:xx:xx

目的 MAC 地址.

OSL Layer 3

都以 nw_ (network) 开头.

nw_src=ip[/netmask] 和 nw_dst=ip[/netmask]

  • 当 dl_type 为 0x0800 或 ip, 这里的 ip 可以是 IPv4 地址或主机名如 192.168.0.1/24 或 www.test.com
  • 当 dl_type 为 0x0806 或 arp, 这里的 ip 分别与IPv4和Ethernet的ARP数据包中的 ar_spa 或 ar_tpa 字段匹配
  • 当 dl_type 为 0x8035 或 rarp, 这里的 ip 分别与IPv4和Ethernet的RARP数据包中的 ar_spa 或 ar_tpa 字段匹配
  • 当 dl_type 为 0x0800, 0x0806 或 0x8035 之外的其他值时, 将忽略 nw_src 和 nw_dst 的值

nw_proto=proto 和 ipproto=proto

若 proto 为:

  • 1, ICMP
  • 2, IGMP
  • 6, TCP
  • 17, UDP
  • 41, IPv6

但通常:

  • nw_proto: 通常用于匹配网络层协议类型, 适用于 IPv4 和 IPv6
  • ipproto:通常用于匹配 IP 协议类型, 适用于 IPv4

nw_tos=tos

TOS, 指 Type of Service, 用于匹配 IP 数据包头中的 ToS 字段. 其为一个 8 位字段:

  • 最高 6 位: 标识 DSCP (Differentiated Services Code Point), 区分不同的服务类型
  • 最低 2 位: 标识 ECN (Explicit Congestion Notification), 用于拥塞控制

ip_dscp=dscp

匹配 IP 数据包头中的 dscp 字段. 取值为 0~63, 即 6 位二进制.

nw_ecn=ecn 或 ip_ecn=ecn

匹配 IP 数据包头中的 ecn 字段. 取值为 0~3, 即 2 位二进制.

nw_ttl=ttl

匹配 IP 数据包头中的 ttl 字段, 取值 0~255, 即 8 位二进制.

OSL Layer 4

都以 xxx_ (xxx 指协议, 如 tcp, udp) 开头.

xxx_src=port 以及 xxx_dst=port

如:

  • tcp_src=port
  • tcp_dst=port
  • udp_src=port
  • udp_dst=port
  • sctp_src=port
  • sctp_dst=port

匹配 TCP, UDP 或 SCTP 源端口或目标端口, 取值 0~65535.

当通配符 dl_type 和 nw_proto 或将其设置为不表示适当协议的值时, 这些设置的值将被忽略.

xxx_src=port/mask 以及 xxx_dst=port/mask

给上述端口匹配加上 mask.

tcp_flags=flags/mask 或 tcp_flags=[+flag...][-flag...]

都用于匹配 TCP 数据包中的 TCP 标志位.

tcp_flags=flags/mask 中 flags 和 mask 都是 16 位二进制数, 每一位的含义为:

  • 0: fin (字符串用于下面 [+flag][-flag]) 查找不再有来自发送方的数据
  • 1: syn 同步同步序列号
  • 2: rst 重置连接
  • 3: psh 推送功能
  • 4: ack 确认字段有效
  • 5: urg 紧急指针字段有效
  • 6: ece ECN回显
  • 7: cer 减少拥塞窗口
  • 8: ns 现时总和
  • 9-11: 保留
  • 12-15: 不处理, 必须为零

tcp_flags=[+flag...][-flag...] 中, 直接指定哪些标志位需要存在, 哪些需要不存在:

  • +flag: 表示必须存在的 TCP 标志位
  • -flag: 表示必须不存在的 TCP 标志位

余下的匹配

还有很多, 暂不列举.

流表动作

output_port

将数据包输出到 OpenFlow 端口号 port.

in_port

在接收数据包的端口上输出数据包.

drop

丢弃数据包.

group:group_id

把数据包输出给 group table 处理, group table 中匹配一项会执行 a bucket of actions.

normal

将数据包交给 OVS 内置的常规交换机行为处理, 即模拟传统的 L2 交换机转发数据包. 即按照传统交换机的学习规则, 查找 MAC 表并转发数据包.

flood

在所有交换机物理端口上输出数据包.

all

在除接收数据包的端口外所有交换机物理端口上输出.

local

将数据包转发到与本地网桥名称相同的网络设备对应的端口. 如:

  • 将匹配的数据包转发到本地网桥 br0 的本地端口

enqueue(port,queue)

将数据包放入指定端口的指定队列中, 需注意交换机是否支持队列.

mod_vlan_vid:vlan_vid

修改 VLAN tag 为指定 vlan_vid.

mod_vlan_pcp:vlan_pcp

修改报文 VLAN 的优先级.

strip_vlan

从数据包中去除 VLAN Tag.

push_vlan:ethertype

为数据包添加一个 VLAN Tag, ethertype 指定其类型, 通常是 0x8100.

mod_dl_src:mac

修改源 MAC 地址.

mod_dl_dst:mac

修改目的 MAC 地址.

mod_nw_src:ip

修改源 IPv4 地址.

mod_nw_dst:ip

修改目的 IPv4 地址.

mod_tp_src:port

修改 TCP 或 UDP 或 SCTP 的源端口.

mod_tp_dst:port

修改 TCP 或 UDP 或 SCTP 的目的端口.

mod_nw_tos:tos

修改 IP 数据包中的 TOS 字段.

mod_nw_ecn:ecn

修改 IP 数据包中的 ecn 字段.

mod_nw_ttl:ttl

修改 IP 数据包中的 ttl 字段.

resubmit([port], [table])

修改数据包的 in_port 值, 并将其转移给 table 处理.

set_tunnel:id

设置 tunnel id, 用于标识数据包所属的隧道.

set_queue:queue

将数据包放入指定队列.

pop_queue

将所有队列恢复.

move:src[start..end]->dst[start..end]

用于将数据包中的某个字段的一部分或全部移动到另一个字段中:

  • src 指定源字段
  • [start..end] 指定要移动的位
  • dst 指定目的字段

如:

load:value−>dst[start..end]

用指定 value 替换指定 dst 的一部分, 如 load:0x1234->NXM_OF_ETH_SRC[0..15].

push:src[start..end]

将指定字段的一部分压入栈, 如 push:NXM_NX_REG2[0..5].

pop:dst[start..end]

将栈顶出站并存储到指定字段中, 如 pop:NXM_NX_REG2[0..5].

multipath(fields,basis,algorithm,n_links,arg,dst[start..end])

使用 basis 作为通用哈希参数对 fields 进行哈希处理, 然后应用多路径链接选择算法 (带有参数 arg) 从 0 到 n_links-1 选择一个输出链接, 并将链接存储到 dst[start..end] 中. 如 multipath(NXM_OF_ETH_SRC, 0, 1, 2, 0, NXM_NX_REG2[0..7]).

端口镜像

端口镜像是指将经过指定端口 (镜像端口) 的报文复制一份到另一个指定端口 (观察端口), 通过观察端口接收到的数据报文, 就可以有效识别虚拟网络的运行情况.

创建语法:

  • set Bridge <bridge_name> mirrors=@m 表示向 <bridge_name> 添加 mirrors, @m 引用后续创建的 mirror
  • --id=@<port0> get Port <port0>
    • --id=@<port0> 创建一个引用 @<port0>
    • get Port <port0> 获取 <port0> 的信息存储到引用中
  • create Mirror name=<mirror_name> 创建一个 mirror

创建 mirror 时, 能用的参数有:

  • 与 mirror source 相关
    • select_all, boolean, 设置为 true 表示网桥上所有流量
    • select_dst_port, string, 表示此端口接收的所有流量
    • select_src_port, string, 表示此端口发送的所有流量
    • select_vlan, int, 表示网桥上指定 VLAN 标签的流量
  • 与 mirror destination 相关
    • output_port, string, 指定观察端口接收流量保温
    • output_vlan, int, 修改报文 VLAN 标签
