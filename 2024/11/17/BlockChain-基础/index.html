BlockChain-基础

介绍

区块链, block chain, 是一种分布式账本技术 (数据不是存储在单一的节点上), 它允许多个参与者在没有中心化控制的情况下共同维护和更新数据. 这种技术以区块 (指数据结构的基本单元) 的形式存储数据, 每个区块包含一组交易信息 (比如交易的发送者和接受者, 以及交易金额等), 并通过加密哈希链接到前一个区块 (确保顺序性), 从而形成一个链条.

为什么需要区块链

参考 B 站讲解视频
区块链中的货币都不是实体, 依靠 “记账” 来进行交易, 当一个区块变动后, 各节点 (记录所有或部分区块信息) 间可以通过 P2P 协议进行更新 (各区块间的联系就形成了一个网络), 每一个区块相当于一个账本, 都是独立存在的.

区块间通过链连接在一起:

(每日的期末结余与第二天的期初余额相等)

记账的人通过为别人记账获取收入.

而谁来记账则需要团体达成一种共识 (即共识机制进行选举, 毕竟记账的人可以拿钱).

综上, 区块链具有几个特征:

  • 各节点通过 P2P 网络路由
  • 分布式账本, 去中心化
  • 通过共识机制选举记账人
  • 账本内容难以篡改 (毕竟分布式)

相关概念

区块

区块指 block chain 中的基本数据结构, 携带一组交易信息, 比如此次交易的发送者和接受者, 以及交易金额等.


(Block 就是把一组交易信息打包, 把这个 block 链接到以前的交易记录上, 就成了 block chain)

当前区块的哈希值是基于区块内所有数据计算得出的, 如果一个区块的内容被更改, 其哈希值也会改变, 从而使后续区块失效. (从而防止篡改吧)

Block 的组成

  • Hash Pointer, 指向前一个 block 的 Hash (当前 block 的 Hash 要用这个来算, 虽然叫指针, 但不包含地址信息)
  • Merkle Tree, 一种二叉树 (binary tree), 每个叶子节点代表区块中的一笔交易, 而每个非叶子节点 (non-leaf node) 是其两个子节点哈希值的哈希 (即父节点的哈希是其两个子节点哈希的组合), 最顶层的称根节点 (root), 代表整棵树的 hash, 用于快速验证交易是否在某个 block 中以及确认某个交易是否被更改 (从该交易的叶子节点向上沿着树走, 计算路径上的哈希值. 通过与根节点的哈希值进行比较, 可以确认这笔交易是否有效)

用户

用户, Users, 指使用区块链技术进行交易或交互的个人或实体, 负责:

  • 发起交易: 创建和发送交易, 例如将数字货币转账给其他用户
  • 接收交易: 接收来自其他用户的交易
  • 使用应用: 在一些区块链平台 (如以太坊) 上, 与去中心化应用 (DApps) 交互

节点

节点, Nodes, 指在区块链网络中运行的计算机或设备. 每个节点都有能力存储区块链的副本, 并参与网络的维护和数据处理.

其分为三类:

  • 全节点, Full Nodes (fully validating node), 存储整个区块链的完整副本, 并参与验证交易和区块, 全节点提高了网络的安全性和去中心化程度
  • 轻节点, Light Nodes, 只存储区块链的一部分数据 (如区块头), 依赖全节点来获取交易信息, 通常用于资源有限的设备
  • 矿工节点, Miner Nodes, 专门用于挖矿的节点, 参与区块的创建和验证, 通常会竞争解决复杂的数学问题以获得奖励

用户通过与节点交互进行交易等操作.

由于每个节点都有相同的 “账本” 信息 (区块信息), 因此是去中心化的 (即分布式账本).

网络

在区块链 (blockchain) 中, 网络 (Network) 是指由多个节点 (Nodes) 组成的系统, 这些节点通过点对点 (P2P) 的协议相互连接和通信, 以共同维护和更新区块链的账本.

网络一般也分三种类型:

  • 共有链, Public BlockChain, 任何人都可以参与和访问的开放网络, 如比特币和以太坊
  • 私有链, Private BlockChain, 受限制的网络, 只有特定的参与者可以访问和操作, 通常用于企业内部应用
  • 联盟链, Consortium BlockChain, 由多个机构共同管理的网络, 参与者通常是特定的组织, 如银行联盟

挖矿

挖矿, Mining, 指工作量证明中计算复杂数学问题的过程. (因为有人称 Bitcoin 为数字黄金, 得到它的过程就是挖矿)

比特币

比特币, Bitcoin, 是一种去中心化 (没有中央管理机构, 由网络中的用户共同维护) 的数字货币, 其使用区块链技术 (一种去中心化的记账系统) 来记录所有交易, 确保交易的安全性和透明性.

比特币会在打包区块的过程中产生, 按照最初的构想 (2008 年), 每打一个包, 产生 50 个比特币, 每十分钟打一次包, 但是 4 年之后, 每打一个包则产生 25 个比特币, 以此类推, 每过 4 年, 产生的比特币都为之前的一半, 以次可以计算出能够产生的比特币总量为:
$$
\displaylines
{
\begin{aligned}
50 \times 6 \times 24 \times 365 \times 4 \times (1 + \frac{1}{2} + (\frac{1}{2})^2 + ….) \approx 21 milion
\end{aligned}
}
$$

PoW 工作量证明

PoW, Proof-of-Work, 每一个想打包 (记账) 的人都需要计算一个很难的数学题, 如果做出来了, 那就有权利进行打包, 也就是能获取手续费以及打包奖励. 而这个计算的过程则称为挖矿.

哈希算法

哈希函数, 也可称数字摘要或散列函数, 它可以把一个字符串通过一定的运算, 转化成一个摘要的形式, 而且难以进行逆运算, 比如 SHA256 (美国国家安全局发明的一种算法), 如果输入一个字符串, 则会输出一个 256 位的二进制数 (不管输入的长度如何), 而且, 字符串的细微变化都会导致生成的二进制数大不相同.

其特点可以简单归纳为:

  • 正向计算容易
  • 反向计算困难 (大部分情况只能慢慢试输入的字符串)

挖矿的具体原理

网络中的每一个人都收集了交易信息打包成块, 想要拼接到当前区块链后以获取奖励:

但在拼接之前, 需要计算出一个复杂的数学问题:

  1. 一个字符串包含当前块的头部, 账单信息, 时间戳, 随机数
  2. 对字符串做两次 SHA256 运算: Hash = sha256(sha256(string))
  3. 要求 256 位 2 进制数的前 n 位必须是 0
  4. 如果算出来了 (通过改变随机数慢慢试), 则用这个 Hash 值作为新块的头部, 并把打包信息拼接到 blockchain 后

n 的确定与全部挖矿人员的算力相关.

注意每一个 block 可以拆为两部分:

  • Header, 包含 hash 值
  • Body, 包含账单信息, 时间戳

PoS, Proof-of-Stock

Smart Contracts

Smart Contracts 指写入 blockchain 中的一段代码, 用于自动执行一些操作.

Blockchain 工作流程

  1. 用户在 blockchain network 中注册账户
  2. 用户发起一笔交易 (Tx), 如资产转移或信息更新
  3. 用户用私钥对交易进行数字签名, 以保证交易的真实性和完整性
  4. 已签名的交易被发送到网络中的 validation nodes
  5. Validation nodes 检查交易的有效性, 确保符合网络规则
  6. 有效的交易被集中在一个交易池中, 等待进一步处理
  7. Miner nodes 从交易池中选择多笔交易打包成 block
  8. Miner nodes 计算解决数学问题, 为新 block 找到有效的 hash 值
  9. 新区块再次接受验证 (查看 hash 值是否符合要求之类的), 以确保其合法性和准确性
  10. 验证成功的区块由 miner nodes 添加到 blockchain 上
  11. 新添加的 block 信息被广播到整个 blockchain 网络, 通知其他 nodes 更新

02-BTC 密码学原理

虽说比特币是加密货币, 但实际上比特币网络中的所有信息都是公开的.

主要用到两种密码学技术:

  • 哈希
  • 签名

哈希

哈希, 密码学中称: cryptographic hash function, 主要用到其三个性质

  • collision resistance, 指难以哈希碰撞

    两个不同的输入算出同样的哈希值, 就是哈希碰撞, 一般来说不可避免. 可以利用这个性质来得到 digest (只要一改变, digest 就会改变)

  • hiding, 指 Hash 的计算是单向的, 难以逆向计算, 也就是说, Hash 值没有泄漏出这个输入的任何信息

两个性质结合起来, 可以用来实现 digital commitment (或称 digital equipment of a sealed envelope).

如果输入空间不够大, 一般在输入之后添加几位随机数.

  • puzzle friendly, 指 Hash 值的计算一般不可预测, 只是看到输入, 很难猜到输出. 要想得到结果, 只能一个一个试, 所以才能用作工作量证明 (proof of work)

比特币中用到的 Hash 算法为 SHA-256 (Secure Hash Algorithm, 256 位输出)

签名

比特币中的账户管理是去中心化的, 创建账户的过程就是创建一对公私钥:

其思想来自 asymmetric encryption algorithm. (加密和解密用的不是同一个密钥, 加密用公钥, 解密用私钥), 解决对称加密中密钥分发不方便的问题.

比特币账户中:

  • 公钥相当于银行账户, 只要对方知道你的公钥, 就能向你转钱
  • 私钥相当于账户密码, 只要对方知道了你的私钥, 就能将钱转走

在这里, 公私钥用于签名:

  • 自己交易之后, 用私钥签名并将交易信息散布到网络
  • 别人用你的公钥来验证

产生公私钥对时, 需要确保用一个 good source of randomness, 不然可能与别人产生相同的公私钥对.

03-BTC 数据结构

Hash pointers

Block chains 中的链表与普通链表的区别在于:

  • 用 hash pointer 替代了普通指针

普通指针, 在计算机内存中直接指向一个内存地址, 通常用于引用变量或对象. 而 Hash Pointer: 不指向一个具体内存地址, 而是包含所指向的数据的哈希值, 用于确保数据的完整性和防篡改.

假设一个区块链如下:

  • 第一个 block 称 genesis block (创世区块), 最近产生的是 most recent block
  • 一个 block 中的 hash pointer 是根据前一个 block 的所有内容计算出的
  • 如果一个 block 内容被篡改, 则会导致后一个 block 的 hash pointer 无法使用
  • 因此只需要在本地保存最后一个 hash pointer 就能知道区块链是否被篡改

普通链表和区块链的区别:

  • 普通链表, 可以更改其中的元素, 而不影响剩下的元素
  • 区块链, 改动一个元素, 会导致后面所有区块不可用

得益于 Hash pointer, 在本地可以只保存最近的一部分区块, 其他区块在需要时可以向网络中的其他节点询问.

Merkle tree

  • 上层是 hash pointers
  • 最下面一层是数据块 (transaction, 交易数据块)
  • 根节点也能计算一个 root hash

好处在于, 只要记住 root hash, 当树中任意一处修改, 都能检测出.

比特币中, 各个区块用 hash pointer 连在一起, 而每个区块所包含的交易是组织成一个 merkle tree 的形式

一个区块分两个部分:

  • block header, 包含 hash pointer, 比特币协议版本, Merkle tree 的根哈希值, 挖矿难度阈值 (target), 随机数 (nonce) 等信息, 不含任何交易信息
  • block body, 含交易列表 (也就是很多个交易)

Merkle tree 的一个用途在于提供 merkle proof, 验证交易信息是否写入 merkle tree:

  • 最下面颜色填充的是当前交易
  • 从当前交易开始网上计算 hash 值, 得到最终的 hash 值后与 root hash 对比

只要一个数据结构是无环的, 其实都能用哈希指针替换普通指针. (有环的会形成循环依赖)

04-BTC 协议

数字货币防范的主要挑战在于: double spending attack (一个货币多次使用).

一个中心化的解决方法是, 每一次交易都由中心机构验证. 需要维护一个数据结构, 验证货币是否已被花出.

而一个去中心化的货币要解决两个问题:

  • 货币的发行由谁决定 (在 BTC 中由挖矿决定)
  • 如何验证交易的有效性, 如何防止 double spending attack (实现这个的数据结构就是区块链)

验证不合法的区块不会被接收到区块链中.

05-BTC 实现

比特币是 transsaction-based ledger, 每一个区块记录的是交易信息.

每一个用户的账户信息需要根据所有的区块来推算 (这是一个坏处, 而以太坊系统中是 accout-based ledger, 则不存在这个问题).

每一个全节点都需要维护一个 UTXO (Unspent Transaction Output) 的表. 用来防范 double spending attack, 验证这个币是否被花掉了.

每个交易可以由多个输入以及多个输出, 但需要保证: total inputs = total outputs, total outputs 可以略微低于 total inputs, 这一部分作为打包的奖励 (transaction fee) 给打包的人. (防止有些节点只打包自己的交易)

Hash puzzles

这里求 Hash 只用到了 block header 里的内容.

Progress free

过去挖了多久, 和将来多久出块没有关系. (保证公平性)

挖矿的意义

虽然挖矿本身求解的数学问题没意义, 但其对维护比特币系统有重大意义. (只要大部分算力掌握在诚实的节点上)

Confirmation

BTC 协议中, 默认 6 个 confirmation 之后认为一个 block 是无法被篡改的.

06-BTC 网络

Bitcoin 协议运行在 application layer, 而底层位于 network layer, 运行 P2P Overlay Network, 在这个网络中, 所有节点都是平等的.

要想加入一个网络, 至少得知道一个 seed node, 然后 seed note 告知你网络中有哪些其他节点.

Bitcoin 网络的设计原则是: 简单, 鲁棒, 而不是高效.

每个节点维护一个邻居节点的集合, 消息以 flooding 的方式在集合间传播.

节点在第一次接收到一个消息时, 会传播给其他邻居节点, 同时记录这个消息已经收到过了, 下次再收到则不用转发.

邻居节点的选取不考虑底层的网络拓扑结构.

每个节点都要维护一个等待转发的集合 (同一时间可能多条消息需要转发), 当消息写入区块链, 则从等待池中去除.

10-BTC 分叉

分叉的原因可能有多种.

如果是同一时间, 两个节点挖到了矿, 两个节点都能发布区块, 此时的分叉称 state fork. (由于对 block chain 当前状态出现分歧而导致的分叉)

Forking attack 也属于 state fork, 此时也能称 deliberate fork.

如果是对 BTC 协议修改产生的分歧, 称 protocol fork. 根据对协议修改的内容的不同, 可以进一步分为:

  • hard fork, 增加新特性, 如区块大小限制, 其导致的分叉是永久性的, 分开的链用 chain ID 来区分 (互相认为对方链是非法的)
  • soft fork, 特性改变造成的分叉是临时性的, 比如给已存在的域添加一些新的含义

零知识证明

零知识证明, 指一方 (证明者) 向另一方 (验证者) 证明一个陈述是正确的, 而无需透露除该称述是正确的外的任何信息.

同态隐藏

  • 如果 x, y 不同, 那么它们的加密函数值 E(x) 和 E(y) 也不同 (不会碰撞)
  • 给定 E(x) 的值, 很难反推出 x 的值 (不可逆)
  • 给定 E(x) 和 E(y) 的值, 我们可以很容易地计算出某些关于 x, y 的加密函数值
    • 同态加法: 通过 E(x) 和 E(y) 计算出 E(x+y) 的值
    • 同态乘法: 通过 E(x) 和 E(y) 计算出 E(xy) 的值
    • 扩展到多项式

例子:

Alice 想要向 Bob 证明她知道一组数 x 和 y 使得 x+y=7, 同时不让 Bob 知道 x 和 y 的具体数值

证明过程:

  • Alice 把 E(x) 和 E(y) 的数值发给 Bob
  • Bob 通过收到的 E(x) 和 E(y) 计算出 E(x+y) 的值
  • Bob 同时计算 E(7) 的值, 如果 E(x+y)=E(7), 那么验证通过, 否则验证失败

03-ETH 交易树和收据树

  • 交易树和收据树通过查找块中的编号来定位

交易树和收据树可以:

  • 提供 merkle proof
  • 提供复杂查询

bloom filter

Bloom filter 这个数据结构能比较高效判断一个元素是否在一个大的集合中.

Bloom filter 将大的集合计算出一个紧凑的摘要, 比如说一个 128 位的向量.

  • 向量每一位初始化为 0
  • 通过哈希函数将集合中的元素映射到向量中的一个位置 (此时把 0 置 1)
  • 映射完所有元素得到摘要

若要判断一个元素是否在这个集合中, 可以先计算这个元素映射到摘要中的位置是否置 1.

有可能出现误报 (明明不在里面说在). 但不会出现漏报 (在里面的说不在).

一个避免哈希碰撞的方法时用一组哈希函数而非一个.

简单的 bloom filter 不支持删除操作. (删除后, 也不能将摘要中的 1 置 0, 因为可能有其他的元素映射到了这个位置)

每个交易执行完之后会形成一个收据, 这个收据里就包含一个 bloom filter, 记录交易的类型, 地址等信息, 发布的区块在其块头里, 也有一个总的 bloom filter, 其为区块里所有交易的 bloom filter 的一个并集.

比如想要查询过去 10 天与某个智能合约相关的交易, 则查找哪一个区块块头中包含想要查询的内容, 如果块头的 bloom filter 中没找到, 则说明该区块中不包含. 如果找到, 则查找其收据.

以太坊的运行过程

可以看作是一个 “交易驱动的状态机” (transaction-driven state machine).

这里的状态指状态树中的所有内容. 交易则是指每次发布区块内的内容. 通过执行这些交易让系统从当前状态转移到下一个状态. 且状态转移必须是确定性的.

04-ETH GHOST 协议

以太坊由于较快的出块速度, 会经常导致临时性的分叉.

如果沿用比特币的模式, 此时大型矿池导致的分叉更容易成为最长合法链:

这会导致 mining centralization 或 centralization bias (中心化带来的优势).

以太坊采用另一种 GHOST 协议作为共识机制. 其核心思想是, 因未能成为最长合法链而作废的区块的矿工, 也能得到一定的出块奖励. (在以太坊中, 这种区块不称 orphan block, 而是 uncle block)

称为 uncle block 是因为其与父区块统一级. (只有分叉后的第一个区块叫 uncle block)

而且, 最新的区块发布时, 可以包含其 uncle block. uncle block 的矿工可以得到 7/8 的出块奖励 (后六级都可以将其视为 uncle block, 但隔得越远, 获得的奖励越少), 而最新区块的矿工每包含一个 uncle block 可以额外获得 1/32 的出块奖励. 但一个区块最多只能包含 2 个 uncle block. 这样做有利于在系统中出现分叉时及时合并. (最初版本的 GHOST 协议)

这样设计的好处在于:

  • 减少保存的状态信息 (比如隔了 100 代)
  • 尽早合并 (此时奖励最多)

以太坊中的手续费称 gas fee. (这个费用 uncle block 无法得到)

以太币也没有定期将 block reward 减半的设计.

在以太坊中, uncle block 中的合约不会执行, 甚至不会检查其合法性. (只检查是否满足挖矿难度)

05-ETH 挖矿算法

Block chain is secured by mining.

以太坊的挖矿算法称 ethash.

以太坊设计的一个目标就是要做到 ASIC resistance, 一个方法就是增加 mining puzzle 对内存的访问需求, 即 memory hard mining puzzle. 因为 ASIC 芯片虽然算力强, 但是对内存访问的优势不大.

以太坊用两个数据集:

  • 一个 16M Cache, 轻节点保存
  • 一个 1G dataset, 矿工保存, 由 Cache 生成
    (大小定期增长)

来生成 memory hard 的 mining puzzle.

算法步骤如下 (含伪代码):

  1. 生成 16M 的 cache
  • 第一个元素是 seed 的 hash
  • 后面的元素都是前一个元素的 hash
  • 每隔 30000 个块, 就会重新生成 seed 并生成新的 cache
  • cache 的初始大小为 16M, 每隔 30000 个块重新生成时增大初始大小的 1/128, 即 128K
  1. 从 cache 生成 1G 大数据集
  • 用伪随机的顺序从 cache 中读取 256 次数, 每次读取的位置是上一个数经过某种计算得到的
  • 最后一个读取到的数的 64 哈希值作为大数据集中的第 i 个元素

矿工用于挖矿的如下:

  • header 指块头信息
  • nonce 是当前尝试的随机值
  • full_size 是大数据集中元素的个数
  • dataset 是大数据集
  • 先通过 header 和 nonce 计算出一个初始的 mix 值
  • 根据 mix 值求出要从 dataset 中读取的元素的下标
  • 一次读取相邻两个元素 (这俩元素独立生成没什么联系)
  • 循环 64 次
  • 最后得到的 mix 值取 hash 后比对

轻节点验证过程:

  • header 是块头
  • nonce 是包含在 header 中的 nonce (而非此时随机生成)
  • full_size 是大数据集中的元素个数
  • cache 是小数据集大小

07-ETH 权益证明

权益证明, Proof of stake.

基于工作量的证明一个普遍的坏处就是浪费电, 其比拼的是算力, 但算力取决于你购买的机器, 机器取决于你投入的资金, 因此本质上比较的是资金.

与其把钱投入到挖矿, 不如投入到区块链的开发, 按照投入的多少决定记账权, 这就是权益证明的基本思想, 有时可以称这种为 virtual mining.

这种方式省去了挖矿的过程, 由此减小了挖矿所带来的能耗.

工作量证明的区块链系统, 维护它安全的资源不是一个闭环: 如果一个人想发动恶意攻击, 那他只需要用外部世界的货币购买超过 51% 算力的机器来挖矿, 而这种干预是直接通过外部资源的.

而权益证明的区块链系统, 维护它安全的资源是一个闭环: 如果一个人想发动恶意攻击, 那么他需要购买货币超 51% 的发行量, 而这种操作往往还会导致货币的升值.

也有不少加密货币采用混合模型, 其也采用工作量证明, 但挖矿的难度取决于你持有的币的数量. (还要考虑 proof of deposit 等)

权益证明早期遇到 nothing at stake 的问题:

遇到分叉时, 可以两个分支都下注 (用自己的权益投票) 而不会受到影响.

以太坊中准备采用的权益证明协议叫做 Casper the Friendly Finality Gadget (FFG). 其在过渡阶段也要与工作量证明混合使用. 其主要为工作量证明提供 Finality, 即一种最终的状态, 包含在 Finality 中的交易不会被取消, 而单纯基于工作量证明的交易是有可能被回滚的.

Casper 协议引入了 Validator (验证证明) 的概念.

  • 要想成为 Validator, 需要投入一定以太币作为保证金, 这个保证金会在系统里被锁定住
  • Validator 的职责是推动系统达成共识, 投票决定哪一条链是最长合法链, 投票权重取决于保证金的数目大小, 过程类似于数据库中的 two-phase commit

Two-phase commit 指:

  • 每挖出 100 个区块就作为一个 epoch
  • 然后需要决定其是否能作为 finality, 需要进行投票
  • 第一轮投票是 prepare message
  • 第二轮投票是 commit message
  • Casper 规定, 每一轮投票, 都要得到 2/3 以上的 validator 才能通过
  • 且 Casper 在实际系统中, 不区分 prepare message 和 commit message, 只需要进行一轮投票, 且把 100 个区块降为 50 个
  • 这一轮投票, 对于上一个 epoch 来说, 是 commit message, 对于下一个 epoch 来说, 是 prepare message, 要连续两轮投票都要得到 2/3 以上才算有效
