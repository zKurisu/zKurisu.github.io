介绍

OpenFlow 是一种允许网络设备 (如交换机和路由器) 以标准化的方式与 SDN 控制器进行通信的网络协议.

  • Open, 表明其开放性质
  • Flow, 指数据流, 即一组数据包的集合 (具备相同特征, 比如来源同一个用户)

OpenFlow 是当前主流的南向接口, 也是实践 SDN 的首选. 是 P4 和 PISA 的前身.

OpenFlow 的概念要早于 SDN. 最初是为了给研究人员提供一个开放平台, 鼓励设备厂商在产品中支持 OpenFlow, 研究人员就可以在真实网络环境中验证新网络协议, 架构和算法的原型.

OpenFlow 也可以称 “通用转发抽象模型”, 所有网络处理模块能够具有通用功能, 而非协议相关. (比如 MAC 模块处理 MAC 转发表, IP 模块处理 IP 转发表, 改为都用流表)

  • 可以把 Flow table 配制成 MAC 转发表, 也可以配制成 IP 转发表 (更加通用)

OpenFlow 的可编程基于 Flow 的颗粒度 (也就是处理 Flow).

相关概念

Flow Switch

Flow Switch, 流交换机, 负责根据流表中的规则处理网络流量.

Flow Table Entries

主要包含三个部分:

  • Rule, 用于匹配
  • Action, 匹配后的行为
  • Stats, 数据包状态

OpenFlow is just a forwarding table management protocol.

Fully Distributed Protocols

Fully Distributed Protocols, 完全分布式协议, 指不依赖于中心化控制或单一节点来做出决定的协议, 这些协议允许网络中所有节点平等地参与到决策和数据处理的过程中. 具有去中心化, 自组织的特点.

SDN Stack

也就是 SDN 的基本架构:

  • Application 层是用户接触的, 向 Controller 发送请求

OpenFlow 消息格式

由消息头 (header) 和消息体 (body) 组成.

消息头中有四个字段:

  • version, 版本号, 8 bits
  • type, 消息类型, 8 bits, 0 表示 HELLO, 14 表示 FLOW_MOD 消息
  • length, 表示整个消息的长度, 包括 header 和 payload, 16 bits
  • xid, transaction id, 用于标识每个 OpenFlow 消息, 以便在请求和响应之间进行匹配, 32 bits

消息体, 其内容和长度取决于 type, 通常:

  • HELLO 消息, 没有消息体, 仅包含消息头
  • FLOW_MOD 消息, 包含流表规则的详细信息

OpenFlow 消息类型

OpenFlow 协议提供三种消息类型:

  • Controller-To-Switch
  • Asynchronous
  • Symmetric

每种消息类型又针对具体的功能分为多个子消息.

Controller-To-Switch

即 Controller 发往 Switch 的消息, 如流表信息, 资源分配策略等控制信息:

Asynchronous

Symmetric

由 Switch 或 Controller 发起的对称消息 (不是单向的, 两边都要发送), 如维护网络连接状态等:

OpenFlow 中的 Priority

优先级 (Priority) 用于决定流表项的匹配顺序. 当多个流表项与同一个数据包匹配时, 交换机会选择优先级最高的流表项来处理该数据包. 优先级的值是一个无符号整数, 值越大, 优先级越高.

OpenFlow 部分字段

command 字段

command 字段用于指定对流表要进行操作的类型, 常见的有:

  • OFPFC_ADD, 添加一个新的流表项
  • OFPFC_MODIFY, 修改现有的流表项
  • OFPFC_MODIFY_STRICT, 严格修改流表项 (所有字段都匹配时修改)
  • OFPFC_DELETE, 删除现有的流表项
  • OFPFC_DELETE_STRICT, 严格删除流表项 (需要所有字段都匹配时删除)

OFPFC 是 OpenFlow Flow Command 的缩写.

flags 字段

flags 是流表项的附加标志位, 用于控制交换机的特殊行为:

  • OFPFF_SEND_FLOW_REM: 当流表项被删除时, 交换机会向控制器发送 FlowRemoved 通知消息
  • OFPFF_CHECK_OVERLAP: 检查流表项是否与现有条目冲突
  • OFPFF_RESET_COUNTS: 重置流表项的计数器
  • OFPFF_NO_PKT_COUNTS: 不统计数据包数量
  • OFPFF_NO_BYT_COUNTS: 不统计字节数量

OXM 匹配机制

OXM, OpenFlow Extensible Match, 是 OpenFlow 中用于匹配字段的通用格式, 在 OpenFlow 1.2 及后续版本中引入.

其利用 TLV (Type, Length, Value) 的结构来定义匹配字段.

每个 OXM 字段包含的信息如下:

  • Class, 类, 标识字段所属类别 (如基本字段、实验性扩展字段)
  • Field, 字段, 即匹配项 (如源 IP, 目的端口)
  • HasMask, 是否有掩码支持, 用来匹配地址
  • Value, 值, 具体的匹配值
