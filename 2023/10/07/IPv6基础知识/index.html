EUI-64 格式

EUI-64 格式, 是一种将 48 位的 MAC 地址扩展为 64 位接口标识符 (Interface Identifier) 的方法, 常用于 IPv6 地址的自动生成.

Link-Local Address

Link-Local Address 是 IPv6 中一种特殊类型的地址, 用于在单个网络链路 (如一个以太网段或 Wi-Fi 网络) 上进行通信, 仅在本地链路范围内有效, 不能跨路由器转发. (相当于是一个本地地址, 非公网 IPv6)

其格式:

  • 前缀: fe80::/10, 前 10 位固定为 1111111010 (其实就是 fe80 固定)
  • 接口标识符: 后 54 位基于接口的 MAC 地址生成

ICMPv6 协议

ICMPv6, Internet Control Message Protocol version 6, 是 ICMP 协议在 IPv6 环境中的扩展. 其功能有:

  • 错误报告, 如数据包无法到达目标地址时, 发送错误信息 (中间路由器或目标主机发回给源主机)
  • 网络诊断, 如 ping 测试, 利用 ICMPv6 Echo Request 和 Echo Reply 消息测试网络连通性
  • 邻居发现, NDP (Neighbor Discovery Protocol) 和 RDP (Router Discovery Protocol) 协议都依赖 ICMPv6
  • 地址自动配置等.

消息类型

主要分两大类.

错误报告消息

用于报告网络中的错误情况:

  • Type 1, Destination Unreachable, 目标不可达
  • Type 2, Packet Too Big, 数据包过大
  • Type 3, Time Exceeded, 超时
  • Type 4, Parameter Problem, 参数问题

信息性消息

用于网络诊断和管理:

  • Type 128: Echo Request (回显请求)
  • Type 129: Echo Reply (回显回复)
  • Type 133: Router Solicitation (路由器请求)
  • Type 134: Router Advertisement (路由器通告)
  • Type 135: Neighbor Solicitation (邻居请求)
  • Type 136: Neighbor Advertisement (邻居通告)

NDP 协议

NDP, Neighbor Discovery Protocol, 是 ICMPv6 的重要组成部分, 用于 IPv6 网络中的地址解析, 路由发现, 地址自动配置, 其取代了 IPv4 中的 ARP (Address Resolution Protocol) 以及 ICMP 路由发现功能.

其主要功能有:

  • 地址解析: 通过 Neighbor Solicitation 和 Neighbor Advertisement 消息, 将 IPv6 地址解析为链路层地址 (如 MAC 地址)
  • 路由器发现: 通过 Router Solicitation 和 Router Advertisement 消息，主机可以发现网络中的路由器并获取网络前缀
  • 地址自动配置: 主机可以通过路由器通告消息自动配置 IPv6 地址 (无状态地址自动配置)
  • 重复地址检测 (DAD): 在分配 IPv6 地址之前, 主机使用 Neighbor Solicitation 消息检查地址是否已被使用
  • 邻居不可达检测, 通过 Neighbor Unreachable Detect 发现邻居链路状态 (也属于 NS 报文, 但此时的源地址为 fe80::link-local 地址, 目标也是单播), 邻居状态有 5 中, INCOMPLETE, REACHABLE, STALE, DELAY, PROBE

地址解析

主要依靠两种报文:

  • Neighbor Solicitation, NS, 邻居请求, 将目标 IPv6 地址解析为 MAC 地址

报文格式为:

  • Type: 135
  • Code: 0
  • Checksum: ICMPv6 报文的校验和
  • Reserved: 保留字段, 必须设置为 0
  • Target Address: 需要解析的 IPv6 地址
  • Options: 可选字段, 通常用来包含源 MAC 地址

其工作流程为

  1. 当主机需要将目标 IPv6 地址解析为链路层地址时, 它会发送一个 NS 报文
  2. NS 报文的目标地址是目标 IPv6 地址的请求节点组播地址 (Solicited-Node Multicast Address)
  3. 目标主机收到 NS 报文后, 会回复一个 NA 报文

Solicited-Node Multicast Address, 其格式为:

  • FF02::1:FF 是固定前缀, 表示这是一个 Solicited-Node Multicast Address
  • XX:XXXX 是目标 IPv6 地址的最后 24 位

其对应的组播 MAC 地址格式为:

  • 33:33 是固定的前缀, 表示是一个 IPv6 组播 MAC 地址
  • XX:XX:XX:XX 是 IPv6 组播地址的最后 32 位


(此时是组播报文)

  • Neighbor Advertisement, NA, 邻居通告, 用于回复 NS 报文, 提供 MAC 地址

报文格式为:

  • Type: 136
  • Code: 0
  • Checksum: ICMPv6 报文的校验和
  • Flags: 标志字段, 包含的标志位有:
    • R, Router Flag, 表示发送者是路由器
    • S, Solicited Flag, 表示该报文是对 NS 报文的相应
    • O, Override Flag, 表示该报文应覆盖缓存中的现有条目
  • Target Address: 需要通告的目标 IPv6 地址
  • Options: 可选字段, 通常包含目标 MAC 地址

其回复流程为:

  1. 当主机收到 NS 报文时, 它会发送一个 NA 报文作为响应
  2. NA 报文的目标地址是 NS 报文的源地址
  3. 如果主机需要主动通知邻居自己的链路层地址, 它也可以发送一个未请求的 NA 报文 (Unsolicited NA)


(此时是单播报文)

其不依靠广播, 而是用组播地址.

邻居状态变化

也利用 NS 和 NA 报文, 过程大致如下:

冲突地址检测

Duplicate Address Detection, DAD, 冲突地址检测, 同样利用 NS 和 NA 报文.

此时, NS 部分报文内容为:

  • 若检测 2000::1
  • source 为 ::, 即未知地址
  • destination 为 ff02::1:ff00:1 (一个示例)

如果没有回复, 则说明没有冲突.

路由器发现

其主要依靠两种报文:

  • Router Solicitation, RS, 路由器请求, 由主机发送

其报文格式为:

  • Type: 133, RS 的 ICMPv6 类型值
  • Code: 0
  • Checksum: ICMPv6 报文的校验和
  • Reserved: 保留字段, 必须设置为 0
  • Options: 可选字段, 通常包含源链路层地址 (Source Link-Layer Address)

另一个报文是:

  • Router Advertisement, RA, 路由器通告. 路由器发送 RA 报文，通告自己的存在和网络配置信息

其报文格式为:

  • Type: 134, RA 的 ICMPv6 类型值
  • Code: 0
  • Checksum: ICMPv6 报文的校验和
  • Cur Hop Limit: 当前跳数限制 (默认值为 64)
  • Flags: 标志字段, 包含以下标志位
    • M (Managed Address Configuration Flag): 表示主机应使用 DHCPv6 获取地址
    • O (Other Configuration Flag): 表示主机应使用 DHCPv6 获取其他配置信息（如 DNS 服务器）
    • Router Lifetime: 路由器的有效时间（以秒为单位）
  • Reachable Time: 邻居可达性检测的时间
  • Retrans Timer: 重传定时器
  • Options: 可选字段，通常包含以下信息：
  • Prefix Information: 网络前缀信息（如前缀长度、有效时间等）
  • Source Link-Layer Address: 路由器的链路层地址

流程为:

  1. 主机启动. 发送 RS 报文到所有路由器组播地址 (FF02::2), 请求路由器发送 RA 报文
  2. 路由器响应. 路由器收到 RS 报文后, 发送 RA 报文到所有节点组播地址 (FF02::1) 或请求主机的单播地址
  3. 主机处理 RA 报文. 主机收到 RA 报文后, 提取其中的网络配置信息 (如默认网关, 网络前缀等). 如果 RA 报文中包含 M 或 O 标志, 主机会使用 DHCPv6 获取地址或其他配置信息
  4. 路由器定期通告. 路由器定期发送 RA 报文, 确保主机能够持续获取最新的网络配置信息

LLDP 协议

LLDP, Link Layer Discovery Protocol, 链路层发现协议, 是一个数据链路层协议.

LLDP 用来发现网络设备的直连邻居 (不能是多跳), 构建邻居关系表, 进而绘制整个网络的拓扑结构.

LLDP 报文默认情况下的发送周期是 30s, expire time (在这个时间内若收到 LLDP 回复报文, 则认为对端是活跃的) 通常是发送周期的 4 倍, 即 120s.

  • dst MAC, 固定为 01:80:C2:00:00:0E, 这是一个组播地址, 专门用于 LLDP 报文
  • src MAC, 发送设备的 MAC 地址
  • etherType, 固定为 0x88CC, 表示为 LLDP 报文
  • LLDPDU (LLDP Data Unit), 包含设备信息, 为 TLV (Type-Length-Value) 结构

具体的工作流程为:

  1. 初始化. 设备启动后, LLDP 模块会初始化, 并开始监听和发送 LLDP 报文
  2. 发送 LLDP 报文. 设备定期从每个启用了 LLDP 的端口发送 LLDP 报文, 报文中包含自己的信息
  3. 接收 LLDP 报文. 设备接收到邻居的 LLDP 报文后, 解析其中的 TLV 信息, 并存储到本地邻居表中
  4. 更新邻居信息. 如果收到新的 LLDP 报文, 设备会更新邻居表中的信息. 如果 TTL 超时, 设备会删除对应的邻居信息
  5. 拓扑发现. 网络管理系统 (如 SNMP 或 CLI) 可以从设备中获取邻居信息, 并构建网络拓扑

MLDv2 协议

Multicast Listener Discovery version 2, MLDv2, 用于 IPv6 网络中的组播成员管理. 其允许主机明确指定它们希望接收或排除来自特定源地址的组播流量.

主要功能可以概括为:

  • 组播组成员管理, 主机可以向路由器报告其希望加入或离开某个组播组
  • 源过滤模式, 主机可以指定只接收来自特定源地址的组播流量 (INCLUDE 模式), 或者接收除特定源地址之外的所有组播流量 (EXCLUDE 模式)
  • 动态更新, 主机可以动态更新其对组播组的兴趣 (例如, 添加或删除源地址)
  • 路由器查询, 路由器可以发送查询消息, 以了解其直连网络中的组播成员信息

在路由器启动时或需要管理组播成员时, 它会发送 Multicast Listener Query 消息. 因为路由器需要知道其直连网络中是否有主机对特定组播组感兴趣, 以便决定是否转发组播流量.

其定义了 2 种消息类型:

  1. Multicast Listener Query (Type 130), 由路由器发送, 用于查询网络中的组播组成员信息, 源地址为 Link-Local Address, 如 fe80::1, 目的地址为所有 MLDv2 监听者地址, ff02::16 (一个专用地址)
  2. Multicast Listener Report (Type 143), 由主机发送, 用于报告其对组播组的兴趣 (加入, 离开或更新)

MLDv2 的工作流程为:

  1. 路由器发送查询消息
    • 路由器定期发送 General Query 消息, 以了解网络中的组播组成员
    • 当路由器需要了解特定组播组的成员信息时, 会发送 Group-Specific Query 消息
  2. 主机发送报告消息
    • 主机收到查询消息后, 会发送 Multicast Listener Report 消息, 报告其对组播组的兴趣
    • 如果主机对某个组播组的兴趣发生变化 (例如, 加入, 离开或更新源地址), 它会主动发送 Multicast Listener Report 消息
  3. 路由器更新组播路由表
    • 路由器根据收到的报告消息更新其组播路由表, 并开始或停止转发相应的组播流量
  4. 主机离开组播组
    • 当主机不再希望接收某个组播组的流量时, 它会发送 Multicast Listener Report 消息, 记录类型为 CHANGE_TO_INCLUDE_MODE 或 CHANGE_TO_EXCLUDE_MODE, 并指定组播地址和源地址
