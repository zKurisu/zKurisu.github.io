介绍

Category theory (范畴论) 是研究数学结构及其关系的抽象数学的分支. 其将数学对象之间的关系作为 first class 来研究.

1 Category: The Essence of Composition

一个 category 由 objects 和 arrows 组成. 其很容易用图解来展示: objects 可以表示为一个 circle/point, 而 arrows 就是箭头.

Category 的核心在于 composition (组合). 如果有一个 arrow 从 object A 指向 object B, 另一个箭头从 object B 指向 object C, 那么就会有一个 arrow composition, 存在一个 arrow 从 object A 指向 object C.

1.1 Arrows as Functions

Arrows 也称为 “morphisms”, 可以视为是 functions.

假如有:

  • 函数 f, 接受 type A 为参数, 输出 type B
  • 函数 g, 接受 type B 为参数, 输出 type C

那么可以通过将 f 的输出传递给 g, 将两者 compose 为一个新的函数: 接受 type A 为参数, 输出 type C. 在数学上, 这记为 $g \odot f$.

需要注意这里函数的排列顺序, 上述读为 “g after f”, 先执行 f 再执行 g.

1.2 Properties of Composition

Composition 有两个非常重要的性质:

  1. Composition is associative, 是可结合的. 假如有 3 个 morphism, f, g 和 h, 下面三种写法是等价的:

$$
\displaylines
{
\begin{aligned}
h \odot (g \odot f) = (h \odot g) \odot f = h \odot g \odot f
\end{aligned}
}
$$

写成 Haskell 的伪代码为:

  1. 对于每个 object A, 都有一个 arrow 指向自己, 称为 a unit of composition. 一个 arrow 与 unit of composition 进行组合, 得到的是自己

$$
\displaylines
{
\begin{aligned}
f \odot id_A = f \newline~ \newline
id_B \odot f = f
\end{aligned}
}
$$

这里其实会有一个疑问, 为什么会需要 id 这样啥都不干的函数呢?

id 可以类比为乘法中的 1 (1 乘以任何数都不改变数值), 表示一种 “不改变东西” 的操作.

范畴论关心 “如何从一个东西变到另一个东西”. 但还是需要有个东西来描述 “不变”.

这里用快递来类比 category, 假设你管理一个快递公司, 世界里只有城市 (比如北京, 上海, 广州), 以及快递路线 (比如 “北京 -> 上海” 表示可以从北京寄包裹到上海):

  • objects 为城市
  • arrow 为快递路线

为了让整个系统工作, 需要定义下面两条规则:

  1. 如果有一条路线 A -> B, 又有一条 B -> C, 那么你必须能提供 A -> C 的服务 (相当于 composition)
  2. 每个城市必须能 “接收并发出自己的包裹” 而不丢失

比如, 一个用户在北京下单, 但收件地址也是北京, 如果没有 “北京 -> 北京” 的路线, 系统就会崩溃, 根本无法派送. 因此需要为每个城市添加一条 “自己 -> 自己” 的虚拟路线.

这里小结一下: 一个 category 包含 objects 和 arrows (morphisms). Arrows 可以组合, 并且其 composition 是 associative 的. 每个 object 都有一个 identity arrow 表示单位组合.

1.3 Composition is the Essence of Programming

当我们遇到一个问题时, 通常要将其 decompose 为 smaller problems, 然后为各个小问题编写解决代码. 而 Programming 的本质就在于此: 通过代码的组合为问题构造解.

Category theory 不鼓励我们去探究 objects 的内部, 仅将其当作一个 abstract nebulous entity. 我们仅仅知道 objects 如何与其他 objects 交互 (如何用 arrows 进行连接).

2 Types and Functions

2.1 Who Needs Types

Type checking 提供了一种减少 nonsensical programs 的方法. 程序在编译期就能找出类型不匹配的错误, 从而减少运行时的问题.

2.2 Types Are About Composability

Category theory is about composing arrows, 但并不是任意两个 arrows 都能 composed. 一个 arrow 的 target object 必须与另一个 arrow 的 source object 一致才行. 在静态强类型系统中, 这种是否匹配能很容易检查出. 在这个前提之下, 很多程序只要能通过编译, 基本就能正确运行, 而不需要编写太多 unit test.

Type 可以看作是 a set of values. 比如 Bool 类型表示 a two-element set of True and False. Char 类型是 a set of all Unicode characters.

Sets 可以是 finite 或 infinite. 比如 String 就是 infinite set.

  • 函数可能 “不真正映射” 所有输入到输出

2.4 Why Do We Need a Mathematical Model

编程语言的 “语法” 能够精确描述, 但是 “语义” (即程序的实际含义和行为) 极难完整, 形式化地定义.

语法通常指 “语言的拼写和结构规则”, 比如:

  • if (x) { ... } 是合法的
  • if x { ... } 是不合法的
  • 很容易形式化, 且无歧义

而语义指 “一段代码到底做什么”, 比如:

  • 变量求值顺序
  • 内存如何分配
  • 并发下的行为
  • 等等, 难以形式化描述

一种对语义进行形式化的方法是 operational semantics, 把程序看作是在理想解释器中一步步执行的过程, 如果要证明程序有某个性质, 需要模拟所有可能的执行路径. 而这通常在程序员的脑袋里进行, 而人类其实不擅长追踪大量状态, 分支, 递归等.

另一种替代方法是 denotational semantics, 用数学描述程序. 核心思想是: 每个程序片段都对应一个数学对象 (如 function, set 等), 一旦程序被翻译为数学对象, 证明程序性质就等于证明数学定理. 例如:

3 Categories Great and Small

3.1 No Objects

最特殊的 category 是没有 objects 和 arrows 的 category.

3.2 Simple Graphs

可以从一个 directed graph 来构建一个 category:

  1. 给每个 node 添加 identity arrow
  2. 如果有两个 arrow 相连, 添加一个 composition arrow
  3. 每添加一个 composition arrow, 就要考虑是否有新的 composition arrow

这样的 category 称为由 graph 得来的 free category.

3.3 Orders

假如有一组 objects, 它们之间的关系 (arrow) 是 less than or equal, 证明是否是一个 category:

  • 如果有 $a \le b$ 以及 $b \le c$
  • 能得到 $a \le c$

可以看到 arrow 是可组合的. 并且对于每个 objects 都有 $a \le a$, 因此这可以构成一个 category, 该 category 也称 preorder.

重新用 category theory 的方式进行定义 preorder:

  • objects: 集合中的所有元素
  • arrows: 对每对满足 $a \le b$ 的元素之间存在, 这也说明任意两个对象之间至多一个 arrow
  • identity arrows: 对于每个 a, 都有 $a \le a$, 因此有 $a \rightarrow a$
  • composition: 若 $a \rightarrow b$, $b \rightarrow c$, 则能 compose 出 $a \rightarrow c$

preorder 的 identity arrow 描述了 “自反性”, composition 描述了 “传递性”, 之所以叫做 pre-order, 是因为其为 partial order 的前置 (…稍微有点无语).

如果在 preorder 的关系上加上 “反对称性”, 则称为 partial order:

  • objects: 集合中的所有元素
  • arrows: 对每对满足 $a \le b$ 的元素之间存在
  • 反对称性: 若存在 $a \rightarrow b$, $b \rightarrow a$, 则有 $a == b$, 即合并这两个 object, 不会存在双向箭头

之所以叫 partial order, 是因为并不是集合中所有的元素都能够在一个线性队伍上比较.

在 partial order 中加入 “完整性 (comparability)”, 则称为 total order:

  • objects: 集合 T 中的元素
  • arrows: 对任意两个对象 a, b, 恰好存在一个方向的箭头 (完整性):
    • 若 $a \le b$, 则是 $a \rightarrow b$
    • 否则是 $b \rightarrow a$

Total order 就能够让所有元素在一个线性队伍上进行比较.

像 preorder 这种 object 之间至多只能有一个 arrow 的 category 称为 thin category.

这里引入一个新概念, hom-set C(a,b) 表示从对象 a 到对象 b 的所有 arrows 的集合. hom-set C(a,a) 中只有 id (identity arrow). 对于 then category 而言, 每个 hom-set 要么为空, 要么只有一个元素.

Preorder 中允许出现循环, 而 partial order 中不允许.

3.5 Monoid as Category

在抽象代数中, Monoid 是一个三元组 $(M, \cdot, e)$, 满足:

  • 封闭性: $a \cdot b \in M$
  • 结合性: $(a \cdot b) \cdot c = a \cdot (b \cdot c)$
  • 单位元: $e \cdot a = a \cdot e = a$

这里 M 是一个 set, $\cdot$ 是 arrow, e 是 unit element.

在 category theory 中, 可以将 Monoid 描述为一个只有单 object 的 category, 以加法为例:

  • objects: 集合 N (这个集合看作是单个容器, 单个 object)
  • arrows: 所有 $adders: \{add_n | n \in N\}$ ($add_n$ 相当于 Haskell 中的 +n, 函数的部分应用)
  • composition: $add_a \odot add_b = add_{a+b}$
  • identity arrow: $add_0 = id$

Monoid 这个名字也来自于 Greek mono, 表示 single. 可以把 Monoid 表示的 category 写为一个较通用的形式:

  • objects: 只有一个对象 * (或叫做 M)
  • arrows: 每个元素 $m \in M$ 对应一个 arrow $f_m: * \rightarrow *$
  • composition: $f_a \odot f_b = f_{a \cdot b}$
  • identity arrow: $id_* = f_e$

* 是一个容器, 它想表达的是: Monoid 的本质不在于容器包含什么元素, 而是这些元素之间是如何通过 composition 相互作用的.

这里对象一直没有改变, 毕竟只有一个对象, 每个 arrow 都是指向自己.
