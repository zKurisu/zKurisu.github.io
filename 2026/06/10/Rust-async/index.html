Rust-async

Rust 语言圣经中 async 部分

介绍

先区分一下一般意义上的同步和异步:

  • 同步: 在线程阻塞时 (block), OS 会把它挂起, 直到 I/O 完成. 这个线程完全占着 CPU 但什么都不干
  • 异步: 线程发起 I/O 请求后就返回, 继续处理下一个任务. 当 I/O 好了再回来收尾 (I/O 结束后会发起中断)

异步的好处就在于等待期间可以执行代码的其他部分, 不浪费 CPU.

异步编程是一种并发编程模型, 比如 1000 个 I/O 任务, 每启动一个 I/O 之后就会立刻进行下一个 I/O 任务的启动, 而不是等待一个 I/O 结束后才启动下一个, 直观上就像所有 I/O 一起启动了一样.

Rust 中的异步由 async/await 关键字提供.

async vs 多线程

对于 CPU 密集型任务, 推荐使用多线程模型 (并行). 而对于 I/O 密集型任务 (比如 Web 服务器, 数据库连接等网络服务), 则推荐使用高并发模型 (线程的切换开销很大, 当需要频繁切换上下文时选择并发模型更好).

本质上, async 底层也是基于线程实现的, 但是它基于线程封装了一个运行时, 可以将多个任务映射到少量线程上, 然后将线程切换变成了任务切换, 后者仅仅是内存中的访问, 因此要高效很多.

async 的一个缺点在于, 编译器会为 async 函数生成状态机, 然后将整个运行时打包进来, 这会造成编译出的二进制可执行文件体积显著增大.

相关概念

什么是 Runtime

这里解释一个术语 Runtime, 经常在很多地方看到, 都感觉比较让人困惑 (感觉很抽象).

Runtime 表示: 为程序运行提供基本的环境 (程序在运行期间必须存在的支持环境).

比如 C 语言的 runtime, 我们有一段 Hello World 代码:

编译时, 还会链接 crt0.o (C RunTime), 其会进行栈和堆初始化, 命令行参数解析, 程序退出清理, atexit 注册的回调执行等等. 这些就是一个 C 程序运行所必须的环境.

Async 的 Runtime 通常需要管理三个部分:

  • Executor (执行器), 持有任务队列, 决定下一个执行哪个 task
  • Reactor (反应器), 监控 OS 的 epoll/kqueue/IOCP 等, 哪个 socket 有数据或者哪个定时器到期了, 就让 Waker 通知 Executor
  • Timer (定时器), 管理 sleep, timeout 等, 到时间就叫醒对应 task

需要注意, 由于 async 的底层实现非常复杂, Rust 没有选择像 Go 语言那样内置完整的特性和运行时, 而是在语言中仅提供必要的特性支持, 然后通过社区来提供 async 的 runtime 支持. 因此, 如果要完整使用 async 异步编程, 需要依赖下面特性和外部库:

  • 所必须的 trait (如 Future), 类型和函数, 由标准库提供实现
  • 关键字 async/await 由 Rust 编译器提供支持
  • 实用函数, 类型和宏由官方开发的 futures 包提供 (不是标准库)
  • async 代码的执行, IO 操作, 任务创建和复杂调度等复杂特性由社区的 async runtime 提供, 如 tokio 和 async-std

Executor

Rust 的 Future 是惰性的, 一种触发方式是在一个 async 函数中用 .await 来调用另一个 async 函数. 但最外层的 async 函数也需要被调用, 此时就需要用到 executor.

Executor 会管理一批 Future (最外层的 async 函数), 然后通过不停 poll 直到全部完成. Executor 只会在最开始主动执行一次 poll, 后面则是等待 Future 调用 wake 来通知它, 然后再 poll. 这种 wake 通知然后 poll 的方式会不断重复, 直到 Future 完成.

Unpin

大部分类型其实并不在意是否被移动, 因此它们都自动实现了 Unpin trait (一个标记 trait, 未定义任何行为, 可以用 ! 语法去除).

但 Pin 并不是 trait, 它是一个结构体:

其包裹一个指针, 确保该指针指向的数据不会被移动, 比如 Pin<&mut T>, Pin<&T>, Pin<Box<T>> 等, 都能确保 T 不会被移动.

能够让 Pin 起作用, 就需要实现 !Unpin. 没有实现 !Unpin 的类型虽然也能被 Pin 包裹, 但其实是可以移动的. 比如 Pin<&mut u8>, 显然 u8 是实现了 Unpin 的, 因此 Pin<&mut u8> 和 &mut u8 没有区别, 一样可以移动.
