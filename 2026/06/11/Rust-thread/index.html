Rust-thread

Rust 语言圣经 thread 部分

介绍

该 blog 整理 Rust 中多线程和并发 (concurrent), 并行 (parallel) 的相关的内容.

下面是区分 concurrent 和 parallel 的简单例子, 排队到咖啡机做咖啡:

  • Concurrent 相当于可以排多列, 交替处理, 但是只有一个咖啡机, 每次只能做一杯咖啡 (一个 core, 多个任务交替运行)
  • Parallel 也是排多列, 但是每一列都有一个咖啡机, 每次能做多杯 (多个 core, 多个任务可以一起运行)
  • 如果是串行, 那么就只会有一个队列, 一个咖啡机

在早期只有一个 CPU 核心时, 想同时处理多个任务, 借助的就是操作系统的多线程. 重点就在于任务间的调度, 什么时候让一个任务让出核心给其他线程使用.

假如一个任务运行时间过长, 就会导致用户界面的假死, 这时需要靠调度器将任务强行切换到后台, 然后按照某些条件从队列中选择下一个任务进行执行.

当 CPU 核心增加的 N 时, 同一时间就能有 N 个任务被处理了, 此时的并行度就是 N.

但操作系统此时的任务肯定远不止 N 个, 这些任务放到 M 个队列中, 接着交给 N 个核心来执行, 实现了一种 M:N 的处理模型, 此时并行和并发是同时发生的.

对并发和并行的正式定义如下:

  • 如果某个系统支持两个或者多个动作同时存在, 那么这个系统就是一个并发系统
  • 如果某个系统支持两个或者多个动作同时执行, 那么这个系统就是一个并行系统

编程语言的并发模型

由于操作系统提供了创建线程的 API, 因此部分语言会直接调用该 API 来创建线程, 最终程序内的线程数和该程序占用的操作系统线程数相等, 一般称之为 1:1 线程模型, 例如 Rust.

还有一些语言在内部实现了自己的线程模型 (绿色线程, 协程), 程序内的 M 个线程会以某种映射方式使用 N 个操作系统线程去运行, 因此成 M:N 线程模型, 其中 M 和 N 并没有特定的彼此限制关系, 比如 Go.

基本使用

由于多线程的代码是同时运行的, 因此在无法保证线程间的执行顺序时, 会导致一些问题:

  • Race condition, 多个线程以非一致性的顺序同时访问数据资源
  • Deadlock, 两个线程都想使用某个资源, 但是又都在等待对方释放资源后才能使用, 最终都无法继续执行

因此在编程时需要尤其注意这些问题.

线程同步: Mutex

Mutex (Mutual Exclusion, 互斥锁), 能让多个线程并发的访问变成排队访问: 同一时间, 只允许一个线程 A 访问该值, 其他线程需要等待 A 访问后才能继续.

单线程中使用示例:

和 Box 类似, 数据被 Mutex 所有, 要访问内部数据, 需要使用方法 m.lock() 向 m 申请一个锁, 该方法会阻塞当前线程, 直到获得锁.

m.lock() 也可能报错, 例如当前正在持有锁的线程 panic 了. 此时其他线程不可能再获得锁, 因此 lock 方法会返回一个错误.

本质上, m.lock() 返回一个智能指针 MutexGuard<T>, 其实现了 Deref trait, 会被自动解引用后获得一个引用类型, 该引用指向 Mutex 内部的数据; 它还实现了 Drop trait, 会在超出作用域后自动释放锁.

Rc 和 Arc

这里主要想讲 Arc, 因为 Rc 是单线程场景, 但这里顺便就一起讲了.

Rust 中一个值只能有一个所有者, 但是需要考虑下面情况:

  • 在图数据结构中, 一个节点可能由多条边共享, 只有该节点没有边指向它时, 才应该被释放
  • 在多线程中, 多个线程可能持有同一个数据, 但由于 Rust 的安全机制, 无法同时获取该数据的可变引用

Rc<T>

Rc (Reference Counting), 通过记录一个数据被引用的次数来确定该数据是否正在被使用 (而非依靠生命周期). 当引用次数归零时, 代表该数据不再被使用, 此时可以被清理.

Arc

Arc (Atomic Rc), 原子化的 Rc<T>, 能保证数据安全的在线程间共享 (当然会有一些性能损耗).
