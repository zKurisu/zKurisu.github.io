第一讲 操作系统概述

操作系统结构:

操作系统对硬件的抽象示例:

将 I/O 设备抽象为文件, 将内存抽象为地址空间, 将处理器资源抽象为进程. 同时文件也是以地址空间形式组织, 进程也包含地址空间和文件作为其组成.

主要需要关注的主题有:

  • 操作系统结构
  • 中断及系统调用
  • 内存管理
  • 进程管理
  • 处理器调度
  • 同步互斥
  • 文件系统
  • I/O 子系统

操作系统起承上启下的作用, 向上为应用开发提供便利 (比如提供 syscall), 向下管理硬件 (比如 CPU 分配, 内存, 硬盘等管理, 也能够更好支持应用开发).

对于寻常提到的 “操作系统”, 比如 Windows, Linux, 安卓, 这些通常是包含了 “应用程序” 这一层. 这里要学习的 “操作系统” 用更精细的范围来讲是 “OS kernel”.

操作系统的一个核心就是 “支持应用”. 如果一个软件不是为了支持应用, 那么它肯定不是 OS.

操作系统通常会对物力资源进行抽象 (也可以说是一种虚拟化), 比如:

物理资源 操作系统抽象 关键概念
CPU 进程/线程 调度、上下文切换
内存 虚拟地址空间 分页、分段、交换
磁盘 文件 文件系统、inode
网络/外设 Socket/设备文件 驱动、中断处理

操作系统内核通常有下面特征:

  • 并发: 计算机系统中同时存在多个运行程序
  • 共享: 程序间 “同时” 访问互斥共享各种资源
  • 虚拟: 每个程序 “独占” 一个完整的计算机
  • 异步: 服务的完成时间不确定, 也可能失败 (这里的异步可能和后面的异步含义不同)

操作系统历史的演变

还有什么典型系统没有提到? 下面系统的特点是什么? 操作系统的组成是什么? 相比于之前有什么改进? 有什么问题?

  • 单用户系统 (LibOS), 一台计算机同一时间只能被一个用户使用, 此时用户独占全部硬件资源 (CPU, 内存, 外设). 由于操作系统的功能极简, 本质上就是一组程序库, 因此也称 Library OS

此时没有 “内核” 的概念, 也没有保护机制, 程序可以直接访问任意内存空间.

  • 批处理系统, 核心思想是将多个作业 (Job) 收集成一批, 由常驻内存的监控程序自动依次执行. 程序员不再直接操作机器, 而是将程序交给操作员. 相比 LibOS 而言, 能减少一些人工干预, 一批任务能自动顺序执行. 但当前面的任务在等待 I/O 处理时, CPU 也不会空出来给后续任务使用
  • 多道程序系统, 核心思想是内存中同时驻留多道作业, 当一道作业等待 I/O 时, CPU 立即切换到另一道作业. 此时的目标是最大化 CPU 利用率

需要硬件支持中断系统, I/O 完成后硬件通知 CPU. 也需要 DMA 支持, 外设直接和内存交换数据, 无需 CPU 参与. 多个作业在内存中需要隔离, 同时 CPU 要有调度程序决定哪个作业获得 CPU.

  • 分时系统, 核心思想是CPU 时间被切分成极短的时间片 (time slice), 轮流转分配给多个联机终端用户. 及时性强, 能让多个用户同时使用一台主机, 且每个用户相互独立

操作系统结构

单体分层结构: 将单体操作系统 (Monolithic OS) 划分为多层 (levels), 每层仅依赖于相邻底层的接口, 比如最底层 (layer 0) 是硬件驱动, 最高层 (layer N) 是用户界面:

这会产生一个问题, 有时候我可能需要跨层的功能, 需要绕过一些中间层, 此时就很不方便.

微内核结构: 尽可能把内核功能移到用户空间, 用户模块间的通信仅通过消息来传递.

  • LPC (Local Procedure Call)
  • HAL (Hardware Abstraction Layer)

好处在于内核精简比较灵活和安全, 但坏处就是频繁的通信会带来很大的性能开销. 在部分安全要求高的场景下可能会使用.

外核结构: 程序直接链接到操作系统库, 只有与硬件交互的内容存在于外核中, 外核虚拟出多个外设来处理这些请求.

这种方式可以为应用定制操作系统库, 也能更好利用外核提供的服务.

虚拟机结构: 虚拟机管理器将单独的机器接口转换成很多的虚拟机, 每个虚拟机都是一个原始计算机系统的有效副本, 并能完成所有的处理器指令.

Compiler 与 OS

Compiler 负责把程序转换为特定硬件的执行码. 而 OS 负责把这段执行码跑起来.

通常我们在 x86 的机器上开发, 但目标硬件环境是 RISC-V. 因此需要熟悉交叉编译的环境.

整个流程通常是 bootloader 加载 OS 并执行, 之后程序代码经过编译器/汇编器/链接器, 从源码变为汇编指令, 再变为机器码, 最终由 OS 来加载执行.

对于编译器, 需要了解一个三元组 (CPU架构/厂商/操作系统) 来描述目标架构信息:

  • none-elf 表示不依赖特定操作系统, 可以在裸机上运行

最终会得到 ELF 文件 (Executable and Linkable Format, 包含可执行机器码和描述信息).

在编写代码时, 有两种库需要注意:

  • 标准库: 通常以来操作系统, 比如 Rust 的 std 标准库, C 的 glibc
  • 核心库: 与操作系统无关, 比如 Rust 的 core 库, C 的 Linux kernel libc (与 kernel 绑定)

OS 内存布局

编译器不会对 heap 部分做管理, 因为这部分变量的大小无法判断.

bss (Block Started by Symbol) 段属于静态内存分配, 通常是指用来存放程序中未初始化的全局变量的一块内存区域. 但在 Rust 中不允许出现未初始化的全局变量, 不然就是 unsafe 的.

data 段用来存放初始化了的全局变量.

text 段是代码段 (code segment/text segment) 是指存放执行代码的内存区域, 这部分区域的大小确定, 通常属于只读. 在代码段中, 也有可能包含一些只读的常数变量.

heap 是用于动态分配的内存段, 可以动态扩张或缩减. 程序调用 malloc 等函数时, 新分配的内存会被动态添加到堆上. 之后调用 free 等函数时, 释放的内存会在堆中被删除.

stack (栈), 是用户存放程序临时创建的局部变量. 函数被调用时, 其参数和函数的返回值也会放到栈中. 由于栈的先进后出特点, 其能够方便地用于保存/恢复当前执行状态. 放在栈上的变量一定是在编译期已知大小的.

可以把 stack 看作一个寄存, 交换临时数据的内存区.

支持函数调用

函数调用本质:

函数调用约定 (Calling Convention), 其规定在某个指令集架构上, 某种编程语言的函数调用如何实现, 其包括下面内容:

  • 函数的输入参数和返回值如何传递
  • 函数调用上下文中调用者/被调用者保存寄存器的划分
  • 其他的在函数调用流程中对于寄存器的使用方法

RISC-V 的函数调用规范如下:

寄存器组 保存者 功能
a0~a7 (x10~x17) 调用者保存 (caller) 用来传递输入参数, 其中的 a0 和 a1 还用来保存返回值
t0~t6 (x5~x7,x28~x31) 调用者保存 (caller) 作为临时寄存器使用, 在被调函数中可以随意使用无需保存
s0~s11 (x8~x9,x18~x27) 被调用者保存 (callee) 作为临时寄存器使用, 被调函数保存后才能在被调函数中使用

M-mode 的能力远超 S-mode, 但并不适合作为通用 OS 的内核态. 如果 OS 在 M-mode 上运行:

  • 每次 OS 做上下文切换, 需要保存/恢复所有 M-mode 状态的寄存器, 开销巨大

  • OS 代码会和具体硬件板子强耦合, 无法移植

    能力 M-mode S-mode
    访问所有 CSR（控制状态寄存器） ✓ 部分
    操作物理内存保护 (PMP) ✓ ✗
    处理所有中断/异常 ✓ 可被委托
    控制所有硬件特性 ✓ 受限
    跨平台 否（和板子绑定） 是（通用）

这种设计下, OS 代码就能与具体硬件解耦合, 通过 SBI 来屏蔽硬件差异, 不管是什么硬件, OS 都发出相同的 SBI 调用.

QEMU 运行模式

QEMU 有两种运行模式:

  • User mode, 即用户态模拟, 如 qemu-riscv64 程序. 能够模拟不同处理器的用户态指令的执行, 并可以直接解析 ELF 可执行文件, 加载运行那些为不同处理器编译的用户级 Linux 应用程序
